受众：在 Paperclip 中设置和运行智能体的操作员

## 1. 这个系统做什么

Paperclip 中的智能体不是持续运行的。  
它们以**心跳**方式运行：由唤醒触发的短执行窗口。

每个心跳：

1. 启动配置的智能体适配器（例如，Claude CLI 或 Codex CLI）
2. 给它当前的提示/上下文
3. 让它工作直到退出、超时或被取消
4. 存储结果（状态、令牌使用情况、错误、日志）
5. 实时更新 UI

## 2. 智能体何时唤醒

智能体可以通过四种方式唤醒：

- `timer`：计划间隔（例如每 5 分钟）
- `assignment`：当工作分配/签出给该智能体时
- `on_demand`：手动唤醒（按钮/API）
- `automation`：为未来自动化系统触发的唤醒

如果智能体已经在运行，新的唤醒会被合并而不是启动重复的运行。

## 3. 每个智能体要配置什么

### 3.1 适配器选择

内置适配器：

- `claude_local`：运行本地 `claude` CLI
- `codex_local`：运行本地 `codex` CLI
- `opencode_local`：运行本地 `opencode` CLI
- `cursor`：在后台模式下运行 Cursor
- `pi_local`：在本地运行嵌入式 Pi 智能体
- `hermes_local`：运行本地 `hermes` CLI（`hermes-paperclip-adapter`）
- `openclaw_gateway`：连接到 OpenClaw 网关端点
- `process`：通用 shell 命令适配器
- `http`：调用外部 HTTP 端点

外部插件适配器（通过适配器管理器或 API 安装）：

- `droid_local`：运行本地 Factory Droid CLI（`@henkey/droid-paperclip-adapter`）

对于本地 CLI 适配器（`claude_local`、`codex_local`、`opencode_local`、`hermes_local`、`droid_local`），Paperclip 假定 CLI 已安装并在主机上认证。

### 3.2 运行时行为

在智能体运行时设置中，配置心跳策略：

- `enabled`：允许计划的心跳
- `intervalSec`：计时器间隔（0 = 禁用）
- `wakeOnAssignment`：分配工作时唤醒
- `wakeOnOnDemand`：允许 ping 风格的按需唤醒
- `wakeOnAutomation`：允许系统自动化唤醒

### 3.3 工作目录和执行限制

对于本地适配器，设置：

- `cwd`（工作目录）
- `timeoutSec`（每个心跳的最大运行时间）
- `graceSec`（超时/取消后强制终止前的时间）
- 可选的环境变量和额外的 CLI 参数
- 在智能体配置中使用**测试环境**来运行适配器特定的诊断，然后再保存

### 3.4 提示模板

您可以设置：

- `promptTemplate`：用于每次运行（第一次运行和恢复的会话）

模板支持变量，如 `{{agent.id}}`、`{{agent.name}}` 和运行上下文值。

> **注意：** `bootstrapPromptTemplate` 已弃用，不应在新智能体中使用。使用它的现有配置将继续工作，但应迁移到托管指令包系统。

## 4. 会话恢复行为

Paperclip 为可恢复适配器存储会话 ID。

- 下一个心跳自动重用保存的会话。
- 这提供了跨心跳的连续性。
- 如果上下文变得陈旧或混乱，您可以重置会话。

在以下情况下使用会话重置：

- 您显著改变了提示策略
- 智能体陷入不良循环
- 您想要干净的重启

## 5. 日志、状态和运行历史

对于每个心跳运行，您获得：

- 运行状态（`queued`、`running`、`succeeded`、`failed`、`timed_out`、`cancelled`）
- 错误文本和 stderr/stdout 摘要
- 适配器可用时的令牌使用情况/成本
- 完整日志（存储在核心运行行之外，针对大输出进行优化）

在本地/开发设置中，完整日志存储在磁盘上配置的运行日志路径下。

## 6. UI 中的实时更新

Paperclip 将运行时/活动更新实时推送到浏览器。

您应该看到以下实时更改：

- 智能体状态
- 心跳运行状态
- 智能体工作引起的任务/活动更新
- 相关的仪表板/成本/活动面板

如果连接断开，UI 会自动重新连接。

## 7. 常见操作模式

### 7.1 简单自主循环

1. 启用计时器唤醒（例如每 300 秒）
2. 保持分配唤醒开启
3. 使用专注于提示模板，告诉智能体在同一心跳中操作，留下持久进度，并将阻塞的工作标记为所有者/操作
4. 观察运行日志并随时间调整提示/配置

### 7.2 事件驱动循环（减少持续轮询）

1. 禁用计时器或设置长间隔
2. 保持唤醒分配启用
3. 使用子问题、评论和按需唤醒进行交接，而不是轮询智能体、会话或进程的循环

### 7.3 安全第一循环

1. 短超时
2. 保守的提示
3. 监控错误并在需要时快速取消
4. 当出现漂移时重置会话

## 8. 故障排除

如果运行反复失败：

1. 检查适配器命令可用性（例如 `claude`/`codex`/`opencode`/`hermes` 已安装并登录）。
2. 验证 `cwd` 存在且可访问。
3. 检查运行错误 + stderr 摘要，然后是完整日志。
4. 确认超时不过低。
5. 重置会话并重试。
6. 如果智能体导致重复的不良更新，则暂停智能体。

典型故障原因：

- CLI 未安装/认证
- 工作目录不良
- 适配器参数/环境格式错误
- 提示过于宽泛或缺少约束
- 进程超时

Claude 特定说明：

- 如果在适配器环境或主机环境中设置了 `ANTHROPIC_API_KEY`，Claude 使用 API 密钥认证而不是订阅登录。Paperclip 在环境测试中显示为警告，而不是硬错误。

## 9. 安全和风险说明

本地 CLI 适配器在主机机器上运行在无沙箱环境中。

这意味着：

- 提示指令很重要
- 配置的凭据/环境变量是敏感的
- 工作目录权限很重要

尽可能从最小特权开始，并避免在广泛的可重用提示中暴露密钥，除非故意需要。

## 10. 最小设置清单

1. 选择适配器（例如 `claude_local`、`codex_local`、`opencode_local`、`hermes_local`、`cursor` 或 `openclaw_gateway`）。外部插件如 `droid_local` 也可通过适配器管理器获得。
2. 将 `cwd` 设置为目标工作区（对于本地适配器）。
3. 可选地添加提示模板（`promptTemplate`）或使用托管指令包。
4. 配置心跳策略（计时器和/或分配唤醒）。
5. 触发手动唤醒。
6. 确认运行成功并记录会话/令牌使用情况。
7. 观看实时更新并迭代提示/配置。
