# 评论唤起过载与编排分层改版计划

**日期：** 2026-05-17  
**类型：** 长期需求 · 改版计划清单（非「最小路径」收口；按面拆单、`执行/` 分期落地）  

## 背景（为何需要这一份）

在人类侧仅提交**一条评论**时，端到端可能出现：**控制面唤起 + CLI 超长会话（多 reasoning 轮次、并行工具、整条 `paperclip` skill 注入）、环境命令重试、`heartbeat_timer` 与 `issue_commented` 在时间线上的叠加观感、stdout JSON 超限导致适配器解析失败**等——综合表现为「简单动作 → 连锁重反应」。这不等于编排器「瘫痪不可用」，但需要**产品语义、默认值、适配器能力与可观测性**分层改版。

## 改版目标（可验收的高层表述）

1. **人类动作可预期**：操作者能理解「这一轮会载入多少规程 / 会否拉全 Skill / timer 会否抢跑」，并在 UI 或服务端策略上可选档位。  
2. **单侧 run 体量可控**：在评论唤起等高频路径上有**默认节流**（含 token、轮次、并行工具、`reasoning_tokens`）。  
3. **可观测且不误导**：单次人类动作与时间线上多条 run / 多条评论的因果关系可读（含 `createdByRunId`、`wakeReason`、合并防抖记录）。  
4. **适配器坚韧**：超长 CLI 会话不再依赖「单行巨 JSON 必 parse 成功」；失败可诊断。  
5. **Windows/Git Bash 执行面减少试错**：减少对 `more`/`/tmp` 等不可靠假设，降低「失败—重试—再吃进上下文」的雪球效应。  

## 计划清单（按改造面分列；每项可单独立项进 `执行/`）

### A. 唤起语义与调度（server / orchestration）

- **A1 评论唤起策略分级**  
  `issue_commented` 下设多档：**仅需回执**、**读线程即止**、**允许拉 API 增补**、**允许全 Skill**、**允许改仓**；按标签 / `executionPolicy` / 角色 / 事务类型选配，或由 Board 默认值 + 工单覆盖。

- **A2 多源唤醒合并与防抖**  
  同一短时窗内同一 `agentId + issueId`：`timer` 与 `automation` **显式优先级**（谁吞谁、谁延迟、谁记入审计）；run 元数据必须带 **effectiveTrigger**（最终生效原因）。

- **A3 `maxConcurrentRuns` 语义产品化**  
  `queued`/`running`/不同 `invocationSource` 组合的**可见说明 + 可选策略**（例：CEO 在处理 comment wake 时 timer 退让 N 秒、或丢弃本轮 timer）。

- **A4 「子单已建 + 指派 + 心跳关闭」剧本**  
  标准链路写清：**谁评论、谁 `checkout`、谁 wake assignee、`wakeOnDemand` 与 pause 同框时的恢复路径**（避免 CEO 长推理试错 `PATCH agent`）。

### B. 提示词 · Skill · 上下文注入（harness / company template / skill 包）

- **B1 Skill 懒加载**  
  `Skill paperclip` 默认不得全量正文进会话；改为**目录 + 章节按需展开**，或服务端注入「摘要版 + 跳转引用」。

- **B2 Wake payload 与 Skill 去重**  
  `fallbackFetchNeeded: false` 时禁止重复载入与 wake 摘要等价的超长规程段落。

- **B3 工作区说明书封顶**  
  `AGENTS.md` / `codebuddyMd`：**token 预算**、与工作项弱相关时的折叠策略、跨仓指北默认不整块灌入唤起路径。

- **B4 角色模板分叉**  
  CEO / CTO / reviewer：**评论唤起路径禁用编码默认**（若章程要求 CEO 不写代码则在模板层硬约束工具白名单）。

- **B5 reasoning 成本控制**  
  对唤起类 run：`reasoning` 开关或档位（模型侧 `reasoning_tokens` / 等价能力）；与 **A1 档位**联动。

### C. CodeBuddy CLI 适配与可观测（packages/adapters、`usage`/`run` 记录）

- **C1 超大 stdout 的稳健解析**  
  `oversized_result_json`、`stdoutTruncated` 时有 **fallback**：分段、`NDJSON`/增量快照、或落盘后 URI 指针；**不要求**单行完整 JSON。

- **C2 结构化 run 摘要与 stdout 解耦**  
  `num_turns`、终结状态、工具统计、vendor usage、终局错误码写入**独立字段**（不绑定整段会话 JSON）。

- **C3 Windows / Git Bash 执行规范内化**  
  文档化 + 预设：`more`、`/tmp` vs `$TEMP`、`cmd.exe` vs `node` 直启（与现有 「Plan A: bypassing cmd.exe」条目对齐）；适配器默认 env 或服务端 preamble 可减少模型试错。

- **C4 「Plan A spawn」矩阵**  
  何种场景走何种启动链；与 **`探查/019`**（心跳定时器）、**`执行/035`**（适配器子进程环境）、**`最佳实践/006`**（并发兜底）等交叉链接（实现时回填）。

### D. execution policy · API 语义（server + UI）

- **D1 审批参与者身份可操作**  
  `local-board` vs 登录 UUID vs Agent Key：**422** 时 UI 明示「你是谁、流程在等谁」，并提供低风险治理操作（不放宽安全边界，只消灭误配置）。

- **D2 审批阶段机器可读进度**  
  除人肉评论「审批通过」外，提供可查询的阶段状态（减少 agent 「猜批文」）。

- **D3 子事务标准 API 契约**  
  「父批子办」：**评论 / `interaction` / `checkout` / wake** 的推荐顺序写入 API 契约与 UI 向导。

### E. UI / 运维体验（ui）

- **E1 时间线与评论溯源**  
  评论、`run`、唤醒原因同屏：`createdByRunId`、`wakeReason`、`effectiveTrigger`。

- **E2 唤起前预览（可选）**  
  「预计注入 Skill？预计工具白名单？」——与 **A1/B1** 联动。

- **E3 「轻量叫醒」操作位**  
  与满分叫醒并存（映射服务端档位）。

### F. 成本与预算（与台账口径对齐）

- **F1 分源预算**  
  `heartbeat_timer`、`issue_commented`、`continuation` 分项限额或告警；与 **`015`** 台账、`023` 粗算对齐。

- **F2 「唤起类」专项报表字段**  
  审计「一句评论烧了啥」（input/output/cache-hit、工具次数、retry 次数）。

- **F3 `total_cost_usd` 语义**  
  若上游为累计口径，适配器与本 run delta 对齐或文书免责（ **`015`** 已有方向，本项收口实现）。

### G. 安全与租户边界（server）

- **G1 CEO PATCH 下级 agent（启心跳、解 pause）权限矩阵**  
  写明允许集、禁区、审计字段。

- **G2 UTF-8 写回通路产品化**  
  减少对「模型手写危险的 `curl -d JSON`」的依赖：提供 SDK/小包或 agent 侧辅助调用。

### H. 测试与发布（CI / 手册）

- **H1 场景化契约测试**  
  comment wake 与 heartbeat **同时撞车**、Skill 全开、stdout 超限、Windows Bash 缺 `more` 四类极小集冒烟。

- **H2 回归指标**  
  comment wake：**p95 tokens**、工具调用次数、p95 latency、适配器 JSON parse 失败率。
