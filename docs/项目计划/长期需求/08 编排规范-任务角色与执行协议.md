# 编排规范 — 任务角色与执行协议

> 替代 Paperclip 原生 CEO/CTO 编排链路。任务从工程计划直接映射为 issue，不走公司目标链。

---

## 1. 原则

**任务驱动，非公司驱动。** Issue 的起点是 Token Bridge 工程计划（P6）和文档域，不是公司目标→CEO 拆任务。

**角色按需，非全量。** 一个 issue 只分配与任务规模匹配的角色子集，不走全角色串行链路。

**结果回写文档域，issue 留痕不走型。** AI 产出写入项目文档域；issue 内只保留摘要和指针。

---

## 2. 角色分类

角色分两类：**心跳角色**（定时巡检，自动唤醒）和**任务角色**（事件驱动，手动触发）。

### 2.1 心跳角色（定时巡检）

| 编号 | 角色 | 职责 | 触发 |
|------|------|------|------|
| H1 | 文档整理 | 巡检 docs/ 目录：整理新产出、更新索引、检测孤立文档、推动过期待归档 | 定时心跳（建议间隔 30-60 分钟） |
| H2 | 运维监督 | 检查各服务健康状态、监控磁盘/内存/日志、发现异常建 issue | 定时心跳（建议间隔 10-30 分钟） |
| H3 | 故障维护 | 处理 H2 发现的问题：重试失败任务、清理孤儿进程、恢复卡住状态 | 由 H2 创建 issue 后手动触发，或定时轻巡检 |

心跳角色不出正式文档，不写业务代码。产出为：
- H1 → 文档索引更新、待处理清单、归档案建议
- H2 → 异常 issue（指派给对应任务角色）、健康摘要
- H3 → 恢复操作记录、issue comment

### 2.2 任务角色（事件驱动）

任务角色不自动唤醒。只被以下事件触发：
- 人从 Jarvis / 工程计划创建 issue
- 心跳角色发现异常后创建 issue
- 上一角色完成后的角色链切换

| 编号 | 角色 | 职责 | 触发条件 |
|------|------|------|----------|
| T1 | 计划 | 拆目标为子任务、排执行顺序、定验收标准 | 新目标进入，需分解为多步时 |
| T2 | 探查 | 读代码、读文档、读外部资料、梳理现状 | 涉及现有代码或外部信息时 |
| T3 | 分析 | 判断方案优劣、识别风险、给出推荐 | 存在多个可行路径时 |
| T4 | 需求 | 整理需求、明确边界、写验收条件 | 需求模糊或需要结构化时 |
| T5 | 执行 | 写代码、改配置、跑命令、交付产物 | 任何实现任务 |
| T6 | 测试 | 验证执行结果、回归检查、边界测试 | 有可测产物时 |
| T7 | 运维 | 部署、监控、健康检查、环境变更 | 涉及部署或环境操作时 |
| T8 | 复盘 | 提取经验、记录教训、归档关键决策 | 重要任务完成后 |
| T9 | 总结 | 合并多步结果、写摘要、关闭 issue | 任务链收尾 |

**任务规模与角色匹配：**

```
小任务（单步可完成）    → T5 + T9
中任务（2-5 步）        → T1/T2 → T5 → T6/T9
大任务（多层子任务）    → T1 → T2 → T3 → T4 → T5 → T6 → T8 → T9
运维类                  → T2 → T7 → T6 → T9
```

## 3. 任务来源

### 3.1 工程计划（P6）

Token Bridge `docs/` 中标记为 P6 优先级的任务，直接映射为 Paperclip issue。

映射规则：
- issue title = 任务标题
- issue description = 目标 + 边界 + 允许操作 + 禁止 + 验收
- issue 标签标记来源：`source:token-p6`

### 3.2 文档域

`docs/` 中状态为"草稿"或"待执行"的计划文档，按需创建 issue。

```

- 回写结果到 [文档域路径]
- issue comment 写入摘要
- 附上指针链接
```

---

## 5. 执行协议

### 5.1 角色切换

- 同一 issue 内按角色链顺序执行
- 上一角色完成后，下一角色从 issue context + run log 继承上下文
- 角色间交接写 comment，包含：做了什么、结论、指针、下一步

### 5.2 回写规则

- **产出内容** → 写入 `文档域`（Token Bridge `docs/` 或项目文档指定位置）
- **issue comment** → 摘要（不超过 10 行）+ 文件指针
- **issue 本身** → 不承载文档内容，只留状态、摘要和指针
- **禁止**：把正式文档内容只写在 issue comment 里

### 5.3 收件箱集成

- issue closed 时，生成最小摘要对象写入 Jarvis 可读路径
- 摘要格式沿用 `07 五仓定位.md` §8 的定义

## 7. 与 Paperclip 原生编排的切割

| 组件 | 状态 |
|------|------|
| Company goal → CEO 拆任务 | 关闭 |
| Board approval 审批流 | 关闭 |
| Automatic agent hiring | 关闭 |
| CEO/CTO agent 角色 | 移除 |
| Paperclip 自带技能提示词 | 不使用 |
| Heartbeat 调度 | 保留（仅心跳角色 H1/H2/H3；`timerHeartbeatEligibleAgentRoles` = ["文档整理","运维监督","故障维护"]） |
| Issue 表 + checkout + run log | 保留 |
| Adapter + resultJson | 保留 |
| Recovery 机制 | 保留 |
