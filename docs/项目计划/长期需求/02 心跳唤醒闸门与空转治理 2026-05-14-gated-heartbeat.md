## 1. 我们在解决什么问题

- **现象：** 定时心跳到点就可能把智能体拉起来跑一轮，**不先判断**这位智能体当前有没有**归属于自己的、可处理的工作**。容易造成 **空转**（起来只做轮询或 noop），**浪费 token**，也让活动日志里出现「干跑」的运行。
- **定性：** 这是**产品规则与实现不一致**：执行者侧不应「无因唤醒」；指挥链上的角色也应有**清晰的可唤醒条件**，而不是默认「到点就全员轮询」。

---

## 2. 核心原则（通俗版）

1. **执行者（干活的人）**
  - **不允许**在**没有任何指派给自己的待处理事项**时，被定时心跳随便拉起来。  
  - 有指派 → 心跳可以把他叫醒 → 他去**领单 / 检出 / 干活 / 结案**这一套不变。
2. **指挥链上级（举例）**
  - **CEO / CTO 等**：业务上可能需要**审回单、批方案、关单**，但即便如此，也不等于「无事也可以轮询玩」。  
  - **更可接受的唤醒**是：存在**明确需要他处理**的事项（例如：下属已提交待审、待批、待关单的单子，或系统规则定义的「仅指给他的回单/审批」）。  
  - **「没有指定任务也可以随便看」若作为例外，必须单独约定**：范围、审计、频率，否则默认 **视同执行者**，一样要走「有待办才唤醒」。
3. **关单要闭环**
  - 下属干完并上报 → 上级不是在心跳里「逛逛」，而是对**指定给自己的**回单做判断；**通过**后还应走到**审批/关单**语义上的**彻底结束**（具体形态可与现有 `issues` / `approvals` 产品能力对齐，此处只写业务要求：**不能停在口头 done**）。
4. **节省 token 的逻辑**
  - **没活的人睡着**；**有活才叫醒**。  
  - 这样避免「全员闹钟 → 全员 GET 收件箱 → 大部分都是空」的固定损耗。

## 3. 场景故事（方便对齐直觉）

**场景 A — 派活与回单**

1. CEO 给 Code 派活；Code 领任务、做完、把结果**回给 CEO**（待阅 / 待审）。
2. **CEO 平时不必被定时心跳空转拉起**；只有当**存在明确需要 CEO 处理的事项**（例如待审回单、待批结案）时，才应用心跳唤醒 CEO。
3. CEO 审完后：**通过** → 走**审批完成 / 关单**路径，让事务在流程上**真正结束**。

**场景 B — 执行者沉睡与再开工**

1. Code 当前没有指派给自己的待办 → **应保持沉睡**（不被无因定时心跳吵醒）。
2. 当上头**又指派了一条给 Code** → 此时才允许心跳（或同等级别的唤醒）把 Code 拉起来，让他去**领单干活**。

## 5. 非目标与开放问题

- **健康探针 / 保活**：若未来需要「证明进程还活着」但不跑模型，应**单独设计**（例如不入账 token 的轻量检查），**不**沿用「完整心跳 run」冒充保活。  
- **指标**：是否统计「因无待办跳过」的次数，用于观察 token 与空转是否下降。  
- **与上游 Paperclip**：本文档描述本 fork 的产品意图；若合并上游须再对齐全局策略。
