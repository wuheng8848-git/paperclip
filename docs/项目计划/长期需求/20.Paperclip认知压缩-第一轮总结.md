## 一、分层定位：回形针 vs CodeBuddy CLI

```
公司/项目  ──┐
事务/调度  ──┤── 回形针（编排大脑）
记忆      ──┤
            │
壳子/工具  ──┤── CodeBuddy CLI（执行手臂）
模型       ──┘
```

- **回形针**：多 Agent 编排、心跳轮询、汇报链、issue→checkout→run 生命周期
- **CodeBuddy CLI**：48 工具、无头模式 -p、Daemon 后台、Hook 系统、Bash 沙箱
- **两者互补**，不是替代。回形针管"谁干什么"，CLI 管"怎么干"

## 二、关键架构概念

| 概念 | 要点 |
|------|------|
| 技能注入 | 7 环节链路：DB→发现→解析→计算→同步→**symlink**→确认。不是 API 传参，是文件系统软链接 |
| 技能两类 | **报警器型**（diagnose，关键词触发）vs **扳手型**（create-agent，语义激活） |
| Heartbeat 方向 | Agent CLI **主动轮询**服务器 `GET /heartbeat-runs`，不是服务器推送 |
| 启动铁律 | issue→checkout→CLI 进程→心跳轮询。checkout 是原子操作，严禁手动改 status |
| 两仓库分离 | 控制面（Paperclip 自身）+ 工作区（Agent 操作的目标仓库），cwd 指向后者 |
| 汇报链 | `reportsTo` 字段串链，逐级上报。Code→CTO→CEO→人类 |

## 六、核心教训

### 9.1 重写 CodeBuddy 适配器解决什么

**现象上以 CodeBuddy 暴露得最狠**，但根因在 **`codebuddy_local` 所代表的「本地子进程 + 心跳执行」范式**，与上引计划文同一类：**spawn 后子树归属弱、取消只触及顶层 PID**。当前适配器典型缺陷：**spawn-and-forget** — 启 CLI 进程，不追踪、不回收、不管理完整生命周期与子孙 PID 树。

| 问题 | 当前 | 重写后 |
|------|------|--------|
| 僵尸进程泄漏（66 个/12 CPUh） | spawn 完不管，孙子进程必然泄漏 | PID 树追踪，run 结束遍历 kill，二次确认 |
| 冷启动开销 | 每次 heartbeat 起新 CLI，重复加载 | Daemon 热会话，或进程池复用 |
| 并发冲突 | 同一 agent 多 run 同时写仓库 | maxConcurrentRuns 限流 + 工作区隔离 |
| 超时无回收 | agent 卡死永久占用，人类手动 done | maxTurns 熔断 + 超时自动 failed + 进程 kill |
| 状态不可观测 | 不知道 CLI 在干什么 | 适配器层上报 run status + PID + CPU/内存 |

### 9.2 是否需要常驻进程

**需要进程池（方案 C），非纯常驻 daemon。**

```
方案 A：按需 spawn（当前）
  heartbeat → spawn CLI → 跑完 → 靠运气退出
  问题：冷启动慢、孙子泄漏、无法管理

方案 B：常驻 Daemon（风险大）
  Daemon 一直活着 → 接任务 → 执行 → 继续等
  风险：Daemon 卡死全线停摆、内存累积泄漏

方案 C：进程池 + TTL（推荐）
  Pool(N) 个热 CLI 进程 → heartbeat 拿 → 跑完归还或销毁
  TTL: 存活 30min 或 N run 后强制回收重建
  好处：无冷启动 + 不无限泄漏 + 单进程死不拖垮全池
```

### 9.3 多进程 + Node 调用丢失/爆发问题

**调用丢失（孤儿进程）**：
- 根因：适配器只等 CLI 退出，CLI 的孙子进程没人管
- **解决（分层）**：  
  - **仓库计划主线**：POSIX 上为心跳路径引入**进程组**并持久化 `process_group_id`，与 **runtime service** 子系统对齐；Windows **文档化 partial**（直接子进程回退 + 未来 Job Object）。详见上文链到的 [PAP-1231](../../../doc/plans/2026-04-08-智能体浏览器进程清理%20agent-browser-process-cleanup.md)。  
  - **本 fork 补强设想**：在 Windows 上 **`resolveProcessGroupId` 常为 null** 时，可用 **WMI / Get-CimInstance 建 PID 树**，run 结束对树执行 kill 并确认死透再收口——与计划文「可观测、可清理」目标一致；实现顺序需与工程评审对齐，避免误杀边界（计划文「Over-killing unrelated processes」风险节）。

**调用爆发（进程洪峰）**：
```
3 agent × heartbeat 10min × maxConcurrent 10 = 峰值 30 CLI
每个 CLI 再 spawn 子进程 → 实际进程数翻倍
```
- 解决：全局进程池上限 + 排队机制 + heartbeat 错峰抖动

**组合爆炸**：
```
spawn-and-forget + 多 agent × 高并发 = 指数级泄漏
```
- 必须从源头堵住：每个 run 都有完整的 PID 树生命周期管理

### 9.4 改进后整体架构

```
上三层（回形针已解决）
  数据库容器化 → 死锁
  智能体配置重配 → 心跳 + 多壳子并行
                    ↓
适配器层（需要重写）
  runChildProcess 周围加：PID树追踪 + 进程池 + 熔断超时 + Windows 原生 kill
                    ↓
                本地 CLI（如 CodeBuddy；同路径亦覆盖 Claude/Codex/Cursor 等子进程树）
```

**工作量不在适配器逻辑本身，在跨平台进程管理的健壮性。**
