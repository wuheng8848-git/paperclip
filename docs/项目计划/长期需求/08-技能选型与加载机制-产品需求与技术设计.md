# 技能选型与加载机制 — 产品需求 & 技术设计

> 版本：v1.1 草稿 | 日期：2026-05-16 · **修订 2026-05-20** | 状态：待评审  
> 作者来源：对话记录 `08 关于技能的选型和配置.md` 整理  
> 适用范围：Paperclip（回形针）公司级技能库 + CLI 自带技能 + Agent 角色技能分配  
> **与 [`执行/045`](../执行/045-Skill懒加载与上下文封顶.md) 分工：** 045 = 已选技能 **少灌字**（懒加载/封顶）；本文 **§3.2** = 已选技能 **少选错/少选多**（角色与任务 **定向指定**）。二者正交，均需落地。

## 1. 为什么需要技能

### 1.1 问题背景

Paperclip 把多个 AI Agent（Claude Code、Codex、Qwen Code 等）组织成"公司"，每个 Agent 有明确角色和职责边界。但存在以下矛盾：

- **模型能力有限**：弱/中模型在复杂任务中容易迷失，需要额外上下文引导
- **全量加载不可行**：把公司全部知识塞进上下文会导致膨胀、角色模糊、技能冲突、难以调试
- **硬编码不灵活**：把技能写死在 agent 指令里，维护成本高，无法按任务动态调整

### 1.2 技能的价值

| 价值 | 说明 |
|------|------|
| **精准上下文** | 每次 run 只拿该轮该用的最小技能包，不塞无关知识 |
| **角色边界** | 技能定义"能做什么 / 不能做什么"，防止越权行为 |
| **可复用 SOP** | 公司技能库是组织记忆，新 agent 入职直接分配技能包 |
| **任务增强** | 对弱模型尤其有效，一次加载精准技能可显著提高一次性完工率 |

### 1.3 核心原则

> **技能不是越多越好，而是找到最好的技能组合——最精简、最实用。**
> **宁多创建角色，也不要搞很多技能然后让所有 agent 都加载。**

## 2. 当前事实（As-Is）

### 2.1 现有机制

| 机制 | 现状 |
|------|------|
| 公司级技能默认加载 | ❌ **不会全量加载**，只带 `required: true` 的内置 Paperclip 技能 |
| 公司技能库作为候选池 | ✅ heartbeat 时把 `companySkills.listRuntimeSkillEntries()` 传给适配器 |
| CLI 技能目录同步 | ✅ Qwen `~/.qwen/skills`、Codex `CODEX_HOME/skills`、Claude `~/.claude/skills` |
| 角色名自动映射技能 | ❌ **目前不支持**，改角色名为"后端开发"不会自动加载对应技能 |
| issue 里 mention 技能 | ✅ 标题/描述/评论里的 `/skills/<id>` 链接会被 heartbeat 扫描，临时追加到本次 run |

### 2.2 技能指定入口

| 入口 | 方式 | 持久性 |
|------|------|--------|
| Agent Skills 页面 | 设置 `desiredSkills` | 永久 |
| issue 描述/评论 | 输入 `/` 选择技能，生成 mention 链接 | 该 issue 后续 run 均生效 |
| 角色名 | 修改角色名称 | ❌ **无自动映射** |

## 3. 目标架构（To-Be）

### 3.1 技能分层

```
┌─────────────────────────────────────────────┐
│             技能四层分类模型                  │
├──────────────┬──────────────────────────────┤
│ baseSkills   │ 每个 agent 都带，极少量，      │
│              │ 稳定规则（协作协议、结账规则）  │
├──────────────┼──────────────────────────────┤
│ roleSkills   │ 按 agent 角色固定带，          │
│              │ 如 dev / researcher / qa / ops │
├──────────────┼──────────────────────────────┤
│ taskSkills   │ 由 issue 标签/项目/文件路径/    │
│              │ adapterType 动态挂载           │
├──────────────┼──────────────────────────────┤
│ forbiddenSkills│ 明确禁止某些角色使用，        │
│              │ 如研究 agent 禁止 company-creator│
└──────────────┴──────────────────────────────┘
```

### 3.2 产品需求：角色与任务定向指定技能（待实现）

**状态：** 待实现（**未**纳入 045 执行范围；落地时另拆执行单，建议母本仍挂本文 + [`08 关于技能的选型和配置.md`](08%20关于技能的选型和配置.md)）

**背景：** 当前 heartbeat 解析 `desiredSkills` 时，主要来源是 agent 页手工配置、`required` 内置技能、以及 issue/comment 里的 `/skills/<id>` mention。**没有**公司级「角色画像 → 技能包」策略层，导致：

- 每个 agent 要靠人工维护 `desiredSkills`，角色一多就漂移；
- 改显示名（如「后端开发」）**不会**自动换技能包；
- 与 **045 懒加载** 叠加前，常出现「候选池很大 + 实际挂载偏多 + 全文进上下文」——简单任务（如单文件翻译）token 虚高。

**需求陈述（董事会口径）：**

> 执行 agent 每轮 run 的技能集合，应由 **角色定向（roleSkills）** 与 **任务定向（taskSkills）** 在控制面 **解析合并** 后确定，而不是依赖模型在 Bash 里摸索、或默认接近「公司候选池全集」。人类在 issue 里 `/` 选技能、或 Board 手工改 `desiredSkills` 仍为 **最高优先级覆盖**。

#### 功能需求

| ID | 需求 | 说明 |
|----|------|------|
| **R1** | **角色定向 `roleSkills`** | 公司可配置：`agentRoleKey`（或雇佣模板 / `capabilities` 等价键，**非**仅显示名）→ 默认 `skillKeys[]`（0–N）。heartbeat 在 `resolvePaperclipDesiredSkillNames`（或后继统一入口）合并进本次 run。 |
| **R2** | **任务定向 `taskSkills`** | 按 issue **标签、项目、`adapterType`、目标路径 glob、事务类型** 等规则追加 0–M 个技能；规则可 per-company 配置，支持禁用某条。 |
| **R3** | **`forbiddenSkills` 硬过滤** | 对 R1/R2/Router 结果做减法：命中禁止列表的技能 **不得** 进入本次 run（即使 issue mention 亦需定义：拒绝 vs 告警仅评论——**默认拒绝并写审计**）。 |
| **R4** | **解析优先级** | 合并顺序与覆盖与 §3.4 一致：`人工 / mention` > `issue·project 固定规则` > `taskSkills` > `roleSkills` > `agent.desiredSkills` 存量 > `required` 内置。 |
| **R5** | **可观测** | 每次 run 可查到 **最终 skill 列表** 及 **来源标签**（`role` / `task` / `mention` / `agent_default` / `required`），便于对照 token 与行为。 |
| **R6** | **与懒加载正交** | 定向指定只决定 **挂哪些 skill key**；045 决定 **每个 key 注入多少正文**。不得用懒加载代替角色策略。 |

#### 非目标（本需求首期不做）

- **不**要求「角色显示名 = 技能包」的自然语言推断（显示名仅 UI）。
- **不**替代 §3.4 **Skill Router**（旁路小模型）；Router 为 **R2 的自动化增强**，排在 **静态 role/task 策略** 之后（见 §5.3）。
- **不**在本需求内改 CLI 厂商自带的技能发现机制，只规范 Paperclip → 适配器 **`desiredSkillNames`** 出口。

#### 验收标准（分期）

| 阶段 | 范围 | 验收 |
|------|------|------|
| **P0** | 公司级 `roleSkills` 表 + 解析合并 + R5 日志/运行清单可见 | 至少 2 个角色（如「开发执行」「CTO 巡检」）配置不同默认包；同一 issue 改指派 agent 后，run 挂载集合随角色变化；**无需**改 issue 标题 |
| **P1** | `taskSkills` 规则（标签/项目/路径） | 例：带 `ui` 标签自动 + `design-guide`；翻译/文档类仅 `paperclip` + `doc-maintenance` |
| **P2** | §3.4 Skill Router | 见 §4.3 |

#### 与现有入口的关系

| 入口 | 定向指定后定位 |
|------|----------------|
| Agent 技能页 `desiredSkills` | **存量默认**；可被 `roleSkills` 覆盖或合并（由 R4 配置决定「合并 vs 以 role 为准」——**默认：role 替换 agent 默认，mention 仍可追加**） |
| issue `/skills` mention | **最高优先级**，用于单次任务增强 |
| 角色显示名 | **不**参与解析；须用稳定 `roleKey` / 模板 id |

### 3.3 技能分配建议（配置范例，非自动实现）

| 角色 | baseSkills | roleSkills | taskSkills（按需） | forbiddenSkills |
|------|-----------|------------|-------------------|-----------------|
| **开发-Qwen** | 协作协议、repo 开发规则 | Qwen 本地适配器、基础开发验证 | adapter 任务→create-agent-adapter；UI 任务→design-guide；文档任务→doc-maintenance | company-creator、research |
| **前端开发** | 协作协议 | frontend/ui 测试、design-guide | 组件任务→ui-component-skill | 后端 DB 迁移 |
| **后端开发** | 协作协议 | API/DB/测试 | adapter 任务→create-agent-adapter | design-guide |
| **研究** | 协作协议 | doc-maintenance、research | — | 任何代码修改技能 |
| **QA** | 协作协议 | test、browser、prcheckloop | — | 实现技能、公司创建 |
| **CTO/大脑** | 协作协议 | project decomposition、risk review、issue triage | — | 大量落地技能（避免自己下场乱改） |

### 3.4 Skill Router（旁路小模型智能分配）

**用户不想在创建任务时显式指定技能**，而是希望有一个旁路小模型根据任务内容自动判断该用哪些技能。

#### 设计

```
issue 创建/更新
  ↓
旁路小模型读取：标题、描述、项目、标签、目标文件、指派 agent/角色
  ↓
对照技能路由表判断：推荐 0-3 个技能
  ↓
写回 issue metadata 或自动追加 skill mention
  ↓
执行 agent heartbeat 时拿到技能
```

#### 技能路由表格式

不给小模型读完整 SKILL.md，而是给一张短表：

| 字段 | 说明 |
|------|------|
| `skillKey` | 技能唯一标识 |
| `适用场景` | 一句话描述 |
| `不适用场景` | 明确排除 |
| `触发关键词` | 用于文本匹配 |
| `适用角色` | 哪些角色可用 |
| `冲突技能` | 不能同时挂载的技能 |
| `最多同用数量` | 限制并发技能数 |

#### 小模型输出格式

```json
{
  "skills": ["qwen-local-adapter"],
  "confidence": 0.86,
  "reason": "issue mentions Qwen CLI, CodePlan API, model 401"
}
```

#### 安全约束

| 约束 | 值 |
|------|---|
| 最多选技能数 | ≤ 3 |
| 允许不选 | ✅ |
| 置信度阈值 | < 0.7 不自动挂载，只评论建议 |
| 敏感技能 | 命中后只建议，不自动挂 |

#### 执行优先级

```
人工显式 / skill mention（最高）
  > issue/project 固定规则
  > Skill Router 推荐
  > agent 默认技能（最低）
```

## 4. 最小可跑版本（MVP）

### 4.0 分期总览（2026-05-20 增补）

| 期 | 能力 | 文档节 |
|----|------|--------|
| **MVP-A** | **§3.2 P0** 角色定向 `roleSkills`（静态表 + 解析） | 先于 Router |
| **MVP-B** | **§3.2 P1** 任务规则 `taskSkills` | 标签/项目/路径 |
| **MVP-C** | **§3.4 Skill Router**（旁路小模型） | 本节 4.1–4.3 |

**045 懒加载** 可与 MVP-A 并行，但 **不能** 阻塞 MVP-A 立项。

### 4.1 范围（MVP-C · Skill Router）

在 **MVP-A/B 已能静态定向** 之后，再建立以下资产：

1. **skills-index 表/文档**：手工维护 10-20 个高质量技能，每个技能只写一行路由说明
2. **小模型分类服务**：用免费小模型做分类输入 → 输出 skill keys
3. **低风险管理**：最多选 2 个技能，低置信度不挂载只评论

### 4.2 输入输出

| 输入 | 输出 |
|------|------|
| issue title + description + agent role | skill keys（0-2 个） |
|  | 置信度 + 理由 |

### 4.3 验收标准

| 标准 | 目标 |
|------|------|
| 技能挂载准确率 | ≥ 70%（人工抽检 10 个 issue） |
| 误挂率 | < 10% |
| 小模型响应延迟 | < 3 秒 |
| 不影响执行 agent 正常 run | ✅ |

## 5. 技能建设策略

### 5.1 数量控制

| 层级 | 建议数量 |
|------|---------|
| 公司级技能库 | 10-20 个高质量技能 |
| 每个 agent 常驻 | 0-2 个 baseSkills + 0-2 个 roleSkills |
| 每次 run 临时追加 | 0-3 个 taskSkills |

### 5.2 技能质量要求

- 每个技能必须有明确触发场景和不适用场景
- 技能之间不能有冲突（或明确标注冲突关系）
- 定期清理无人使用的僵尸技能

### 5.3 建设顺序

1. 先建立 baseSkills（所有 agent 都需要的协作规则）
2. **落地 §3.2 P0：角色定向 `roleSkills`（产品化配置 + 解析，待实现）**
3. **落地 §3.2 P1：任务定向 `taskSkills` 规则**
4. 045：已选技能懒加载 / 封顶（见 [`执行/045`](../执行/045-Skill懒加载与上下文封顶.md)）
5. 最后实现 Skill Router 自动分配（§3.4 / MVP-C）

## 6. 风险与对策

| 风险 | 对策 |
|------|------|
| 小模型判断不准 | 置信度阈值保护，低置信度只建议不挂载 |
| 技能冲突导致行为异常 | 冲突技能表 + 执行优先级机制 |
| 技能库膨胀 | 定期清理 + 数量上限控制 |
| 调试困难 | 每次 run 记录实际加载了哪些技能，便于回溯 |
