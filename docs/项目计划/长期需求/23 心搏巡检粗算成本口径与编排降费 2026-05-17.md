## 1. 定口径 ——「巡检三千里」算什么、不算什么

### 1.1 算什么（粗算，显式假定链）

本条只服务 **Rough order-of-magnitude**，用于：**沟通预算压力、对齐「编排不优化很贵」的长期叙事**。  
不满足财务审计或供应商对账的最终数字。

**假定链 A —— 节拍上界**

- **`heartbeat.intervalSec` 采用实例默认值 `300`**（每 **5** 分钟一次 tick；见 **`defaultTimerHeartbeatIntervalSec`**）。
- **一天**：`86400 ÷ 300 = 288` 次 **timer 节拍**。
- 「每趟都打满节拍且每趟都能落成 **`heartbeatRuns`**」是 **代数上界**；实际控制面还有 **闸门**（见下文「不算」）会砍掉部分 tick。

**假定链 B —— 单笔「巡检等价开销」占位**

- **锚定**：一次 **巡检型心跳 run**（长系统说明 + Paperclip/task 上下文 + CodeBuddy DeepSeek/OpenAI-compat 链路）在实践中可与 **¥0.3 / 趟**量级同档（取自 **厂商控制台原始凭证 ↔ 单次 run stdout 会话汇总用量**已对平的样本）。
- **不主张**：天下所有 run 恒定三毛；亦不主张 **`total_cost_usd` 或未解析 `usage_json` 的账本**与该样本一致。**粗算占位**只认：**「这趟形态」可当巡检档参考点**。

**合成 —— 「无优化单月巡检」数量级**

- `288 趟/日 × ¥0.3 × 30 日 ≈ ¥2592` → **口语化可取「巡检约三千人民币/月量级」**，指 **单个全时节拍巡检主体**在上述两条假定同时成立时的 **API 开销顶格想象空间**。
- **多线并行**：若同一治理域内 **多套独立会话/instance**（看进展、读汇报、验代码等各跑一轮）同日并发，账单侧近似 **线性叠乘**——「请一个分身就多一档量级」同属本条粗算适用域。

### 1.2 不算什么（请勿误用本条）

| 不适用 | 说明 |
| --- | --- |
| **精确成本** | 实际价目、币种、KV 档位、会话截断、`usage_json`/`cost_events` 是否落库——以 **供应商与控制面已实现字段**为准；细表见 **[015]**。 |
| **人事/管理费** | 不含人类工资、证照、法务、设备等。 |
| **产出度量** | 本条 **不立** KPI；「值不值」须在 **事务产出、风险规避、工时替代**等业务侧另表。 |

### 1.3 与现有闸门的关系（为什么「三千」多数是上限遐想）

定时路径上，已实现 **「timer + 在公司内无可调度经办事项则不建 `heartbeatRuns`」** 等闸门（对齐 **[007]**），故 **287 次空想 tick 不会全部变成真 run**。  
**粗算括号**仍有用：在未调优或未配置闸门的历史配置、或为 **CEO「无单也需清醒」等产品例外**留白时，**节拍上界仍是压力测试用的硬天花板**。

### 1.4 模型档位与价目敏感度假定（仍为粗算，不比价、不背书官网）

本条 **不立** 「同期能力最强」，只采纳协作侧共识：**现行 DeepSeek/`deepseek-v4-pro`（经 CodeBuddy 兼容链）定位为「可用与成本之间的均衡选型」**，在**同一够用档**内取 **相对较低价位**的路线。

**折扣窗口（协作占位）**：社区/控制台口述 **折价活动于 `2026-05-31` 结束一段**——后续即便仍有折，也常落在 **约三点五～四点五折档**（较现行口述 **二点五折档**等价系数上移）、未必立刻回到标价。但无论档位移如何，**价目上浮与己方编排节流可能同时发生**，粗算上可暂 **视作相互对冲**：月度 **「~¥3000 / 巡检头」仍可当心智锚**，直到 **`usage` 实测**重写占位单价。

**风险句（写死进长期需求）：** **若不持续推进编排节流**，一旦供应商 **实质涨价**，**同上节拍下的现金支出会立刻变敏感**，「三千封顶遐想」会很快 **低估了真账单**——故 **降费需求不是 optional nice-to-have，而是与外部环境同频的生存项**。（与 **[015]** 「累计报价 / 账本不对称」并存：价目在变，编排也要变。）

## 2. 长期需求 —— 如何把「巡检三千」往下压（编排与方法）

条目按 **不靠猜模型降价**、优先 **编排与工程**拆解；可分阶段落入执行单与技术债。

### 2.1 节拍与唤醒策略（最先动刀）

- **拉长 `intervalSec`**：业务允许则 **十分钟 / 十五分钟**量级直接腰斩 tick 上限。
- **默认策略**：沿用技能与实例约定——**仅为真需要周期性清醒的角色启用 timer**；余人 **指派 / 按需 Wake**。（参见 **`.agents/skills/paperclip-create-agent`**「默认关掉定时心跳」叙事。）

### 2.2 合并重复工作与分裂职责

- **禁止同源上下文在多路并行 run 无脑复制**：巡检、汇总、深挖应 **分拆职责**或使用 **单次 run 内的阶段化**，避免「同一简报被三个头像各啃一遍前缀」。
- **读汇报 vs 扫代码**：只读链路走 **更小 payload / 结构化 API**，避免整块对话史反复灌入模型——由 **产品与适配器契约**逐步实现（可先文档后实现）。

### 2.3 事务与指派层面的 dedupe

- **同一事务、同一短时窗**：编排层应收敛 **谁先 wake、是否需要接力**，减少 **指派抖动**带来的 **连串 wake + 短命 cancel**。
- **与审阅链路对齐**：已进入稳定 `in_review` 且无新信号时 **抑制重复巡检**。（结合现有 **`execution_policy` / continuation** 实践逐步收紧。）

### 2.4 会话、技能与会话前缀

- **技能包裁剪**：巡检档仅加载当期角色所需技能，不全量堆叠 CLI 说明书。
- **系统说明瘦身**：重复的「全局宪法」在多轮中与 **轮转、摘要**策略配合，避免单行心跳 **89 之类步数 × 前缀爆炸**单靠缓存挡账。（缓存命中 **省钱未必省条数**。）

### 2.5 可观测与账务闭环（保证粗算可被事实收紧）

- **解析失败仍为真成本**：沿用 **[015]** 结论——`**codebuddy_parse_error**`、`**usage_json`** 空缺 **不豁免**提供商扣费。
- **产品债**：减少对 **stdout-only** 的依赖，保证 **规范化计量**在成功路径上稳定落 **`usage_json`**，使「巡检单价」可被 **实测滚动均值**置换掉「三毛占位」。
- **对账**：继续用 **`pnpm usage:reconcile`** 等在 **工时层**校对「体感三毛」与实际档位。

## 3. 验收（长期，分期）

本条 **不立一次性验收**；每期落地应绑定具体执行单（如闸门扩展、节流策略、指派 dedupe、`usage_json` 鲁棒性等）。
