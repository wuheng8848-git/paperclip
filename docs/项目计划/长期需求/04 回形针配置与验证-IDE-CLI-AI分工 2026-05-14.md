## 1. 三层能力：本来就有 vs 要补 vs 怎么变成「一次跑通」

### 1.1 回形针本来就有 API

控制平面以 **`/api`** 暴露 REST 能力（健康检查、公司、智能体、事务、runs、活动等）。**真值以仓库 `AGENTS.md` / `doc/` 与实现为准**——本文只定**分工与意图**，不写端点大全。

### 1.2 缺口要我们自己补

API **盖不到**或**用起来太碎**的地方（例如：一键健康矩阵、批量对账、本地与 Board 状态差异报告、某适配器专项探测），用**小脚本、薄封装、或 IDE 任务**补齐。**原则：**少造轮子 duplication；能调用官方端点就先调，再只在缺口上写 glue。

### 1.3 「能跑一次就好」的最佳实践长什么样

可复用打法尽量收敛成同一套路：

**脚本（或可重复命令） + 可调 API + 明确执行路数**（从哪一步进到哪一步、成功判据是什么）。

这样**人**只触发或验收，**不靠**每次人肉点遍 UI 与拼接 curl。

## 2. 快速测试：谁来做「智能体配置 + CLI 壳子 + 参数 + 角色」

### 2.1 不默认由我穷举

**智能体配置、CLI 壳子、参数矩阵、角色模版**——若一件件由我手工穷举，既慢又易过时。更合理的分工：

- **AI（或半自动清单）**承担「拉通 / 对比 / 节选」：类似**模型大全**——能穷举就穷举，不能就**精选** + 说明取舍理由。
- **我自己没跑过**的路径：先**引用别人已有的最佳实践**（上游文档、社区、仓库内 [`最佳实践/`](../最佳实践/README.md)），再小步验证，避免从零发明流程。

### 2.2 与 IDE / CLI 的关系

- **IDE**：适合放「读配置、起服务、跑脚本、看差异」的**固定入口**（任务、规则、skill），减少上下文丢失。  
- **CLI**：适配器与心搏的真实执行面；**验证**应能落在**短命令序**上，而不是每次长篇交互。

### 2.3 和环境/容器的关系

**搞容器、抠环境、钻研 CLI** 属于**没有办法的办法**——在自动化与 API 不够用时才由我深蹲。常态下目标是：**环境是背景板，业务是主角**。
