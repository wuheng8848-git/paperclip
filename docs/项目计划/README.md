# 项目计划（Routic 式工作方式）

本目录为**仓库内协作**（任务、探查、验收证据）；**不**按对外资料库口径维护。对外向文档见 `docs/` 其余部分及 `AGENTS.md`「`docs/` 域文书」条目。

本目录与 `01.项目需求说明.md`、`00.项目任务清单.md`（含**验收标准**）、`执行/`、`探查/`、`最佳实践/`、`长期需求/` 配套使用。下面是**固定顺序**，不按顺序容易返工。

1. **任务不清楚 → 先讨论**（与人或 Board 对齐目标、边界、成功标准；不要猜任务）。
2. **讨论仍不清楚 → 先探查**：本地 **代码 + 仓库文档（如 `doc/`、`AGENTS.md`）+ [`index.md`](index.md) 全量链接**；必要时 **上网检索**补上下文。
3. **需求落定 → 再执行**（写进需求说明或任务单后再动代码/动环境）。
4. **执行结束 → 收口边界**：拆 **下一单** 承接剩余工作，或把新共识 **记回需求**；**回写 `最佳实践/`**（可复用的路径、命令、坑）。
5. **验证结论与证据 → 写进对应 `执行/` 任务单**；**验收标准**见 `00.项目任务清单.md`；**残留风险**记在 `01.项目需求说明.md`「残留风险」节。
6. **`00.项目任务清单.md`** 顶部「状态｜任务单」**表**只列状态与链接（不写执行过程）；**验收标准**、规则与「验证证据存放处」写在同文件专节；任务标题与执行过程一律放在 `执行/` 各单内。

**关于 `执行/` 文件名里的三位前缀（如 `001-`、`022-`）：** 只表示**任务单进入仓库的时间线**（建档先后），**不表示业务优先级**，也不当作路线图排序；是否优先做哪一单，以讨论结论、`01.项目需求说明.md` 里的开放问题以及 Board 为准。

**任务单状态语种：** `执行/*.md` 文首 `status:` **只使用中文**（与 `00.项目任务清单.md` 所列五种取值**逐字一致**），不写 `todo` / `active` / `done` 等英文枚举。
