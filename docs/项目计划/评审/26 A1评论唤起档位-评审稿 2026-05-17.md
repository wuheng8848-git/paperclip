**和「默认轻」怎么不打架：**

**亲历（约束实现，不是牢骚就算了）：**

这和「标签取代轻/重」无关；根子是 **每轮叫醒仍在重复膨胀全线程 / 全 Skill / 长摘要**，和 **041 / 045（懒加载、去重、封顶）** 要砍的是同一件事。**label 规则本期不做**，先把 **默认轻 + 少重复灌 + 续会话优先** 做实。

## 第 4 条白话：两种存法到底啥意思？

事务上要能写一句：**这条单评论叫醒用哪一档**（覆盖公司默认）。技术上有两种落点：

**A — 塞进现有的 `executionPolicy`（一大坨 JSON 里加一小段）**

- **人话：** 「执行策略」里本来就有阶段、审批、monitor 等；把「评论叫醒档位」也塞进去，**流程相关开关都在一个兜里**。
- **好的地方：** 少一个新顶栏字段名；改 issue 往往已经要动这一块。
- **费劲 / 风险：** 那个 JSON 已经肥，再加字段 = schema、校验、接口整条链都得跟；**非流程的人**打开单据不一定一眼看到「叫醒档」。

**B — 事务上单独字段（例如顶层一列 `commentWakeTier`）**

- **人话：** 跟「现在走到哪个审批阶段」**分开放**，叫醒档位 **单独一项**，一眼能找。
- **好的地方：** 语义清爽，和你说的「少绕弯」一致。
- **费劲 / 风险：** 多一次 **库列或 API 字段 + 迁移**；要和 `executionPolicy` **约定谁覆盖谁**，避免两处打架。

**已选 B（2026-05-17）：** 事务 **`issues` 顶层单独字段**（实现名如 `commentWakeTier`）。**不是**每条评论一条字段 —— 语义是 **整张单** 的评论叫醒用哪档；**也不是**塞进 `executionPolicy` JSON。

## 第 5 条白话：label 是啥？「自动选档」是啥？

**label 是啥**  
就是事务上贴的 **标签**（例如 `urgent`、`backend`、你们自己起的「轻问」「重活」）。在本仓库里事务有 **labels / labelIds**，跟标题、正文一样是事务属性。

**自动选档是啥**  
**不**靠人每条评论前手动选档，而是 **先配规则**，例如：

- 打了标签 **「轻问」** → 评论叫醒 **自动用轻档**
- 打了标签 **「重活」** → **自动用满档**

第一期可以 **完全不做** 这条规则引擎：只做 **公司默认轻** +（若你做了第 4 条的事务覆盖）**手填这一条单用哪档**。  

**已表态（本期）：不要。** 不指望「有标签就自动等于轻档/重档」——**真瓶颈**是评论路径 **重复膨胀上下文**（见上「亲历」），不是缺一个 label 映射表。
