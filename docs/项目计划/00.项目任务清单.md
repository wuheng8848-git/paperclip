## 验收标准

1. 需求边界清楚。  
2. 变更面可控（本轮主要为控制面 API / Board 状态，无产品业务大改）。  
3. 对外行为有描述（routic：分配 → 自动签出/唤醒 → agent 执行 → issue `done`）。  
4. 各任务在对应 `执行/*.md` 任务单内已写清可复查的验证证据（命令、时间、结论、链接）；**不**另建与 `00` 主表对行的总验收证据表。  
5. 残留风险与未跑检查已写入 [`01.项目需求说明.md`](01.项目需求说明.md)「**残留风险**」节。  

## 验证证据存放处

**验证证据只记在任务单里**：每条 `执行/` 任务单正文（或该单指向的 `探查/`、`最佳实践/` 附件）承担「做了什么、怎么证明」；**不在**本主表用「日期 / 检查 / 结果 / 备注」式逐行对账。

## 规则

### 1. 状态词与双处一致

- 本表「状态」列与对应 `执行/*.md` 文首 YAML 的 **`status:`** 必须**逐字相同**，且只能是上文「状态取值」中的五种之一。  
- 任务单若缺 `status:`，先在任务单补全，再记入本表。  
- 链接里文件名前的三位数字（如 `001-`、`022-`）**只表示任务单进入仓库的时间线**（建档先后），**不代表业务优先级**；是否优先做哪一单，以讨论结论、`01.项目需求说明.md` 开放问题及 Board 为准。

### 2. 更新顺序（先任务单，后主表）

改某一行的状态时：**先**在对应 `执行/` 文件中改 `status:`，**再**改本表同一行的「状态」。禁止只改本表不改任务单，或两处写成不同字。

### 3. 「已完成」闸门

**不得**在对应 `执行/` 任务单内没有可复查验证证据的情况下，把本表该行标成「已完成」。证据写法见上文「验证证据存放处」。

### 4. 执行中 / 待审：禁止硬改核心字段

工单或任务单处于「**进行中**」或「**待审**」，或已有未收尾的 live run / 明显执行中状态时：**不得**用 Board、API 或其它方式去改指派、项目、工作区、标题、描述等核心字段。须先与人对齐，再结束或挂起当前执行，或**另开新工单**承接变更后的目标。

### 5. 转指派 / 项目 / 工作区须留痕

改指派、换项目或工作区时，须有可追溯说明（Board 评论或新工单正文写清原因与边界）。禁止仅靠一串 PATCH 清字段、改绑却不留记录。优先用**新子单 / 迁移单**承接，避免在运行中硬改父单。
