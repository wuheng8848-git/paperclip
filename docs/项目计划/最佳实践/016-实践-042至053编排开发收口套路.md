**用途：** 统一 **`执行/042`** ～ **`执行/053`**（及同类「动控制面 / 心跳 / 适配器 / 看板」）的协作节奏，避免没摸清边界就改、改完说不清验在哪。**不取代** [`README.md`](../README.md) 里「执行单从哪来」等总则；本条是**这一批工单加码的最低仪式**。

**本批来源：** **`042`～`053` 全部从计划母本拆出**——**[`长期需求/24 评论唤起过载与编排分层改版计划 2026-05-17.md`](../长期需求/24%20评论唤起过载与编排分层改版计划%202026-05-17.md)**（文中 **§「执行单拆分（041–053）」** 表）。[`01.项目需求说明.md`](../01.项目需求说明.md) 可作横向约束，但**拆单真值以 `24` 为准**。每条执行单 **母本 / 拆单出处** 须链回 **`24` 对应 §（A1～H）** 及本表行。

**整轮收口（本批全办结或宣布阶段结案后必做）：** 在 **`24`** 内 **[实现与需求对齐（整轮回写）](../长期需求/24%20评论唤起过载与编排分层改版计划%202026-05-17.md#impl-alignment-24-round)** 补表：每条 **A～H** 与 **`041`～`053`** 最终是 **已满足 / 部分 / 未做+原因**，并指向执行单里的验证证据；避免「代码散了但需求母本仍是空头支票」。

## 1. 固定顺序（含义：每一步给后面挡返工）

| 步 | 做什么 | 给谁看 | 产出放哪 |
| --- | --- | --- | --- |
| **① 探查** | 现状代码路径、数据从哪来、**做 / 不做**的边界、依赖谁 | 干活的人 + 审阅的人心里先有地图 | 新建 [`探查/`](../探查/) **`NNN-探查-…md`**；**小额**可写在对应 **`执行/NNN-…md`** 的「探查结论」节，但**边界不清时禁止省独立探查** |
| **② 技术设计** | 准备改哪些层（`server` / `packages` / `ui` / 适配器）、契约怎么变、风险与回滚 | 人类要知道「你们准备怎么改」 | [`长期需求/`](../长期需求/) 下 **`…技术设计….md`**，或 **`执行/`** 内独立「技术设计」节 + 互链 |
| **③ 评审** | 有没有**开口选项**（产品分叉、架构二选一、兼容策略未定）？ | 有开口 → 人类拍板后再动刀 | 可另写 **`…评审稿….md`**（参照 041 系列），或 Board / 对话结论**写回设计第一节「已定案」**。**无开口**须在执行单写明：**「无开口选项，按技术设计直接实施。」** |
| **④ 技术执行** | 写代码、迁库、改 UI | — | 遵守仓库总则（公司边界、契约同步、[`AGENTS.md`](../../AGENTS.md)） |
| **⑤ 测试** | **复杂 / 易回归**：**先测后码（TDD）**——红灯 → 实现 → 绿灯 | 评审者复查 | 默认可执行：`pnpm exec vitest run …`；路径多在 **`server/src/__tests__/`**，亦有 **`packages/*`** 内测试；**在执行单写清跑了哪条命令** |
| **⑥ 回写执行单** | 状态、人话落地说明、**测与码在哪儿** | 以后接手的第一个人 | 见下文 **§3**；[`00.项目任务清单.md`](../00.项目任务清单.md) 与文首 **`status:`** 保持一致 |
| **⑧ 回写需求母本 `24`** | **`041`～`053` 本批整体收口后**，对照计划清单 **A～H** 在 **`24`** 填 **「实现与需求对齐」** 表（已满足 / 部分 / 延期及原因 + 执行单链） | 审阅母本的人、后续排期 | **[`长期需求/24 …`](../长期需求/24%20评论唤起过载与编排分层改版计划%202026-05-17.md)** §「实现与需求对齐」 |
| **⑨ 结案文件名（可选）** | 与 041 一致：结案后文件名加 **`-已完成`**，防与进行中单混淆 | — | **更名时同一轮**修 [`index.md`](../index.md)、主表、互链（见 [`023-文档更名删除修链`](../执行/023-文档更名删除修链.md) 精神） |

**「复杂」的实操约定：** 多模块耦合、心跳 / 唤醒 / 排队、新 DB 列或对外行为变化 → **按 TDD 做**；纯文案或单车文件小修可酌情省略，但须在执行单一句话说明「为何未起新测」。

## 2. 开口选项 vs 直做

- **开口选项**：多种做法在**产品体验、数据语义、兼容旧实例**上长期不可兼得，必须人类选一边（或分阶段）。**未完成评审前**，执行单 `status:` 用 **「待办」** 或 **「待审」**（与主表一致），**不大面积落代码**。  
- **无开口**：技术设计里已经把关键取舍写死，评审栏写 **「无开口」** → 可进入 **「进行中」** 按设计实施。  
- **与主表「待审」的配合**：需要人类显式看过设计/评审结论再开工时，主表与执行单可同时用 **「待审」**；开工前再改为 **「进行中」**。

## 3. 执行单回写：必须含什么（验收的人只看单子也能续）

结案或阶段收口时，对应 **`执行/NNN-….md`** 须补全（可粘贴 §4 骨架）：

1. **`status:`** 与 [`00.项目任务清单.md`](../00.项目任务清单.md) **同一行一致**。  
2. **验证证据**：日期、**实际跑过的命令**（勿写「待补」占坑）。  
3. **落地说明（人话）**：不堆内部变量名，说清「用户/操作者能感到什么变了」。  
4. **代码与测试索引**（强制）：  
   - **代码**：列 **`server/...` `packages/...` `ui/...`** 等路径（或按功能点分行）。  
   - **测试**：测试文件路径 + 推荐的一条 **`pnpm exec vitest run …`**（或 `pnpm test` 若全 suite 更合适）。  
5. **残余与后续**：未写对外 SPEC、未跑全仓 typecheck 等**如实写**，避免假完成。  
6. **（仅本改版批次）** 单条执行单结案**不代替**母本收尾：`042`～`053` 阶段整体结案后，仍须在 **[`24`](../长期需求/24%20评论唤起过载与编排分层改版计划%202026-05-17.md#impl-alignment-24-round)** 填 **实现与需求对齐** 表。

**对外产品文档**：若本轮触达操作者可见行为，还须遵守 [`AGENTS.md`](../../AGENTS.md) 文书闸（专文是否在 `doc/` / `docs/` 或仅计划区，按闸判断）。
