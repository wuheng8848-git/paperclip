# 014 — 实践：事务审阅、审批与多人接力编排

**用途：** 在给 Paperclip 派活、收口时对齐口语 —— **审阅**、**审批**、**经办**、**接力** 各指什么；减少「以为 done 就能换手」或「把评审当成门禁」这类误会。

**返回：** [`README.md`](README.md)

---

## 1. 词义分界（建议口径）

| 口语 | 建议对齐 | 说明 |
| --- | --- | --- |
| **审阅（review）** | 人读 diff / 读 comment / 给意见 | **不等于**系统门禁通过；可在 issue `in_review` 阶段并行多人阅读 |
| **审批（approval）** | 预算门控、`pending_approval`、受管操作那类 **硬闸门** | 以 SPEC / 产品配置为准；未放行则执行平面不得擅自推进 |
| **经办（assignee）** | issue 当前 **assigneeAgentId** 所承载的推进责任 | Heartbeats、timer 闸门等多以「有无可归因经办 + 状态」判断 |
| **接力（handoff）** | 改 assignee、拆子 issue、或明确下一棒 comment | 须在 Board / API 上 **留下指派事实**，避免口头移交 |

---

## 2. 验收心智（编排侧）

1. **验收对象**：既是「代码对不对」，也是「Paperclip 状态与指派是否反映真实责任」——后者便于后续自动化与审计。  
2. **`done` 之前**：确认 **经办链条**已写明（谁提交、谁审、谁合并），避免 done 后仍被 timer / recovery 拉回泛唤醒。  
3. **多人并行**：适合拆 **子 issue** 或多 assign 场景用评论约定边界；单 issue 多角色抢同一终态时先对齐 **[009-董事会派单协议](009-实践-董事会派单协议.md)**。  
4. **证据**：复杂收口请在对应 **`执行/`** 单写 API / 时间 / issue slug；本实践不单建证据表。
