## 1）先建立心智模型

| 概念 | 人话 | 验收时怎么用 |
| --- | --- | --- |
| **issue（事务）** | Board 上一条任务 | 看 `status`（`done` / `in_progress` / …）是否 **符合你定义的完成** |
| **run（运行）** | 某次心跳真正跑起来的一趟 | 看 `status`（`succeeded` / `failed` / `cancelled` / `running`） |
| **run 成功 ≠ 事务完成** | 适配器进程正常退出，不代表业务交付够 | 两者 **分开验**；董事会口径见 [009](009-实践-董事会派单协议.md) §6 |
| **succeeded** | 运行 **正常结束** | P0 闭环常见闸门：**不要**把应成功的 run 留成 `cancelled` |
| **cancelled** | 运行被 **控制面或对账** 作废 | 事务已 `done` 但 run 仍 `cancelled` → 优先怀疑 **关单对账** 类修复是否回归 |
| **failed** | 适配器或业务路径报错结束 | 看 `error` / `errorCode` / `exitCode` |

**UI 入口真值：** 侧边栏 **工作 → 运行清单**（路由 `/{公司前缀}/runs`），勿写「编排注入」——见 [023](023-实践-Board界面中文口径与文档对照表.md)。

## 2）验收闸门（最小通过条件）

按改动类型勾选；**未列出的专项**（如 059 旁路、050 唤起归因）仍挂在对应 **执行/** 单 §5，本文只给 **run 通用底稿**。

### 2.1 环境（约 1 分钟）

| # | 检查 | 通过标准 |
| --- | --- | --- |
| E1 | API | `GET /api/health` → `"status":"ok"` |
| E2 | 实例 | 端口 / 公司与你要验的 Board 一致（例：`http://127.0.0.1:4100`） |
| E3 | 部署模式 | **`local_trusted`** 下 curl 常可读 run；**`authenticated`** 需 Cookie / Bearer（同 [002](002-实践-事务运行记录API取证路径.md) §0） |

### 2.2 触发一次可观测的 run

| # | 检查 | 通过标准 |
| --- | --- | --- |
| T1 | 有 attributable 工作 | 事务已指派智能体，或 `npx paperclipai heartbeat run --agent-id …` 手动 invoke |
| T2 | 能拿到 **runId** | Board 运行清单、或 `GET /api/companies/{id}/heartbeat-runs?limit=1`、或 CLI 输出 |
| T3 | 任务边界清楚 | 验收用事务 **描述写清「做到哪算过」**（例：ROU-7 只输出时间 + 评论 ok + `done`） |

### 2.3 运行中（`status=running`，可选但推荐）

不必读懂 NDJSON；只确认 **「还在动」**：

| # | 看什么 | 通过标准 |
| --- | --- | --- |
| R1 | 日志增量 | **运行清单 → 运行详情 → 运行记录**：行数/字节在涨；或 `GET …/heartbeat-runs/{id}/log` 的 NDJSON 行 `ts` 向前 |
| R2 | 控制面进度 | `GET …/heartbeat-runs/{id}` 的 **`lastOutputSeq` / `lastOutputBytes` / `lastOutputAt`** 批次推进（允许批量 flush，不必每 chunk 一条 API） |
| R3 | 旁路 UI（若产品已做） | 摘要区 **「距上次输出」** 在吐字时回落、停吐后递增（例：[059](../执行/059-CodeBuddy流式日志与存活旁路观测.md)） |

### 2.4 跑完后（必做）

| # | 看什么 | 通过标准 |
| --- | --- | --- |
| F1 | **run 终态** | 预期 **`succeeded`**（测失败路径时则预期 `failed` + 可读 `errorCode`） |
| F2 | **exitCode** | 成功路径通常 **`0`**；非 0 须有解释 |
| F3 | **日志可复查** | `logBytes` > 0；UI **原始** 模式可见 **`HH:mm:ss.mmm`** 时间戳（有则加分，无则记缺口） |
| F4 | **resultJson / 摘要** | 适配器有结构化结果时，运行详情 **执行回传** 或 API `resultJson` 与任务一致（不要求人类读全量 stdout） |
| F5 | **issue ↔ run 对账** | 若事务已 `done`：对应 run **不应**仍 `cancelled` 且带「事务已进入终态…为对账已取消」类文案（[060 P0](../执行/060-2026-05-20-验证闭环与编排首日执行.md) 回归点） |

### 2.5 事务层（业务验收，与 run 层分开）

| # | 看什么 | 通过标准 |
| --- | --- | --- |
| I1 | 事务 `status` | 符合任务单完成定义（常为 `done`） |
| I2 | 评论 / 交付物 | 线程里有 **可核验** 中文摘要或文件路径（非整段日志粘贴） |
| I3 | 董事会 / 009 | 需要人类拍板时，**run succeeded  alone 不够** |

## 3）推荐顺序（默认 SOP）

```text
环境(E) → 建/选事务(T) → 指派 + 运行 → [运行中 R 可选] → run 终态(F) → issue 对账(I) → 证据落盘
```

### 3.1 API 最小集（跑完后 3 条）

```http
GET /api/health
GET /api/issues/{ROU-n 或 uuid}
GET /api/issues/{ref}/runs
GET /api/heartbeat-runs/{runId}
```

**`--base`：** 与实例一致（例 `http://127.0.0.1:4100`）；**不要**写成带 `/api` 的根（见 [002](002-实践-事务运行记录API取证路径.md) / [004](004-实践-事务心跳与僵尸run排障.md)）。

**列表字段：** `GET /issues/…/runs` 返回 **`runId`**，不是 `id`。

### 3.2 Board 最小路径

1. **工作 → 运行清单** → 点目标 run（或 `/runs/{runId}`）。
2. **运行记录** 标签：摘要（状态、时长、适配器）、转写 / **原始** 日志。
3. 需要 prompt / 唤起时再看 **输入编排**、**最终提示词**（非 P0 必验，除非执行单写明）。

### 3.3 脚本聚合（同一事务多条 run 时）

```sh
pnpm issue:forensics -- --company <companyUuid> --issue ROU-7
pnpm issue:forensics -- --company <companyUuid> --issue ROU-7 --run-id <uuid>
```

深查 API 顺序与参数见 [002 §1.4](002-实践-事务运行记录API取证路径.md)。

### 3.4 CLI 触发（本地 smoke）

```sh
npx paperclipai heartbeat run --agent-id <uuid> --api-base http://127.0.0.1:4100 --source on_demand --trigger manual
```

注意：若同时存在 **assignment 自动唤醒**，可能并行两条 run；验收时 **以事务上下文那条为准**（看 `contextSnapshot.issueId`）。

## 4）判读速查（验不过时先归类）

| 现象 | 先怀疑 | 转哪篇 |
| --- | --- | --- |
| run 长期 `running`、无新日志 | 僵尸 / 进程丢 | [004](004-实践-事务心跳与僵尸run排障.md) |
| `process_lost_retry` / `retryOfRunId` | 丢进程重试链 | [002](002-实践-事务运行记录API取证路径.md) + 探查 004/007 |
| `succeeded` 但 token 离谱 | 多 turn / 肥工具 JSON | [018](018-实践-排障-run与token多轮.md) |
| 评论 / 摘要乱码 | 编码链路 | [021](021-实践-乱码根因与修订方案.md) |
| `done` + run `cancelled` | 关单对账 | 对照 [060 P0](../执行/060-2026-05-20-验证闭环与编排首日执行.md)；查近期 merge |
| 429 / 适配器秒失败 | CodeBuddy 路由 / 配额 | [007](007-CodeBuddy-火山引擎端点与配额.md) |

## 5）证据落盘模板（写进 `执行/` 或探查）

**表行（推荐）：**

| 项目 | 日期 | 公司 | issue id | run id | 结论 |
| --- | --- | --- | --- | --- | --- |
| P0 run 闭环 | YYYY-MM-DD | routic | ROU-n · `uuid…` | `uuid…` | **通过/失败**：run `succeeded` + issue `done`；一句话现象 |

**命令摘要（按需粘贴）：**

```text
GET /api/health → ok
GET /api/heartbeat-runs/{runId} → status succeeded, exitCode 0, lastOutputSeq N, logBytes M
GET /api/issues/{issueId} → status done
```

专项验收（059 旁路、050 唤起归因等）在结论列 **显式写「非本文 P0」或链执行单**，避免和通用 run 闸门混为一谈。
