# 015 · 实践 · CLI 累计计量与成本控制面台账口径

> **用途**：说清楚 **无头 CLI / 宿主链路上报的用量与金额** 往往是 **会话或更长周期累计**，以及 **Paperclip 当前账本各字段到底按「单次」还是「扣减后用增量」**。  
> **读者**：要自己解释成本、或对账控制台明细与 `heartbeat_run` 时的人类与助手。  
> **真值仍以代码为准**；本篇只沉淀协作结论与排障语义，不写产品承诺。

---

## 1. 上游常见形态：累计 vs 本轮

很多编码 Agent / CLI 在每轮结束时回传的 **`usage`**（各家字段名相近）有时是：

- **本轮 HTTP 调用**的 token；
- **同一会话内至今累计**的 token（下一轮报 30 万不代表「这轮吃了 30 万」，而是「从会话开始到现在总计 30 万」）。

**金额 `total_cost_usd` / `costUsd` 也存在同类风险**：若是 **累计账单**，却把每一轮 heartbeat 完结时的数字 **整块记进账本**，就会在控制面上 **重复计同一笔前缀成本**——除非上游约定每次回报的是 **本轮增量**。

社区/上游的常见提醒可以概括成：**「看见的数可能是桶里水位，不是这一瓢」**；要本轮真值需在 **同一时间轴**上对 **上一轮回报**做一次 **算术差（扣减）**。

---

## 2. Paperclip 对 Token：已有一层「会话级扣减」（增量）

在服务层，当 **存在可解析的会话标识**（`persistedSessionId` / legacy session 等链路）且本轮适配器 **`usage`** 非空时，会尝试：

1. 找到 **同一 `agentId + session`，在本条 `run` 之前最近一次** 完结 run；  
2. 从其 **`usageJson`** 读出上一次的 **`raw*` 累计分量**（`readRawUsageTotals`）；  
3. 用 **`deriveNormalizedUsageDelta`** 将 **本轮 raw** 减 **上轮 raw**，得到更接近 **本条 run 生命周期内增量** 的 token 计数；
4. 写入本条 run 的 **`usageJson`** 时，可同时保留 **`rawInputTokens`** 等与 **`usageSource: "session_delta"`**（表示「token 分项里有一部分是按差分推出来的」）。

**仍须注意**：没有会话、首轮、或上一轮读不到 **`usageJson`** 时，**增量逻辑可能退化成「整块 raw 照单全收」**；不同适配器对 `usage` 的语义也未必一致。**读懂一条 run 时请同时看 `raw*` 与 `usageSource`，不要单靠「单字段即本轮」。**

**与会话轮转的关系**：服务端做 **会话压缩 / 轮转**（例如触及 **raw input** 阈值）时，比较的也是 **会话侧累计标尺**一类的量；这和「账本 token 分项是否差分」是同一套「累计世界」里的不同告警线。

---

## 3. Paperclip 对美元成本：默认「适配器报多少记多少」（未再扣上一轮累计）

账本路径上，**`costUsd`**（CodeBuddy 等适配器取自结果 JSON 的 **`total_cost_usd`** 等）在服务层 **`normalizeBilledCostCents`** 后直接累加进 **`cost_events` / runtime**，**没有像 token 那样的「减上一条同会话 run 的 cost」步骤**。

因此：

- 若 **`total_cost_usd` 语义是「本轮」或「本条请求」**：台账与直觉一致；
- 若 **`total_cost_usd` 语义是「本会话累计」**：每来一次 heartbeat 就会把 **整桶累计**再记一笔，**存在系统性高估**，需要 **上游改成增量**，或产品在服务端 **补差额逻辑**（当前仓库未内置）。

这与 **token 分项**上已经存在的 **会话差分** 是 **不对称** 的：**读账本时别把「成本控制面总金额」等同于「已通过差分校准的真值」，除非你已经验证过 CLI 币种与累计语义.**

---

## 4. CodeBuddy × DeepSeek（本 fork 语境下的提醒）

`codebuddy_local` 把 **`costUsd`** 设为解析结果里的 **`total_cost_usd`**，**Paperclip 不在这里按 DeepSeek 中文官网价表重算**；币种字段名写的是 **USD**，实际背后可能是厂商/Tencent CodeBuddy 自己的换算。**供应商控制台（USD / CNY、命中/未命中）与 Paperclip `cost_events` 不会自动逐项勾稽**。

## 5. 对账脚手架（不写账本）

为方便 **三条口径并排**（编排可见拼装、适配器入账、控制台/API 导出），仓库提供只读脚本，例如：

```text
pnpm usage:reconcile -- --run-id <heartbeat_run_uuid> ...
```

可加 **`--vendor-json`**（单次接口 `usage`）、**`--deepseek-detail-csv` / `--deepseek-detail-json-array`**（控制台导出按时间窗聚合）等。脚本输出仅用于理解与取证，不会改写 **`cost_events`**。

与 **`pnpm issue:forensics`** 同属 **取证 / 对齐**工具链；机制说明见 **`020-实践-排障指南.md`** §D → **012**。

### 5.1 控制台「分项长表」长什么样（图示一类导出）

供应商侧常见导出不是「一行里直接给齐 prompt/output/hit/miss」，而是 **一行一个分项**，示意列包括：

| 语义 | 典型列名举例 |
| --- | --- |
| 日历日（UTC） | `utc_date`（如 `YYYY/M/D`） |
| 模型 | `model`（如 `deepseek-v4-pro` / `deepseek-v4-flash`） |
| 密钥维度 | `api_key_name`、`api_key` |
| 分项类型 | `type`：`output_tokens`、`input_cache_hit_tokens`、`input_cache_miss_tokens`、`request_count` 等 |
| 计价 | `price`（该行分项的单价） |
| 数量 | `amount`（该行分项计数） |

**可读成本**：对有单价的 token 分项，可把 **本条费用理解为 `price * amount`**（币种以控制台/导出为准；`request_count` 类行可能没有单价列，需单独看产品说明）。

**粒度提醒**：仅有 **`utc_date`（按日）** 时，和单条 **`heartbeat_run` 的起止时刻**只能是 **同一天内的粗对齐**；若要卡到本轮 run，需要 **带时分秒的时间戳列**或其它更细的导出。

**与本仓库 **`pnpm usage:reconcile`** 的关系**：Track 4（`--deepseek-detail-csv` / `--deepseek-detail-json-array`）当前按 **表头启发式**找「时间列 + 宽列形式的 prompt/completion/hit/miss/金额」，**并不会按 `type` 字段自动透视长表**。若手里只有图示这种 **`type + amount`** 布局，需先在表格里 **透视成宽列**（或算出按窗汇总的金额/分量）再喂给脚本，否则会 **推断不到可用列**。需要时用 **`--dashboard-cols`**（见 `scripts/usage-token-reconcile.mjs` 头注释）指认列索引或表头子串。

## 6. 实操建议（给操作者的一句清单）

| 要问的事 | 建议 |
| --- | --- |
| 这轮到底吃了多少 prompt token？ | 看本条 **`usage_json`**：**`raw*`** vs **差分分量**一眼分清；结合 **`usageSource`**。 |
| 累计是不是快触顶轮转？ | 看 **会话维度的 raw / 轮转理由**文案（与原 run 证据链一起看）。 |
| 这条 run 真金白银花了多少？ | **`costUsd` 已累计语义未证之前**，用 **控制台导出 + run 时间窗** 对齐；勿盲信单字段。 |
| 我想修「钱多记了」 | 产品上需 **服务端差额**或 **适配器改成增量报价**；对账脚本只辅助定位，不负责改账。 |
