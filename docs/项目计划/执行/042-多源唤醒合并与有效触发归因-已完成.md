# 多源唤醒合并与有效触发归因

**母本 / 拆单出处：** [`24 评论唤起过载与编排分层改版计划`](../长期需求/24%20评论唤起过载与编排分层改版计划%202026-05-17.md) 里的 **A2**（外加文首第三条总目标：时间线上别误导人）。

**探查：** [`024-探查-042多源唤醒合并与归因-现状与落点`](../探查/024-探查-042多源唤醒合并与归因-现状与落点.md)

---

## 背景与范围（人话）

现实里会出现：**同一小段时间、同一个智能体、同一条事务（或同一件活）**，被好几条路同时「叫醒」——比如**定时心跳**和**评论/自动化**叠在一起。单子要的是：合并时别糊成一笔糊涂账，事后能答 **「到底算谁触发的、谁被挤掉了」**。

---

## 打算怎么改（逻辑 · 人话版）

**记在哪儿：** 不新开数据库列。就在 **这次运行快照**（存在心搏 **`run` 记录自带的那份 JSON 上下文**）里加一块 **`effectiveTrigger`**。接口/看板如果本来就能看运行上下文，继续看同一份即可。

**这块里写什么（第一版）：**

- **当前认账的那次叫醒**：谁发起的（来源）、因为啥事（原因）、可选的补充说明。
- **被合进去、但不再当家作主的那几次**：打个列表，带上大概时间，方便审计「刚才其实还有谁也叫过」。

**什么时候写：**

1. **每次新来的叫醒**：如果快照里还没有，就先补一条「当前就这一个是正主」。
2. **和已经在排队或正在跑的那次 Run 合并时**：按下面的**优先级**重新算谁当正主，输了的丢进「被吸进来的」列表。

**优先级顺口溜：**

- **定时器**最轻；**自动流、派工、人点起来**这一档更重（都是「正经活儿先」）。
- **同一档里谁后到谁算数**（离操作者更近那一次优先）。

**和代码里三处「本来就合并」的地方对齐：**  
同一条事务上正在执行的、先延后再并的、以及不涉及事务的那条合并路径——**原来怎么合并上下文，现在还怎么合并，只是多写 `effectiveTrigger`**。

---

## 评审结论

**没有要你拍板的分叉**：上面这套优先级就和母本里「人/编排相对 timer 别被带偏」一个方向。

---

## 落地说明（操作者视角）

- **能感到啥：** 单独看不出界面大变；**翻运行详情/上下文**时，能多看到 **「最后算谁叫醒的」** 和 **「中间被吃掉的几次」**。
- **还没做满的两块：**  
  - 改事务接口里「同一拍里好几个叫醒先压成一条」的那段 **没动**；  
  - **活动日志**没有专门为这件事多写故事（真要查还可以从表 **`agent_wakeup_requests`** 里状态是 coalesced 的记录往下摸）。
