## 产品陈述：唤起归因应回答什么（2026-05-20 董事会口径）

> 与 **042 原始设计**（多源合并时「谁压过谁」）**互补**：合并场景仍保留 `absorbed`；**所有 run** 都应用**一句人话**说清叫醒因果，而不是只展示 `分配 · 分配交办 · system`。

### 一句人话的四个要素

| # | 用户要问的 | 界面应明确写出（示例） | 数据从哪来（已有 / 待接） |
| --- | --- | --- | --- |
| 1 | **叫醒主因** | 来自 **评论** / **分配交办** / **定时心跳** / **按需手动** / **自动化** | `effectiveTrigger.winning.reason` 或 `wakeReason`；`wakeSource` |
| 2 | **若是分配：谁派的** | **董事会（人类）** / **某智能体** / **系统规则** | `agent_wakeup_requests.requestedByActorType` + `requestedByActorId`（**已有，UI 未接**）；勿单靠 `triggerDetail: system`（指派路径常写 system，易误导） |
| 3 | **若是分配：因何派** | 新建事务并指派 / 改经办人 / 子单接力 / 恢复单 / … | `contextSnapshot.source`（如 `issue.create`）、wakeup `payload.mutation`、`contextSource` |
| 4 | **若是评论：哪条评论** | 链到评论摘要或 `wakeCommentId` | `wakeCommentId` / `commentId` / `paperclipWake.comments`（**已有，唤起归因卡片未接**） |

**多源合并时再加两行（042 原意）：**

- **最终认账：** 上述四要素里「赢」的那一路（不必叫「合并后认账」若从未合并，可改 **「本次叫醒」**）。
- **被挤掉：** 列表写出其它路（定时被评论挤掉等）+ 时间。

### 与 042 原文的差异（为何你会忘）

| 042 当初写的 | 你现在的直觉 |
| --- | --- |
| 短时窗 **多路叫醒合并**，`effectiveTrigger` 记赢家与 `absorbed` | **每次 run** 都要能读懂「谁、为何、凭啥叫醒」 |
| 偏 **审计 / 排障**（防 timer 盖掉评论） | 偏 **运营可读**（董事会不用猜 `system`） |
| UI 第一期只映射 `winningInvocation` 三字段 | 要接 **评论 id、请求者、分配原因** 等快照字段 |

**结论：** 042 **没错**，但 **只做了一半**；**唤起归因人话** 优先在 **运行清单 → 运行详情 → 输入编排** 卡片打磨（见 [060 §7.1](060-2026-05-20-验证闭环与编排首日执行.md#71-问题记录唤起归因卡片内容不符合定位)）。事务页 **动态** 标签负责「跑得怎样」；**不必** 与运行清单重复整块卡片。

### 事务详情三标签（界面真名 · 2026-05-20）

| 标签 | 已有 | 待办（董事会） |
| --- | --- | --- |
| **聊天** | `IssueChatThread`，讨论为主；改动面宜小 | 保持干净；**不** 为主战场加唤起归因 |
| **动态** | **运行账本**、费用摘要、活动、审批、子事务等 | 汉化扫尾；确认是否够用后再决定是否加链 **运行清单** |
| **相关工作** | `IssueRelatedWorkPanel` 已汉化（060 · `5b560a45`） | — |

**运营向 UI：** 倾向把 **「…」隐藏项平铺**（事务头、评论更多操作等），便于董事会看 run；与上游「普通用户少看 run」假设不同——另开 UI 小单，不塞进 050 第二期默认可交付范围。

### Run 呈现统一：清单 + 详情（董事会 · 2026-05-20 · **仅规划，未开工**）

**原子：** 一条 **run** = 一条 `HeartbeatRun`（心跳/适配器执行记录），全产品共用同一 ID。

| 能力 | 公司级 **运行清单** | 智能体下 **运行** 标签 |
| --- | --- | --- |
| **清单** | **`/runs`** · `heartbeatsApi.listPaged` · 表格式 · 可筛 **智能体 / 状态 / 来源** · 分页 | `/agents/:id/runs` · `heartbeatsApi.list(company, agentId)` · `HeartbeatRunListItem` 主从列表 · **等价于清单 + `agentId` 固定** |
| **详情** | **`/runs/:runId`** · **五标签**（入队 / 输入编排含 **唤起归因** / 终稿 / 执行 / 记录）· 执行区 **复用** `HeartbeatRunDetailPanel` | `/agents/:id/runs/:runId` · **仅** `HeartbeatRunDetailPanel`（转录、日志、指标等）· **无** 唤起归因与编排标签 |

**现状结论：** 侧边栏 **运行清单** 已不是另一套数据，而是把 **智能体运行详情里「编排/审计」向的一层** 提到公司级，并加上分页与来源筛选；**执行面** 两边仍共用 `HeartbeatRunDetailPanel`，但 **智能体详情页未挂上** 运行清单那套 **详情外壳**，故两处体验不一致、需双处维护文案与字段。

**目标（吸收后替换，单点维护）：**

1. 抽出可参数化的 **运行清单页**（`companyId` + 可选 `agentId` 预设筛选 + `basePath` 面包屑）。
2. 抽出可参数化的 **运行详情页**（现行 `OrchestrationInjectionRunDetail` 为真值；唤起归因人话在此外壳上迭代）。
3. 智能体 **运行** 标签改为 **嵌入或跳转** 上述两页，不再单独维护一套详情壳；智能体 **仪表盘** 上的图表/最近运行卡片可保留（非清单真值）。
4. 深链策略（后定）：事务 **查看运行** 可继续 `/agents/.../runs/...` 重定向到公司详情，或统一为 **`/runs/:id?agent=...`**——以「打开后看到与运行清单相同详情」为准；旧 `orchestration-injection/*` 仅作重定向（[023 §6](../最佳实践/023-实践-Board界面中文口径与文档对照表.md#6-路由改名orchestration-injection--runs董事会--2026-05-20--已落地)）。

**代码索引（统一时动）：** `OrchestrationInjection.tsx`、`OrchestrationInjectionRunDetail.tsx`、`AgentDetail.tsx`（`RunsTab`）、`HeartbeatRunDetailPanel.tsx`、`HeartbeatRunListItem.tsx`。
