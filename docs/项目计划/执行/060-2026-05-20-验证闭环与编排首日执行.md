### 7.1 问题记录：「唤起归因」卡片内容不符合定位

**样本：** ROU-6 · run `101c048a-b3b6-445a-b99f-449dca123028` · 入口：**运行清单 → 运行详情 → 标签「输入编排」**。

**产品定位（[042](042-多源唤醒合并与有效触发归因-已完成.md) / [050](050-运行时间线唤起溯源与档位-UI.md) 第一期）：**  
让人回答——**同一短时窗多路叫醒合并后，最终算谁触发**；若有被挤掉的，列出 **被合并吸收的其他触发**。不是重复运行列表里的来源列，也不是提示词编排说明。

**界面上实际看到（与 API `contextSnapshot.effectiveTrigger` 一致）：**

| 字段 | 显示 |
| --- | --- |
| 合并后认账的触发 | **分配 · 分配交办** |
| 补充说明 | **`system`**（英文，未汉化） |
| 被合并吸收的其他触发 | **无** |
| 最后一次合并时间 | **无** |

**为何不符合定位（人话）：**

1. **没有「合并」却仍叫「合并后认账」** — `absorbed` 为空、无 `lastMergeAt`，对用户像普通指派叫醒，卡片标题误导。  
2. **答不出「谁叫醒」** — ROU-6 是人工 **invoke** 做翻译；卡片只有「分配 / 分配交办 / system」，看不出董事会手动触发、哪条评论、是否定时心跳被挤掉等。  
3. **与运行列表信息重复** — 列表已有来源/原因；单源场景下卡片几乎无增量。  
4. **位置不对** — 卡片放在 **「输入编排」** 里、紧挨 **最终提示词**，像 prompt 附属信息，不像「叫醒因果」；更合理应在 **运行快照** 旁或独立摘要区（050 第二期事务页亦同理）。  
5. **补充说明未走人机文案** — `triggerDetail: system` 原样露出，与 [056](056-运行详情观测与编排叙事人机文案一轮-2026-05-18.md) 方向不一致。

**API 取证（便于后续改 UI/服务端文案）：**

```json
"effectiveTrigger": {
  "v": 1,
  "absorbed": [],
  "winningInvocation": {
    "reason": "issue_assigned",
    "source": "assignment",
    "triggerDetail": "system"
  }
}
```

**处置建议（记入 backlog，今日不在 060 开大改）：**

| 优先级 | 方向 |
| --- | --- |
| P1 | **单源降级展示**：无 `absorbed` 时改标题/文案（如「本次叫醒」），并串联 **invoke 操作者**、**wake 评论 id**（若 `contextSnapshot` 有） |
| P1 | **换区**：唤起归因挪出「输入编排」，贴近 **运行快照** 或运行详情首屏 |
| P2 | **汉化** `triggerDetail` 枚举（`system` → 系统派工 等） |
| P2 | 050 **第二期** 事务 **事务运行账本** 复用同一套人话组件时再统一 |

**关联：** [050](050-运行时间线唤起溯源与档位-UI.md) · [042](042-多源唤醒合并与有效触发归因-已完成.md) · [023](../最佳实践/023-实践-Board界面中文口径与文档对照表.md)

### 7.2 产品复述：唤起归因应有一句明确说法（2026-05-20）

董事会口径（与 [050 §产品陈述](050-运行时间线唤起溯源与档位-UI.md#产品陈述唤起归因应回答什么2026-05-20-董事会口径) 同步）：

1. **主因：** 这次 run 是 **评论叫醒**、**分配交办**、**定时**、**手动按需** 还是别的。  
2. **若分配：** 是 **人类（董事会）**、**某智能体** 还是 **系统规则** 触发的指派——不能只有一个 `system`。  
3. **若分配：** **为什么派**（建单、改经办、接力、恢复…）。  
4. **若评论：** **哪条评论**（可点开）。

042 原始设计侧重 **多路合并谁赢**；上面四条是 **every run 的人话归因**。ROU-6 快照里已有 `source: issue.create`、`wakeTriggerDetail` 等，**唤起归因卡片未接**，故体感「不像归因」。
