## 二、根因分析：管道产物为何无人消费

**一句话**：生成是批量的、回溯的、自动的；消费是手动的、实时的、需要心智投入的。两端完全不匹配。

### 具体拆解

1. **时间错位** — 3 月的会话到 4 月底才被 LLM 回溯分析吐出候选，上下文早凉了。人不会回去审一个多月前的事。

2. **量把人淹死** — 38 天 × 每天 4-8 条 ≈ 200+ 候选技能，同样数量的候选规则。审一条 5 分钟，审完需要 30+ 小时。生成速度远大于审阅速度。

3. **格式是"给人审的"，机制是"无人值守"** — 每个候选有优先级、置信度、执行步骤，但没有通知、没有 deadline、没有阻塞。是管道的外部排废口，不是工作流的环节。

4. **没有低成本采纳路径** — "已批准 → 已迁入正式技能域"需要人读完 → 理解 → 评估冲突 → 改写 → 提交，不是"点一下批准"就能完成。

5. **生成方和消费方不是同一个人** — 管道是 LLM 自动跑的，它不知道你今天在做什么，命中率极低。

### 跨天趋势分析的致命伤

跨天趋势分析的数据链路：

```
原始会话（有真信息）
  → 转录摘要（LLM 提炼，损失 60%）
    → 行为分析报告（LLM 解读，再次损失）
      → 跨天趋势（LLM 从二次加工品里挑模式）
```

每转一道损失一层信噪比。且没有接入 ground truth（git diff、实际代码变更），只能看到 LLM 文本在反复提关键词，看不到仓库实际发生的变化。

真正让跨天对比有用的前提是：**人工注入关键事件节点**（哪天上线生产、哪天踩坑、哪天换方案），然后看 AI 行为在这些节点前后有无结构性变化。

## 三、关键洞察：AI 驱动开发的本质问题

### 发散 vs 收敛

| 维度 | AI 擅长的 | AI 不擅长的 |
|------|----------|------------|
| 方向 | 发散 — 给可能性 | 收敛 — 给决策 |
| 速度 | 生成代码/方案极快 | 判断"哪个对" |
| 输出 | 多选项（甩锅给人挑） | 单推荐 + 为什么 |

**核心问题**：AI 把最贵的资源——人的判断力——消耗在了"选择"上，而不是"决策"上。

Cursor 给 4-5 个改进选项，信息正确、选择多，但你需要先读懂每个方案、做 diff、评估权衡、再拍板。每一步都在烧判断力。这是**生成能力过剩，收敛能力缺失**。

### 解决方案原则

1. **收敛优先，不发散** — AI 给 1 个推荐 + 为什么，不给 5 个选项让人挑
2. **锚定地面真相** — 不从 LLM 输出里套娃，锚在 git diff、真实 bug、真实指标
3. **人类判断力做乘法** — 人审 3 个，AI 学模式应用到剩下 225 个
4. **时间窗口要窄** — 超过 N 天未消费自动归档
5. **AI 做减法** — "100 个会话里，只有这 2 个模式值得关注"

### 文档驱动开发模型

人写目标文档 → AI 读文档干活 → 可回滚 → 有记录 → 从干活中学习 → 下一轮更好。

关键转变：人从"驾驶员"降维为"监察员"，判断力花在目标标定和纠偏上，不在执行的每一步上。

## 四、五仓架构对齐

| 仓库 | 职责 | 角色 |
|------|------|------|
| **Jarvis** | 管吴衡（人） | 时间、注意力、文档、记忆、跨域索引 |
| **Paperclip** | 管 AI 工人 | 执行、heartbeat、回滚、审计 |
| **Token** | 管产品真源 | 代码、工程证据 |
| **Routic** | 管业务真源 | 经营、财务、策略 |
| **工具优化** | 管方法论真源 | 工具知识、规则 |

**流转路径**：人在 Jarvis 写目标 → Paperclip 读目标拆 issue 派工人 → 工人干完留 run log → Jarvis 只读摘要（保护注意力）→ 下一轮继承上下文。

**之前管道问题的根因**：Jarvis 管道生成了候选，但没有流入 Paperclip 的 issue 系统。Jarvis → Paperclip 的推送路径没通。

## 五、Paperclip 精简决策（核心）

### 问题

CEO agent 在回形针里比较傻，编排层过度设计。决定做减法。

### 精简清单

| 组件 | 操作 | 原因 |
|------|------|------|
| Company goal 体系 | **关** | goal → CEO 拆任务链路不可靠 |
| CEO agent 角色编排 | **关** | "
拆任务、选工人"质量不行 |
| Heartbeat 自动调度 | **关调度，留手动** | 关 heartbeat.ts 定时 tick，保留手动 invoke |
| Budget 控制 | **关** | POC 阶段不需要 |
| Approval gate | **关** | 审批流阻塞大于价值 |
| Plugin 系统 | **关** | 当前不需要 |
| Issue + checkout + run | **留** | 核心任务载体 |
| Adapter invoke | **留** | 执行入口 |
| Recovery | **留** | 异常兜底 |
| resultJson + run events | **留** | 结果和审计 |

### 保留组件的代码位置（大概）

- issues 表 + heartbeat_runs → 任务和运行记录
- adapters/ → 模型调用适配器
- recovery/ → 孤儿 run 收割、静默 run 检测、阻塞链检测
- heartbeat_run_events → 运行事件记录

### 新任务来源

Token Bridge 工程计划直接映射成 issue，不走公司目标链。

### 新角色池（9 角色，按任务规模选子集）

计划 → 探查 → 分析 → 需求 → 执行 → 测试 → 运维 → 复盘 → 总结

- 不是全链路强行走完（会提示词汇爆炸）
- 按任务规模判断需要哪些角色
- 适配器选型不重要，重要的是模型选型（已有定论）

### 回写机制

1. 结果写到文档域
2. issue 里只留摘要 + 指针链接
3. 不把原始 run log 堆进 Jarvis
