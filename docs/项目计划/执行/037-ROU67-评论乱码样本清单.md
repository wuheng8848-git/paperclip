## 二、乱码类型分类

### 类型 A：写入时损坏 — GBK→UTF-8 解码错误（U+FFFD 替换字符）

**特征**：文本中包含大量 `�`（U+FFFD，Unicode 替换字符），是 GBK 编码的字节流被 Node.js `String(chunk)` 按 UTF-8 解码时，无法识别的字节被替换为 U+FFFD。

**数据路径**：
```
Agent stdout（GBK 编码，Windows CP936）
  → Paperclip spawn 子进程捕获
  → server-utils.ts: String(chunk) 按 UTF-8 解码
  → GBK 字节无法识别 → 替换为 U+FFFD
  → resultJson / comment body 写入数据库
  → 前端渲染
```

**证据**：Hex 分析显示 `efbfbd`（U+FFFD 的 UTF-8 编码）在乱码区域密集出现。

**影响范围**：38 个样本中的大部分（约 30 个），涉及 agent 名字、中文标题、中文评论文本。

### 类型 B：写入时损坏 — Shell 内联 JSON 编码损坏（问号替换）

**特征**：中文文本整体变成 `?`，或大量 `????` 连续出现，夹杂少量 ASCII 文本。这是 agent 在 Windows 控制台使用 `curl -d '{"comment":"中文"}'` 内联 JSON 时，中文经 CP936/GBK 编码损坏。

**数据路径**：
```
Agent 运行中执行 curl -d '{...中文...}'
  → Windows 控制台（CP936/GBK）
  → curl 发送 HTTP 请求
  → Paperclip API 接收（已是损坏的 UTF-8）
  → 写入数据库
  → 前端渲染
```

**证据**：
- ROU-54 评论 `17aa1d33`：`?????????:?API?prompt??(??)+ ...`，全是 `?` 替换
- ROU-45 评论 `80bb4957`：`? ROU-45 ????????:`，大面积问号
- ROU-50 评论 `2b871f48`：`???:?????????? ...`
- ROU-42 评论 `9495d673`：`ROU-42 LiteLLM OSS ?????????????????????`

**影响范围**：约 8 个样本，集中在 Qwen agent 通过 curl 写评论的场景。

### 类型 C：未发现 — 存储损坏

数据库存储层面（PostgreSQL UTF-8 text 列）未发现损坏证据。数据库接收到的数据在写入前已经损坏，存储和读取链路本身是安全的（ROU-66 代码审计已确认）。

### 类型 D：未发现 — 读取/渲染损坏

API 响应头正确设置 `Content-Type: application/json; charset=utf-8`，前端 `react-markdown` 正确处理 Unicode。`resultJson` 中的中文（如 ROU-58 的 summary/result）能正常返回，说明读取/渲染链路无问题。

## 四、根因判定总结

### 4.1 两条损坏路径

| 路径 | 损坏环节 | 乱码形态 | 受影响 agent | 严重程度 |
|------|---------|---------|-------------|---------|
| **A. stdout→String(chunk)** | 服务端读取子进程 stdout 时，GBK 字节被当 UTF-8 解码 | U+FFFD 替换字符（`�`） | 所有在 Windows 本地运行的 agent（Qwen、CodeBuddy） | 高（影响 heartbeat 自动评论） |
| **B. curl -d 内联 JSON** | agent 在 Windows 控制台用 curl 发送含中文的 JSON | 问号替换（`???`） | Qwen agent（SKILL.md 示例导致） | 高（影响 agent 主动写评论） |

### 4.2 关键区分证据

**路径 A 与路径 B 的区分**：

1. **ROU-58 的关键证据**：
   - `resultJson.summary` 和 `resultJson.result`：**正常中文** ✅
   - agent 主动写的评论（479f333f, 0434ebbf）：**乱码** ❌
   - 结论：`heartbeat resultJson` 路径正常，agent 主动 `curl` 写评论路径损坏

2. **ROU-54 的关键证据**：
   - 评论 `17aa1d33`：全是 `?` 替换，无 U+FFFD
   - 这是典型的 Shell 内联 `JSON` 在 Windows 控制台编码损坏（路径 B）

3. **ROU-35 的关键证据**：
   - 事务标题和描述中 agent 名字乱码（U+FFFD）
   - 这说明 agent 名字在创建事务时（路径 A）就已经损坏
   - agent 名字来自 `heartbeat` 自动评论（`buildHeartbeatRunIssueComment`），经过 `stdout` 解码

### 4.3 未受影响的路径

| 路径 | 状态 | 证据 |
|------|------|------|
| 数据库存储 | ✅ 安全 | PostgreSQL UTF-8 text 列，ROU-66 代码审计确认 |
| API 读取 | ✅ 安全 | `res.json()` 设置 `charset=utf-8`，`resultJson` 中文正常返回 |
| 前端渲染 | ✅ 安全 | `react-markdown` 正确处理 Unicode，用户手动输入的中文正常显示 |
| heartbeat resultJson | ✅ 安全 | Qwen `-o stream-json` 输出 UTF-8 JSON，`JSON.parse` 正常 |

## 五、可疑数据形态分析

### 5.1 U+FFFD 替换字符的 Hex 特征

通过 Hex 分析，乱码区域的字节模式为：
- `efbfbd` = U+FFFD 的 UTF-8 编码
- 连续出现 `efbfbdefbfbd` = 多个 U+FFFD 连续
- 中间夹杂的 `d0be`, `c4bf`, `d7b4` 等 = GBK 编码的中文单字节被当 UTF-8 解码后的残留

### 5.2 问号替换的特征

问号替换（路径 B）的文本特征：
- `?` (0x3F) 是 ASCII 问号
- 原中文文本完全被 `?` 替换，无法恢复原文
- 常见模式：`????:?????` 或 `??????????:?API?prompt??`
- ASCII 部分保留（如 `API`, `prompt`, `PATCH` 等英文词）

## 六、结论

1. **所有乱码均为写入时损坏**，存储和读取链路安全
2. **两条损坏路径**均已确认：
   - 路径 A（stdout 编码）：影响 heartbeat 自动评论，GBY→UTF-8 解码错误
   - 路径 B（curl 内联 JSON）：影响 agent 主动写评论，Windows 控制台编码损坏
3. **影响范围**：46 个样本，覆盖 221 条评论中的 38 条（17.2%），以及 50 个事务中的 5 个标题和 5 个描述
4. **严重程度**：7 个样本为严重乱码（大段中文全部损坏），不可恢复
5. **修复方向**：参考探查 017 的止血方案（改 SKILL.md + prompt 模板 + 服务端 GBK 回退）
