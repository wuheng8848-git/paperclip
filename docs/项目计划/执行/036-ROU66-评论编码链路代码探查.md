## 审查结论

**整体评级: ✅ 编码链路基本安全，无阻断性缺陷**

评论从创建到渲染的完整链路在 UTF-8 编码下是连贯的，不存在数据截断或乱码风险。但发现若干低优先级观察项和一处中等风险点（Mermaid SVG 的 `dangerouslySetInnerHTML`）。

## 逐检查点审查

### 1. API 入参：POST /api/issues/:id/comments

| 项目 | 发现 | 风险 |
|------|------|------|
| Content-Type | `express.json()` 全局中间件（`server/src/app.ts:140`），无 `charset` 选项 | ⚠️ 低 |
| Body Parser | `express.json({ limit: "10mb" })`，默认只解析 `application/json` | ✅ 正常 |
| Charset 强制 | 未设置。Express 默认接受任何 `charset` 参数（如 `application/json; charset=utf-8`） | ⚠️ 低 |
| 验证 | `validate(addIssueCommentSchema)` 中间件解析 `req.body`，Zod 验证 | ✅ 正常 |

**详细说明**:
- `express.json()` 内部使用 `body-parser` 的 `json()`，默认 `type: "application/json"`，即只对 `Content-Type` 包含 `application/json` 的请求解析
- **未指定 `charset` 选项** → Express 默认行为：接受 `charset=utf-8`、`charset=utf-16` 等。实际上 `body-parser` 内部用 `iconv-lite` 解码，如果客户端发送非 UTF-8 编码的 JSON body（如 `charset=iso-8859-1`），`iconv-lite` 会尝试转码为 JS 内部 UTF-16 字符串
- **实际风险极低**：几乎所有现代 HTTP 客户端都以 UTF-8 发送 JSON。UI 客户端 (`ui/src/api/client.ts:19`) 显式设置 `Content-Type: application/json`（无 charset 后缀），浏览器 `fetch` 默认以 UTF-8 编码 JSON body

### 2. 服务端处理：中间件编码处理、请求体解码

| 项目 | 发现 | 风险 |
|------|------|------|
| Zod 验证 | `addIssueCommentSchema.body` = `multilineTextSchema.pipe(z.string().min(1))` | ✅ 正常 |
| 换行标准化 | `multilineTextSchema` 将 `\\n` → `\n`、`\\r\\n` → `\n`、`\\r` → `\n` | ✅ 正常 |
| 用户名脱敏 | `redactCurrentUserText()` 对 body 做用户名/路径脱敏后再存入 DB | ✅ 正常 |
| 无额外编码处理 | 服务层不额外检查/转换编码 | ✅ 正常（依赖 JS 字符串已是 Unicode） |

**详细说明**:
- `multilineTextSchema`（`packages/shared/src/validators/text.ts`）做的是 JSON 字符串内转义换行符的标准化，不是编码转换
- `redactCurrentUserText`（`server/src/log-redaction.ts`）用正则替换用户名和路径，使用 `Array.from(trimmed).length` 计算字符数（正确处理 Unicode 代理对）
- 所有处理在 JS 字符串层面操作，此时已经是 Unicode，编码已由 `body-parser` 解码完成

### 3. 数据库存储：comments 表字段类型、collation、连接 charset

| 项目 | 发现 | 风险 |
|------|------|------|
| body 字段类型 | `text NOT NULL`（PostgreSQL `text` 类型，无长度限制） | ✅ 正常 |
| 列级 collation | 无显式 collate 指定 | ⚠️ 低 |
| 数据库 encoding | `ensurePostgresDatabase()` 创建 `UTF8` encoding, `lc_collate='C'`, `lc_ctype='C'` | ⚠️ 低 |
| 嵌入式 PG | `initdbFlags: ["--encoding=UTF8", "--locale=C", "--lc-messages=C"]` | ✅ 正常 |
| Docker PG | `postgres:17-alpine`，无自定义 locale，可能默认 `en_US.utf8` | ⚠️ 低 |
| 连接 charset | `postgres` JS 驱动默认 UTF-8，连接字符串无 charset 参数 | ✅ 正常 |
| GIN trigram 索引 | `issue_comments_body_search_idx` 使用 `gin_trgm_ops` | ⚠️ 低 |

**详细说明**:

- **PostgreSQL `text` 类型**：原生支持 UTF-8，可存储任意 Unicode 文本（包括 CJK、emoji 等），无长度限制
- **`lc_collate='C'`**：使用字节序排序，不是 Unicode 感知排序。对评论存储和读取**无影响**（只影响 `ORDER BY` 等排序操作），但 GIN trigram 索引在 `C` locale 下的行为可能与 `en_US.utf8` 下不同
- **Docker locale 不一致**：应用创建的数据库用 `C` locale，Docker 直接启动的 PG 可能用 `en_US.utf8`。两者都支持 UTF-8 存储，不影响数据正确性，但 trigram 搜索行为可能略有差异
- **`postgres` JS 驱动**：默认以 UTF-8 编解码与 PostgreSQL 通信，`client_encoding` 默认为 `UTF8`

### 4. API 读取：GET 评论的响应头 charset、JSON 序列化

| 项目 | 发现 | 风险 |
|------|------|------|
| 响应头 charset | `res.json(comments)` / `res.json(comment)`，Express 默认设置 `Content-Type: application/json; charset=utf-8` | ✅ 正常 |
| JSON 序列化 | Express `res.json()` 调用 `JSON.stringify()`，JS 原生 Unicode 支持 | ✅ 正常 |
| 非 ASCII 字符 | `JSON.stringify` 将非 ASCII 字符原样输出（不转义为 `\uXXXX`），这是合法 JSON | ✅ 正常 |

**详细说明**:
- Express `res.json()` 自动设置 `Content-Type: application/json; charset=utf-8`，这是 Express 的默认行为
- `JSON.stringify()` 对 BMP 字符（U+0000–U+FFFF）原样输出，对补充平面字符（emoji 等）输出 UTF-16 代理对，这是标准 JSON 行为
- UI 客户端 `res.json()` 解码响应时使用 `fetch` API，默认以 UTF-8 解码 JSON

### 5. Markdown 渲染：特殊字符转义、Unicode 处理

| 项目 | 发现 | 风险 |
|------|------|------|
| 渲染库 | `react-markdown` + `remark-gfm` | ✅ 正常 |
| HTML 注入防护 | `react-markdown` 默认不渲染原始 HTML（测试确认 `<script>` 被转义） | ✅ 正常 |
| `javascript:` URL | `safeMarkdownUrlTransform()` 过滤，测试确认被清空 | ✅ 正常 |
| 外部链接 | 添加 `target="_blank" rel="noopener noreferrer"` | ✅ 正常 |
| Mermaid SVG | `dangerouslySetInnerHTML={{ __html: svg }}` | ⚠️ 中 |
| Mermaid 安全级别 | `securityLevel: "strict"` | ✅ 正常（缓解项） |
| Unicode 处理 | `react-markdown` 正确处理 Unicode 文本 | ✅ 正常 |
| 软换行 | `remarkSoftBreaks` 插件将 `\n` 转为 `<br/>` | ✅ 正常 |

**详细说明**:

- **Mermaid SVG 的 `dangerouslySetInnerHTML`**（`MarkdownBody.tsx:475`）：
  - Mermaid 渲染后输出 SVG 字符串，通过 `dangerouslySetInnerHTML` 注入
  - **缓解措施**：`securityLevel: "strict"` 禁止 Mermaid 中的事件绑定和点击交互
  - **残留风险**：如果 Mermaid 库本身有 XSS 漏洞，或 SVG 包含 `<foreignObject>` 嵌入 HTML，可能绕过限制
  - **实际风险**：中等。Mermaid `securityLevel: "strict"` 是业界标准做法，但 `dangerouslySetInnerHTML` 本身是 React 中的反模式

- **HTML 转义测试确认**（`MarkdownBody.test.tsx:126-138`）：
  - `<script>`, `<iframe>`, `<p onclick>` 均被正确转义为文本
  - `javascript:` URL 被清空为空字符串

## 风险汇总

| # | 风险 | 严重度 | 位置 | 说明 |
|---|------|--------|------|------|
| R1 | Mermaid SVG 使用 `dangerouslySetInnerHTML` | ⚠️ 中 | `ui/src/components/MarkdownBody.tsx:475` | 有 `securityLevel: "strict"` 缓解，但本质上是注入原始 HTML。建议后续引入 DOMPurify 对 SVG 输出做二次消毒 |
| R2 | `express.json()` 未显式设置 `charset` | 💡 低 | `server/src/app.ts:140` | 理论上允许非 UTF-8 编码的 JSON body，但现代客户端均使用 UTF-8，实际不会触发 |
| R3 | 数据库 `lc_collate='C'` 与 Docker 可能的 `en_US.utf8` 不一致 | 💡 低 | `packages/db/src/client.ts:772` vs `docker/docker-compose.yml` | 不影响数据正确性，但 GIN trigram 搜索排序行为可能不同 |
| R4 | `requestConfirmationHrefSchema` 的 URL 过滤不包含 `vbscript:` | 💡 低 | `packages/shared/src/validators/issue.ts:518-523` | 仅过滤 `javascript:`, `data:`, `//`；`vbscript:` 在现代浏览器中不受支持，但严格安全标准下应一并过滤 |

## 数据流图

```
客户端 (UTF-8 JSON)
  │
  ▼
express.json() [body-parser + iconv-lite]
  │ → JS 内部 Unicode 字符串
  ▼
Zod validate(addIssueCommentSchema)
  │ → multilineTextSchema: \n 标准化
  ▼
redactCurrentUserText()
  │ → 用户名/路径脱敏
  ▼
Drizzle ORM → postgres JS 驱动 (UTF-8)
  │ → PostgreSQL text 列 (UTF-8, lc_collate=C)
  ▼
Drizzle ORM → redactIssueComment()
  │ → 读取时再次脱敏
  ▼
Express res.json() → Content-Type: application/json; charset=utf-8
  │
  ▼
浏览器 fetch → JSON.parse() → JS Unicode 字符串
  │
  ▼
<MarkdownBody>{comment.body}</MarkdownBody>
  │ → react-markdown + remark-gfm
  │ → HTML 转义（默认不渲染原始 HTML）
  │ → javascript: URL 过滤
  │ → Mermaid: dangerouslySetInnerHTML (securityLevel: "strict")
  ▼
用户看到渲染后的评论
```
