## 步骤 2 实际调试记录（2026-05-15，续）

### 核心问题

Paperclip 心跳系统唤醒 test-co Agent 时，spawn CodeBuddy CLI 立即崩溃：

```
退出码: 3221225794  (Windows STATUS_DLL_INIT_FAILED / 0xC0000142)
表现:   进程启动 ~250ms 后死亡，stdout=0字节，stderr=0字节
时间线: 5月13日正常（CodeBuddy 旧版），5月15日全部失败（CodeBuddy v2.97.1）
```

### 根因分析

**高度疑似 `@lydell/node-pty` 原生模块（ConPTY）初始化失败。**

证据链：
1. CodeBuddy v2.97.1 引入了 `@lydell/node-pty`，包含 `conpty.node` 和 `conpty_console_list.node` 原生 addon
2. ConPTY 需要一个真正的 Windows 控制台对象来初始化——`DllMain(DLL_PROCESS_ATTACH)` 期间尝试分配伪控制台
3. Paperclip 的 `runChildProcess()` 使用 `spawn()` + `stdio: 'pipe'`，**没有分配控制台**
4. DLL 加载时 `DllMain` 返回 FALSE → 进程被 Windows 立即终止
5. 命令行直接运行正常：因为终端环境有真实的 console host 和 conhost.exe

### spawn 方式验证

在 Git Bash 环境中测试了 3 种 spawn 方式，**全部正常**（exit 0，输出 618 字节 JSON）：

| 方式 | 命令 | 结果 |
|------|------|------|
| cmd.exe 包裹 | `spawn('cmd.exe', ['/d','/s','/c', line])` | OK |
| 直接 node 调用 | `spawn('node.exe', ['入口脚本', ...args])` | OK |
| 3 并发 spawn | 同时 3 个 spawn | 全部 OK |

**关键结论：问题不出在 spawn 参数或命令格式，而出在 spawn 的宿主进程上下文。**
Paperclip server 进程内部 spawn 时，子进程继承的 console/环境与终端环境不同。

### Paperclip spawn 流程分析

Adapter 调用链：
```
heartbeat → execute.ts → runChildProcess() → resolveSpawnTarget() → spawn()
```

`resolveSpawnTarget()` 对 `.cmd` 文件做了二次包裹：
1. `resolveCommandPath()` 找到 `codebuddy.cmd` 完整路径
2. `resolveWindowsCmdShell()` 包裹成 `cmd.exe /d /s /c "codebuddy.cmd arg1 arg2..."`
3. 然后用 `spawn('cmd.exe', args, {shell: false, env: sanitized})` 执行

注意：`quoteForCmd` 对不含特殊字符的路径**不加引号**，所以实际是 `cmd.exe /d /s /c C:\Users\wuhen\AppData\Roaming\npm\codebuddy.cmd arg1...`

环境变量处理：
- `sanitizeInheritedPaperclipEnv()` 过滤掉大部分 `PAPERCLIP_*` 变量
- `ensurePathInEnv()` 确保 PATH 存在
- **不设置任何 DLL 搜索路径相关变量**

### 诊断日志状态

已在 `execute.ts` 添加 spawn 前环境快照日志（写 `.paperclip-debug-spawn.log`），但 **尚未触发**：

- 原因：心跳系统重启后未处理 agent
- 心跳显示 "enabled (30000ms)" 但实际未执行
- Agent 状态已从 "error" 手动重置为 "idle"，仍无心跳活动
- 可能原因：连续 5 次失败（5月15日）触发了某个 backoff/冷却机制

## 方案 A 实施：跳过 cmd.exe 直接 spawn node（2026-05-15）

### 原理

当前流程：
```
Paperclip server → spawn(cmd.exe, ["/d","/s","/c", "codebuddy.cmd --print ..."])
  → cmd.exe → 读取 codebuddy.cmd → 启动 node.exe → 加载 conpty.node → CRASH
```

修改后：
```
Paperclip server → spawn(node.exe, ["path/to/codebuddy/bin/codebuddy", "--print", ...])
  → node.exe → 加载 conpty.node → ???
```

### 实现

在 `execute.ts` 中，调用 `runChildProcess` 之前插入 Plan A 逻辑：

```typescript
// Plan A: On Windows, bypass cmd.exe wrapping to avoid STATUS_DLL_INIT_FAILED
let spawnCommand = command;
let spawnArgs = [...args];
if (process.platform === "win32") {
  try {
    const whichOut = execSync(`where ${command}`, { encoding: "utf8" })
      .split("\n").map(s => s.trim()).filter(Boolean);
    const cmdPath = whichOut.find(p => /\.cmd$/i.test(p));
    if (cmdPath) {
      const cmdDir = path.dirname(cmdPath);
      const entryScript = path.join(
        cmdDir, "node_modules", "@tencent-ai", "codebuddy-code", "bin", "codebuddy"
      );
      const nodeOut = execSync("where node", { encoding: "utf8" })
        .split("\n").map(s => s.trim()).filter(Boolean);
      const nodeBin = nodeOut[0];
      if (nodeBin && fs.existsSync(entryScript)) {
        spawnCommand = nodeBin;
        spawnArgs = [entryScript, ...args];
      }
    }
  } catch {
    // Fall back to cmd.exe wrapping
  }
}
// 然后：runChildProcess(runId, spawnCommand, spawnArgs, ...)
```

### 回落保护

如果 Plan A 任何环节失败（node 不在 PATH、entryScript 不存在等），自动回落原始 cmd.exe 包裹路径。
