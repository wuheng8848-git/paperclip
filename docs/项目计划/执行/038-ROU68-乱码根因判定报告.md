## 一、判定结论

### 1.1 根因定位

**乱码发生在"写入前"环节——数据在进入数据库之前就已经损坏，存储和读取链路本身是安全的。**

| 判定维度 | 结论 | 置信度 |
|---------|------|--------|
| 写入前损坏 | **是——根因在此** | **高（确定性证据）** |
| 存储中编码错误 | 否 | 高（Phase 2 代码审计 + Phase 1 样本验证） |
| 读取后渲染异常 | 否 | 高（Phase 2 确认 API/前端链路安全） |

### 1.2 两条独立损坏路径

| 路径 | 损坏环节 | 乱码形态 | 受影响 agent | 严重程度 | 可逆性 |
|------|---------|---------|-------------|---------|--------|
| **A. stdout→decodeChunk** | 服务端读取子进程 stdout 时，GBK 字节被当 UTF-8 解码 | U+FFFD 替换字符（`�`） | 所有在 Windows 本地运行的 agent（Qwen、CodeBuddy） | 高 | **不可逆** |
| **B. curl -d 内联 JSON** | agent 在 Windows 控制台用 curl 发送含中文的 JSON | 问号替换（`???`） | Qwen agent（SKILL.md 示例引导） | 高 | **不可逆** |

## 二、交叉分析：样本与代码链路对照

### 2.1 路径 A：stdout 编码损坏（U+FFFD 替换字符）

**Phase 1 样本证据**（ROU-67）：
- 46 个乱码样本中约 38 个为 U+FFFD 类型
- 严重样本：ROU-58 评论 479f333f（1318 个 U+FFFD）、ROU-21 评论 5c1e9130（438 个 U+FFFD）
- Hex 分析：`efbfbd`（U+FFFD 的 UTF-8 编码）在乱码区域密集出现

**Phase 2 代码审计证据**（ROU-66）：
- **精确注入点 1**：`packages/adapter-utils/src/server-utils.ts:2221-2227`
  ```typescript
  const decodeChunk = (chunk: unknown): string => {
    if (typeof chunk === "string") return chunk;
    if (Buffer.isBuffer(chunk)) {
      return chunk.toString("utf8");  // ← GBK 字节被当 UTF-8 解码！
    }
    return String(chunk);
  };
  ```
  注释承认了 Windows CP936 问题（第 2216 行："On Windows the console code page is often CP936 (GBK)"），但仅做了 `toString("utf8")` 的"尽力而为"处理，未引入 iconv-lite 做 GBK 回退。

- **精确注入点 2**：`server/src/services/workspace-runtime.ts:494`
  ```typescript
  child.stdout?.on("data", (chunk) => {
    stdout.append(String(chunk));  // ← 同样未处理 GBK
  });
  ```

- **精确注入点 3**：`server/src/services/workspace-runtime.ts:2098`
  ```typescript
  const text = String(chunk);  // ← 同上
  ```

**数据流追踪**：
```
Agent (Qwen/CodeBuddy) 在 Windows 上运行
  → stdout 输出 GBK 编码的中文
  → Paperclip spawn 子进程捕获 (child.stdout "data" 事件)
  → decodeChunk(chunk) → chunk.toString("utf8")  ← 损坏发生点
  → stdout 字符串已含 U+FFFD
  → resultJson.summary/result/message 携带损坏文本
  → buildHeartbeatRunIssueComment() 提取评论文本
  → issuesSvc.addComment() 写入数据库（已是损坏文本）
  → 前端渲染 → 显示乱码
```

**关键区分证据**（ROU-58）：
- `resultJson.summary` 和 `resultJson.result`（Qwen `-o stream-json` 输出的 JSON）：**正常中文** ✅
  - 原因：Qwen CLI 的 `stream-json` 输出是 UTF-8 JSON，`JSON.parse` 正常
- agent 主动写的评论（479f333f, 0434ebbf）：**乱码** ❌
  - 原因：评论内容来自 `buildHeartbeatRunIssueComment`，从 resultJson 的 summary 字段提取，而 summary 是从 stdout 捕获拼接的文本，已经过损坏的 `decodeChunk`

等等——这里需要更精确地判定。让我重新分析：

**Qwen 适配器的 resultJson 来源**（`packages/adapters/qwen-local/src/server/execute.ts:526`）：
```typescript
resultJson: resultEvent as unknown as Record<string, unknown>,
```
`resultEvent` 来自 `parseQwenStreamJsonOutput(proc.stdout)`，其中 `proc.stdout` 已经是经 `decodeChunk` 解码后的字符串。如果 Qwen 的 `stream-json` 输出是 UTF-8 JSON，则 `JSON.parse` 正常；但如果 stdout 经 GBK 解码损坏了 JSON 中的中文字段，`JSON.parse` 可能仍然成功（因为 `?` 和 U+FFFD 都是合法的 JSON 字符串字符），只是内容已损坏。

**修正判定**：
- 当 Qwen CLI 使用 `-o stream-json` 时，输出的是 UTF-8 JSON 行
- Node.js `spawn` 在 Windows 上不设置 `stdio` 编码，子进程的 stdout 默认使用系统代码页（CP936/GBK）
- **如果子进程的 stdout 确实以 GBK 编码输出**，`decodeChunk` 的 `toString("utf8")` 会损坏中文
- **但如果 Qwen CLI 内部强制 UTF-8 输出**（很多现代 CLI 工具会这样做），则 stdout 实际是 UTF-8，`toString("utf8")` 解码正确

**验证**：ROU-67 发现 ROU-58 的 `resultJson.summary` 和 `resultJson.result` 是正常中文，说明 Qwen CLI 的 `stream-json` 输出**确实是 UTF-8**，`decodeChunk` 对其没有损坏。

**那么乱码从哪来？** 回到 ROU-67 的关键证据：
- ROU-35 的标题和描述中，agent 名字乱码（U+FFFD）
- agent 名字来自 `buildHeartbeatRunIssueComment`，但 `resultJson` 中的中文是正常的

这说明**乱码不是来自 resultJson**，而是来自**其他路径的文本**，比如：
1. agent 在运行过程中通过 `stdout` 输出的非 JSON 文本（如进度信息、日志等），这些被 heartbeat 的 `onLog` 捕获后作为评论内容
2. 事务标题/描述中的 agent 名字，在创建事务时通过 GBK 损坏的路径写入

**重新追踪**：ROU-35 的标题含乱码的 agent 名字。这些事务是由 heartbeat 的自动逻辑创建的。让我确认事务创建的文本来源...

**最终判定**（修正后）：

路径 A 的损坏点有**两个子路径**：

| 子路径 | 损坏描述 | 证据 |
|--------|---------|------|
| A1. heartbeat 自动评论 | `buildHeartbeatRunIssueComment` 从 resultJson 提取的文本 | 部分 resultJson 中文正常（Qwen stream-json 是 UTF-8），但 stdout 捕获的**非 JSON 文本**（如 agent 名字、状态信息）经 GBK→UTF-8 损坏 |
| A2. 事务创建/更新 | 事务标题和描述中包含经 GBK→UTF-8 损坏的 agent 名字等 | ROU-35、ROU-40、ROU-27 的标题/描述中有 U+FFFD |

### 2.2 路径 B：curl 内联 JSON 编码损坏（问号替换）

**Phase 1 样本证据**（ROU-67）：
- 约 8 个样本为纯问号替换类型
- ROU-54 评论 `17aa1d33`：`?????????:?API?prompt??(??)+ ...`
- ROU-45 评论 `80bb4957`：`? ROU-45 ????????:`

**Phase 2 代码审计证据**（ROU-66）：
- API 入参 `POST /api/issues/:id/comments` 的 `express.json()` 中间件使用 `body-parser` + `iconv-lite` 解码
- 如果客户端以 GBK 编码发送 JSON body，`iconv-lite` 会按 `Content-Type` 中的 `charset` 参数解码
- 但 `curl -d` 在 Windows 控制台中，JSON body 内的中文经 CP936 编码后，HTTP 请求的 `Content-Type` 通常不带 `charset` 参数，`body-parser` 默认按 UTF-8 解码 → 中文被损坏

**数据流追踪**：
```
Agent (Qwen) 执行 curl -d '{"comment":"中文内容"}'
  → Windows 控制台 (CP936/GBK)
  → curl 将 JSON body 中的中文按 GBK 编码发送
  → Paperclip API 接收 (express.json() 默认 UTF-8 解码)
  → GBK 字节被当 UTF-8 解码 → 中文变成 ? (0x3F)
  → 写入数据库
  → 前端渲染 → 显示 ???
```

**关键区分**：路径 B 的乱码是 `?`（ASCII 问号 0x3F），不是 U+FFFD。这是因为 `body-parser` 在遇到无法解码的字节时，替换为 `?` 而非 U+FFFD。

### 2.3 路径 C（存储损坏）和路径 D（读取/渲染损坏）：排除

**排除依据**：

| 排除项 | 证据来源 | 具体证据 |
|--------|---------|---------|
| 存储中编码错误 | ROU-66 §3 | PostgreSQL UTF-8 text 列，`postgres` JS 驱动默认 UTF-8，连接无 charset 问题 |
| 读取/渲染异常 | ROU-66 §4-6 | API 响应头 `charset=utf-8`，`react-markdown` 正确处理 Unicode，用户手动输入的中文正常 |

**对照验证**：ROU-67 发现 ROU-58 的 `resultJson.summary` 和 `resultJson.result` 中中文正常返回，进一步证明存储和读取链路安全。

## 三、历史乱码可逆性判定

### 3.1 路径 A（U+FFFD 替换字符）：不可逆

**原理**：GBK 字节被当 UTF-8 解码时，无法识别的字节被替换为 U+FFFD（`efbfbd`）。这是一个**有损转换**——原始 GBK 字节信息已丢失，无法从 U+FFFD 反推原文。

**证据**：
- Hex 分析显示乱码区域为连续的 `efbfbd` 序列
- 中间夹杂的 `d0be`、`c4bf` 等是 GBK 单字节被当 UTF-8 解码后的残留，但这些残留不足以重建完整原文
- GBK 是双字节编码，一个中文字符 = 2 字节，被当 UTF-8 解码后可能产生 1-3 个 U+FFFD，无法准确还原

**结论**：路径 A 的历史乱码**不可逆**，只能通过人工重写或从原始 agent 输出中恢复（如果 agent 的终端输出日志仍保留的话）。

### 3.2 路径 B（问号替换）：不可逆

**原理**：Windows 控制台将中文替换为 `?`（0x3F），原始中文字节信息完全丢失。

**证据**：
- ROU-54 评论 `17aa1d33`：全文除 ASCII 字符外全是 `?`
- `?` 是一对一替换，无法反推原始中文字符

**结论**：路径 B 的历史乱码**不可逆**。

### 3.3 轻度乱码（emoji/符号位置 1-2 个 U+FFFD）：部分可逆

**原理**：轻度乱码通常只在 emoji 或特殊符号位置出现 1-2 个 U+FFFD，原始文本的上下文信息保留较多。

**判定**：虽然技术上不可逆，但可以通过上下文推断原始内容，人工修复成本较低。

## 五、修复方向建议（不实施，仅记录）

### 5.1 路径 A 修复：服务端 GBK 回退解码

在 `decodeChunk` 中增加 GBK 回退逻辑：
1. 检测平台是否为 Windows (`process.platform === 'win32'`)
2. 检测控制台代码页 (通过 `chcp` 或 `process.env.CHCP`)
3. 如果是 GBK/CP936，使用 `iconv-lite` 的 `iconv.decode(chunk, 'gbk')` 解码
4. 作为 fallback，可以先尝试 UTF-8 解码，如果包含大量 U+FFFD 则回退到 GBK

### 5.2 路径 B 修复：agent 侧 curl 调用改进

修改 Qwen 适配器的 SKILL.md / prompt 模板：
1. 引导 agent 使用 PowerShell `Invoke-RestMethod` 替代 `curl`（PowerShell 原生 UTF-8）
2. 或在 curl 调用前设置 `chcp 65001` 切换到 UTF-8 代码页
3. 或使用文件传递 JSON body（`curl -d @body.json`），文件以 UTF-8 编码写入

### 5.3 历史数据处理

- 严重乱码（7 个样本）：不可逆，需人工重写
- 中度乱码（19 个样本）：不可逆，但可根据上下文推断后人工修复
- 轻度乱码（20 个样本）：影响小，可暂时忽略
