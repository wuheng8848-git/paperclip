## 背景与范围（人话）

**原诉求：** 人类一条评论下去，背后可能拉起很重的规程（全技能、长上下文等）。希望 **「叫醒智能体」这件事能分档**——有的只需要知会一声，有的才允许拉满 API / 技能 / 改仓库。远期还可以和标签、执行策略、角色等挂勾；默认想让公司别一开局就最重。

**首期实际做成了啥（和评审稿 26 一致）：**

- 每条事务上多了一个**可选的「评论叫醒档位」**（存在库里、接口能读能写、列表能拉出来），**默认偏省token的那种**。  
- **@提到某人** 的叫醒，在同样配置下会**再轻一档**（少塞上下文）。  
- **多条评论一齐进来**：在偏轻的档位下，会把正文 **去重、压成更短的摘要** 再塞进叫醒包，避免重复废话占满屏。  
- 详情里的标签自动帮人选档 **没有做**；第一期不碰这块。

## 交付物（可验收 · 人话对照）

1. **条款说清楚档位是啥、默认啥、谁覆盖谁** —— 工程侧在共享包里 **枚举 + 校验** 已有；若指对外的长篇产品文档，还没单开一篇（要的话走文书闸往 `doc/` / `docs/` 或计划区最佳实践补）。  
2. **代码真读这个字段** —— 服务端写库、列表带出；两处发评论/改事务的叫醒路径会把「算出来的档」带进叫醒快照；心搏组包时会按档决定是否合并压缩评论正文。  
3. **能对比轻档 vs 重档** —— 理想是同样一句评论、两套档位，看工具面或注入体积差多少；完整演示留到后面和 **042、045** 等一起加强。

## 落地说明（人话）

- **库里**：事务表多了「评论唤醒档位」一列；迁移已生成（随仓库迁移编号走）。
- **接口**：创建/更新事务时可选填该字段；拉列表时能读到。
- **评论叫醒谁**：不管是「改事务顺带留言」还是「专门发一条评论」，只要会触发叫醒，唤醒请求里都会带上**算好的档位**（@人 比叫醒经办人更省上下文那一档）。
- **心跳**：组包给 Agent 时，会按档位决定评论内容要不要合并压缩（轻档、多条时）。
- **单测**：专门测了「档位怎么算」「多条正文怎么压」，以及几条评论相关的路由叫醒行为。
