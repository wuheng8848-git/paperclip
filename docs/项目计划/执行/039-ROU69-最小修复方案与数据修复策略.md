## 一、最小修复点清单

### 1.1 路径 A 修复：服务端 stdout/stderr 解码 GBK 回退

**根因回顾**：Node.js `spawn` 子进程在 Windows 上默认使用系统代码页（CP936/GBK）输出 stdout。服务端 `decodeChunk` / `String(chunk)` 按 UTF-8 解码，GBK 字节被替换为 U+FFFD。

#### 修复点 A-1（主注入点，影响最广）

| 项目 | 内容 |
|------|------|
| **文件** | `packages/adapter-utils/src/server-utils.ts` |
| **行号** | 2214-2227（`decodeChunk` 函数） |
| **现状** | `chunk.toString("utf8")`，注释承认 CP936 问题但未实现回退 |
| **方案** | 引入 `iconv-lite`，在 Windows 平台检测控制台代码页，GBK 时用 `iconv.decode(chunk, 'gbk')` 解码 |
| **依赖** | 需新增 `iconv-lite` 包依赖（`packages/adapter-utils/package.json`） |

**修改草案**：

```typescript
// packages/adapter-utils/src/server-utils.ts

import iconv from "iconv-lite";

// 检测是否需要 GBK 回退（Windows + CP936/GBK 代码页）
function needsGbkFallback(): boolean {
  if (process.platform !== "win32") return false;
  // 常见 GBK 代码页：936, 54936 (GB18030)
  const cp = process.env.CHCP ?? "";
  return cp === "936" || cp === "54936";
}

const _useGbk = needsGbkFallback();

const decodeChunk = (chunk: unknown): string => {
  if (typeof chunk === "string") return chunk;
  if (Buffer.isBuffer(chunk)) {
    if (_useGbk) {
      // 先尝试 UTF-8；若包含 U+FFFD 则回退 GBK
      const utf8 = chunk.toString("utf8");
      if (!utf8.includes("\uFFFD")) return utf8;
      return iconv.decode(chunk, "gbk");
    }
    return chunk.toString("utf8");
  }
  return String(chunk);
};
```

**设计要点**：
- 仅在 Windows 平台启用 GBK 回退（Linux/macOS 不受影响）
- 先尝试 UTF-8，不含 U+FFFD 时直接返回（避免误转纯 ASCII/UTF-8 内容）
- 含 U+FFFD 时回退到 GBK 解码（iconv-lite 的 GBK 解码器能正确处理 GBK 双字节）
- 使用环境变量 `CHCP` 而非运行时调用 `chcp` 命令（避免同步子进程开销）
- 需要在适配器 spawn 子进程时传递 `CHCP` 环境变量（见修复点 A-5）

## 三、历史乱码可逆性评估

### 3.1 结论：全部不可逆

| 乱码类型 | 样本数 | 可逆性 | 原因 |
|---------|--------|--------|------|
| 路径 A（U+FFFD 替换字符） | ~38 个 | **不可逆** | GBK 双字节被当 UTF-8 解码后，原始字节信息已丢失为 U+FFFD，无法反推 |
| 路径 B（问号替换） | ~8 个 | **不可逆** | 中文字符被替换为 ASCII `?`(0x3F)，一对一有损替换，原始信息完全丢失 |
| 轻度乱码（1-2 个 U+FFFD） | ~20 个 | **技术上不可逆，但可人工推断** | 上下文信息保留较多，可凭语义推断原文 |

### 3.2 处理建议

| 严重程度 | 样本数 | 建议处理方式 | 是否需要数据修复脚本 |
|---------|--------|------------|-------------------|
| 严重（大段中文全损坏） | 7 | 人工重写或删除 | 否 |
| 中度（agent 名字/部分中文损坏） | 19 | 人工修正或标记 | 可选（批量清理脚本） |
| 轻度（1-2 个替换字符） | 20 | 可忽略或批量替换 U+FFFD 为空 | 可选 |

## 四、数据修复策略

### 4.1 策略选择

**推荐策略：标记 + 人工修正，不做自动替换**

理由：
1. 所有乱码均不可逆，自动替换无法恢复原文
2. 严重乱码（7 个样本）需要人工判断上下文后重写
3. 轻度乱码影响小，不值得冒自动修改的风险

### 4.2 数据修复脚本草案

**用途**：标记乱码评论，便于后续人工筛选和修正。

```javascript
// scripts/mark-garbled-comments.cjs
// 用途：扫描所有评论，标记含乱码的评论（不做修改）
// 运行：node scripts/mark-garbled-comments.cjs

const API_BASE = process.env.PAPERCLIP_API_URL || "http://localhost:3000";
const API_KEY = process.env.PAPERCLIP_API_KEY;

async function fetchJSON(path) {
  const res = await fetch(`${API_BASE}${path}`, {
    headers: { Authorization: `Bearer ${API_KEY}` },
  });
  if (!res.ok) throw new Error(`${res.status} ${res.statusText}`);
  return res.json();
}

function detectGarbled(text) {
  if (!text) return null;
  const ufffdCount = (text.match(/\uFFFD/g) || []).length;
  const questionMarkRuns = (text.match(/\?{3,}/g) || []).length;
  if (ufffdCount > 5 || questionMarkRuns > 2) {
    return {
      severity: ufffdCount > 50 ? "severe" : ufffdCount > 5 ? "moderate" : "light",
      ufffdCount,
      questionMarkRuns,
    };
  }
  return null;
}

async function main() {
  console.log("扫描所有事务和评论...\n");
  const issues = await fetchJSON("/api/issues");
  const garbled = [];

  for (const issue of issues) {
    // 检查标题和描述
    const titleResult = detectGarbled(issue.title);
    if (titleResult) {
      garbled.push({ type: "title", issueId: issue.id, ...titleResult });
    }
    const descResult = detectGarbled(issue.description);
    if (descResult) {
      garbled.push({ type: "description", issueId: issue.id, ...descResult });
    }

    // 检查评论
    const comments = await fetchJSON(`/api/issues/${issue.id}/comments`);
    for (const comment of comments) {
      const result = detectGarbled(comment.body);
      if (result) {
        garbled.push({
          type: "comment",
          issueId: issue.id,
          commentId: comment.id,
          createdAt: comment.createdAt,
          ...result,
        });
      }
    }
  }

  // 输出报告
  console.log(`扫描完成。发现 ${garbled.length} 个乱码条目：\n`);
  console.table(garbled);

  // 按严重程度分组
  const severe = garbled.filter((g) => g.severity === "severe");
  const moderate = garbled.filter((g) => g.severity === "moderate");
  const light = garbled.filter((g) => g.severity === "light");

  console.log(`\n严重: ${severe.length} 个（需人工重写）`);
  console.log(`中度: ${moderate.length} 个（需人工审核）`);
  console.log(`轻度: ${light.length} 个（可忽略或批量清理）`);

  // 输出 JSON 报告
  const fs = require("fs");
  fs.writeFileSync(
    "garbled-audit-report.json",
    JSON.stringify({ scanDate: new Date().toISOString(), total: garbled.length, severe, moderate, light }, null, 2)
  );
  console.log("\n详细报告已写入 garbled-audit-report.json");
}

main().catch(console.error);
```

  console.log(`模式: ${DRY_RUN ? "🔍 预览模式（不修改）" : "⚠️ 执行模式（将修改数据）"}`);
  console.log("");

  for (const item of lightItems) {
    // 读取评论
    const res = await fetch(`${API_BASE}/api/issues/${item.issueId}/comments`, {
      headers: { Authorization: `Bearer ${API_KEY}` },
    });
    const comments = await res.json();
    const comment = comments.find((c) => c.id === item.commentId);
    if (!comment) continue;

    // 清理 U+FFFD
    const cleaned = comment.body.replace(/\uFFFD+/g, "").replace(/\s{2,}/g, " ").trim();

    if (cleaned === comment.body) {
      console.log(`  [跳过] ${item.commentId} — 清理后无变化`);
      continue;
    }

    console.log(`  [${DRY_RUN ? "预览" : "修改"}] ${item.issueId}/${item.commentId}`);
    console.log(`    原文长度: ${comment.body.length}, 清理后: ${cleaned.length}`);

    if (!DRY_RUN) {
      // 注意：Paperclip API 可能不支持修改评论，需要确认
      // 这里仅做标记，实际修改可能需要直接操作数据库
      console.log(`    ⚠️ API 可能不支持 PATCH 评论，需确认修改方式`);
    }
  }

  if (DRY_RUN) {
    console.log("\n如需执行，请添加 --execute 参数");
  }
}

main().catch(console.error);

## 六、风险提示

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| `iconv-lite` 包体积影响 | 低 | iconv-lite ~340KB，对服务端可接受 |
| GBK 回退误判（纯 UTF-8 内容含 U+FFFD） | 低 | 仅在 Windows 生效，且先尝试 UTF-8 不含 U+FFFD 时直接返回 |
| `chcp` 命令同步调用开销 | 低 | 仅在适配器启动时调用一次，缓存结果 |
| PowerShell 5.1 `ConvertTo-Json` 编码 | 中 | PS 5.1 默认输出系统代码页编码，可能不是 UTF-8 |
| SSH 远程服务器编码不确定 | 中 | `ssh.ts` 修复点 A-4 只处理本地 Windows，远程服务器编码需另行处理 |
