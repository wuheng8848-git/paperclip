# 054 · 运行原子诊断：四标签与信息架构

## 背景与共识

- **最小原子**：单次 `heartbeat` **运行（run）** 是控制面上愿意拆到的最小单位；再往下（字节、原始日志行等）走别的取证面，与本页叙事分离。
- **依赖层级**：**公司 → 项目 → 团队 → 事务 → 执行** 链住绝大部分唤醒上下文；**事务侧可见的结果与评论** 往往由 **多次 run**（成功/失败/重试/续跑）叠出来——与本页「看这一轮控面发生了什么」不混谈。
- **读者目标**（在本原子内）：先回答三件事是否「看起来有问题」——**① 输出拼成的上下文/提示词**、**② 调度**、**③ 回传结果**。*具体是哪类业务问题* 允许后续下钻，**本轮不展开复杂取证**；先把系统 **已有数据与能力** 按结构拼进来。

## 信息架构：四个一级标签

| 标签 | 主要回答 | 备注 |
| --- | --- | --- |
| **唤起入队** | 唤醒/request 是否获准进入执行、是否被合并/排队挡在门外 | **预算**、**审批** 若在 **run 尚未进入可执行队列** 前挡路，归此类（未成功入队/阻塞理由）。 |
| **输入编排** | 控面写下的上下文、`contextSnapshot`、唤醒载荷、有效触发归因等 | 与 `adapter.invoke` 里**实际下发**可在文案上区分「计划 vs 已 invoke」。 |
| **执行回传** | 适配器执行阶段事件、状态/日志/用量（按现有 API 能拿到的） | **执行中**被预算等掐断 → 归 **执行中断** 类叙事，不叫「入队失败」。可设二级卡 **执行中断**（取消/超时/进程丢失/硬停等）；**不在此页展开「交付物成不成形」的细读**。 |
| **运行记录** | `heartbeat_runs` 行级：状态、时间、摘要、`resultJson`、错误、与兄弟 run 的关联入口 | 给一句链到事务/转写即可，不把长文交付物展开在这里。 |

**明确排除（不在此页细看）**：交付物质量、评论正文、事务长对话——**仅保留跳转入口**（issue/评论/转写）。

## 与现有实现的关系

- 当前 **运行清单** 已有：列表 + **详情页** `runs/:runId`（旧 `orchestration-injection/runs/:runId` 待重定向），卡片较多，**需按上表四标签重组展示**（避免单页堆叠、与读者「三条疑问」对齐）。
- 数据侧仍主要依赖已有能力：`heartbeatsApi.get`、 `heartbeatsApi.events`（如 `adapter.invoke`）及已解析字段；**不为本轮单开新 API**，除非实现时发现真缺口再另拆任务。

## 本轮交付范围（建议）

1. 在 **运行原子详情** 上落地 **四标签**（或等效主导航），内容以 **重组现有块** 为主，文案与 `ui/src/lib/i18n.ts` 同步。
2. 四标签内对齐三大疑问：**提示词/上下文** → 主落在 **输入编排** + **执行回传**（invoke 证据）；**调度** → **唤起入队** + 运行记录中的时间与状态；**回传结果** → **运行记录** + 执行回传中的终态/中断信息。
3. **预算/审批**：仅在 **尚未形成可执行 run** 或 **记录在唤醒链路上** 时，放在 **唤起入队** 说明；run 已建后的中途停止归 **执行回传** / **运行记录**。
4. **验收**：人类可走通「列表 → 详情 → 四标签下各块有内容或空态说明」；与 `AGENTS.md` / 双向真值相关的用户可见文案有中文落点。

## 技术实现（已定案，本阶段不编码）

以下为与讨论对齐的**实现契约**，开发时照此落盘；未写明处实现阶段再补，不在这里扩需求。

### 1. 页面骨架与主导航

- **四标签**作为运行详情页（`OrchestrationInjectionRunDetail`）的一级结构：**唤起入队**、**输入编排**、**执行回传**、**运行记录**。
- **实现方式**：使用 `ui` 既有 **Tabs**（shadcn，与设计体系一致），每个 `TabsContent` 对应一栏；**不改变**全局面包屑、侧栏、主内容区 `Layout` 行为。
- **标签与 URL**：优先 **同一路径** `…/runs/:runId` 下切换（`useSearchParams` 如 `?tab=input` 便于复制链接）或纯客户端 state——具体二选一再定，本单验收要求至少「刷新/分享场景可说明」。
- **唤起入队**在仅有「运行已存在」证据时可能内容偏少：允许 **短说明 + 链到编排闸门 / 心跳任务**，**不**为撑版面造假数据。

### 2. 现有详情块 → 四标签归类（真值表）

以下为 **当前** `OrchestrationInjectionRunDetail` 内各块搬迁归属；搬迁时以 **语义** 为准，组件形态（是否保留 `Card`）见 §4。

| 当前块（语义） | 目标标签 |
| --- | --- |
| 运行摘要：运行 ID、状态、来源、时间、智能体链接 | **运行记录** |
| 唤起归因（effective trigger / absorbed） | **输入编排** |
| 运行快照 JSON、唤醒载荷 JSON | **输入编排** |
| 适配器调用（adapter.invoke：命令、cwd、commandNotes 等） | **执行回传** |
| 最终提示词（分块、缓存对齐、prompt 指标 pill、整块/段落展示） | **输入编排**（拼装与控面视角）；若需强调「与 invoke 一致」，在 **执行回传** 内 **仅保留交叉引用或一句链**，避免两处维护两大段重复正文 |
| （日后）预算/审批挡在入队前 | **唤起入队** |

### 3. 智能体 `RunDetail` 并入时的归类（可分阶段验收）

与 `AgentDetail` 内 **RunsTab / RunDetail** 能力对齐时，**不重做控面诊断叙事**，按下表归位；**重操作区**（取消/重试/续跑）**只放一处**（建议 **运行记录** 底部或 **执行回传** 顶，二选一整仓统一）。

| 智能体运行侧能力（类型） | 目标标签 |
| --- | --- |
| 取消、重试、process_lost 续跑等 | **执行回传** 或 **运行记录**（整仓只选一端固定） |
| 用量/时长类指标、与 run 行强绑的摘要 | **运行记录** 或并入 **执行回传** 中与现有 Metric 区同组（避免三处碎片） |
| 转写 / stdout·stderr / 类日志流 | **执行回传** |
| 关联 issue、checkout 等链出 | **运行记录**（列表或链接即可） |

**原则**：智能体页偏 **操作与可读执行输出**；「控面拼什么」仍在 **输入编排**，避免把整段智能体运行 UI 嵌进 **输入编排**。

### 4. 布局与栅格（短双列、长全宽）

- **短内容**（以 KV、小段说明为主、垂直高度相近）：可采用 **响应式双列**，例如 `grid gap-4 md:grid-cols-2`；**窄屏单列**。
- **长内容**（长 JSON、`pre` 提示词、转写）：**全宽单列**，不强行与邻块并排，避免一列极矮一列极长。
- **中间态**：一侧短 KV、一侧长代码时，可用 **非对称栅格**（如 `md:grid-cols-[1fr_2fr]`），不必固定 50:50。
- **与 Card 的关系**：**不新增卡片类型**；现有块可继续用 `Card` 作带 `rounded-lg` 的容器，或改为 **单块 `border border-border rounded-lg` + 节标题**——以 **同一页内视觉节奏统一** 为落点；四标签下优先 **减少「半屏八张卡」的重复 header 条带**。

### 6. 建议实现顺序

1. **Tabs 骨架 + 四标签文案 + 空态**（各栏可为占位，验证导航与 i18n）。
2. **按真值表搬迁现有块**（功能行为不变，仅位置与顺序）。
3. **加栅格**：对「运行记录」短 KV、「输入编排」中可并排的短节做双列；长 JSON/提示词保持全宽。
4. **（可选本单内）** URL query 与标签同步。
5. **（下一阶段）** 智能体 `RunDetail` 块归类并入 + 是否抽 `HeartbeatRunDiagnosticsPanel` 类组件。

### 7. 风险与待定

- **重复内容**：最终提示词若在编排与回传两处出现，必须以 **单一真源 + 链接** 约定，避免双向真值分裂。
- **唤起入队数据空洞**：允许「暂无」说明，不阻塞发版；有 API 后再补。
- **性能**：`events` 轮询策略保持与现实现一致；拆 Tab 后勿重复 mount 多份重查询（可用同一 query、按 Tab lazy render 视实现权衡）。
