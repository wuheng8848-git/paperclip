# 023 — 文档更名、删除重复稿、修链（临时 Obsidian/中文知识库布局）

**返回：**[`../00.项目任务清单.md`](../00.项目任务清单.md)

> **性质说明：本套「编号 + 中文目录 + 删英稿」做法是临时的、本 fork 自用布局**；不等同上游 Paperclip 文档策略，也未必长期保留。后续若回归文档站或双语策略，可按本单「回滚要点」逐项评估。

## 元数据

| 字段 | 内容 |
|------|------|
| 目的 | 在本仓库内形成 **中文优先、Obsidian 左侧可读** 的文档树；统一产品名词；去掉中英重复稿中**已对齐**的一侧；改名/挪目录后 **修链** |
| 状态 | 见文首 `status:` |
| 证据 | 已落地目录与提交记录（如 `docs/部署`、`docs/适配器`、`docs/API接口` 编号；`docs/指南` 及子目录；`board-operator` / `agent-developer` 英稿删除等）——具体 SHA 由执行人在推进/收尾时补写 |

## 已采用的做法（路子）

1. **Obsidian 排序**  
   在需固定阅读顺序的目录内，给文件名加 **`01`、`02`…** 前缀（两位数字 + 空格 + 原「中文标题 + 英文 slug」文件名），便于资源管理器/库内左侧按名排序。

2. **中文目录名**  
   顶层与二级目录尽量用中文（例：`docs/guides` → `docs/指南`；`board-operator` → `董事会操作员`；`agent-developer` → `智能体开发者`），与 `docs/部署`、`docs/适配器` 等并列习惯一致。

3. **名词体系统一（中文正文）**  
   - **智能体**：指 Paperclip 内 agent 执行者（原混用「代理」处已统一，**保留**专名如 **代理公司规范**）。  
   - **事务**：指 issue 实体及与之绑定的状态/线程/文档等（与日常「问题/故障」含义区分；示例评论里「对齐有问题」等 **不** 改成事务）。  

4. **删英稿的前提**  
   仅当存在 **已与英文稿对齐的中文稿**（结构、链接目标、代码块/API 路径一致）时，删除该主题下的 **英文平行稿**，避免两套真值。

5. **修链**  
   任意 `git mv`、重命名、改路径后：在仓库内 `rg` 旧路径（含 `/guides/`、`docs/guides`、`某英文目录名` 等），更新 Markdown/配置中的相对链接与站点式路径（如 `/指南/…`）。

## 最佳实践

- **先 grep 再动手**：改名/删除前列出所有引用，改完再 grep 一遍旧串，避免死链。  
- **批量替换用明确模式**：优先整条路径替换（如 `/guides/board-operator/` → `/指南/董事会操作员/`），避免误伤 URL（例如外站 `…/docs/guides/evaluation-…`）。  
- **编号插入**：中间插入新篇可用 `04a` 或在 `10` 后开 `15` 段；避免频繁整体重编号。  
- **未翻译英文稿必须保留（重要）**  
  - 若某目录 **仅有英文**、**没有**对应中文译本：**不得**仅为「去英文化」而删除；应 **保留英文稿** 作为当前唯一真值。  
  - 可选：旁挂占位中文 stub（链接到英文）或列入翻译 backlog；**以 00 主表 / 本单跟踪** 即可。  
- **双语 intentionally**：`docs/start/core-concepts.md` 与 `核心概念 core-concepts.md` 一类若产品需要双语入口，删除策略 **不适用**；只改链接目标，不强行并稿。  
- **临时性**：重大目录策略变更前在本单或 `01.项目需求说明.md` 记「残留风险/回滚」；合并上游前对照差异清单。

## 验收标准（本任务单自身）

1. 本文件可被用作后续「同名操作」的检查表。  
2. **「英文未译则保留」** 写清楚且与历史已做删除（仅针对已对齐成对文档）不矛盾。  
3. 若继续执行本路线：对应变更在任务单末尾 **简要记录** 日期、范围、关联 commit（可选）。

## 验证证据

- 执行人在推进或关闭本单时，于本节补充：涉及目录列表、`rg` 检查说明、相关 commit SHA。
