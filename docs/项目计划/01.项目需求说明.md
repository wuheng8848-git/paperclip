# 需求说明

## 使用方与痛点

**吴衡** — 一个人 + 一群 AI，建一家可复制的软件开发公司。

当前痛点：

- 手动管理多个 agent 的状态、任务分配、执行追踪——没有统一控制平面  
- 不知道 agent 在干什么、花了多少钱、产出是什么  
- 每次启动 agent 都要手动操作：创建 issue → checkout → 启动进程 → 盯心跳  
- 贾维斯（CEO/策略层）和赫尔墨斯（信使/路由层）缺少可靠的执行平面  

## 目标结果

当 Paperclip 在当前环境中就绪时：

- routic 公司的 agent（b064fe96 Cursor agent）可以通过心跳自动拉取排队中的 run 并执行  
- 完整的 issue→checkout→heartbeat→run→completion 链路至少跑通一次  
- 吴衡可以通过 Paperclip Board 看到：谁在干什么、花了多少钱、产出是什么  
- 僵尸 run 能被识别和清理（不会堆积阻塞后续任务）  

**不应改变：**

- Paperclip 上游代码的核心行为（本 fork 不是上游贡献）  
- 现有数据库结构和 API 契约  
- 已有的 routic 公司配置和 agent 定义  

## 文档与代码（双向真值）

**已明确的两个目的：**

1. **文档指导代码** — 用中文资料与需求说明界定系统应提供的能力与边界，实现与之对齐，禁止无背书擅自「缩水」。  
2. **代码回写文档** — 行为或配置语义变更时，同步更新 `doc/`、`docs/`（二级主题目录）及本文件，避免出现「只看懂 Board 界面、不懂后台在生效什么」的第二套真值。

**计划任务推进受阻的已知原因（须持续收口，不等同于单纯「软件坏了」）：**

- **适配器过度最小化**：为省事少接字段、少实现与编排层一致的能力，与文档或 SPEC 不一致 → 运行失败；**须先定文档/需求边界再写代码**，不得反向用「能跑一行算一行」替代契约。  
- **中文一手资料不足**：操作者若只知道 UI 长什么样、不知道 `adapterType`、心搏、技能注入路径、关单条件等，**无法稳定指挥 AI**；`docs/` 须说明「界面背后哪条功能在生效、改哪里才能顺手用起来」。  
- **产品与编排语义未完全收口**：技能如何进入单次 `execute`、心搏与 wake、事务如何结束等，若边界未写清，**心跳乱跳、工单收不了尾**易被误判为缺陷列表；需把**设计意图与限制**写进需求/文档并与实现对账，区分实现 bug 与规则未交代清楚。

本节约束针对**本仓库可见性与契约完整性**；对上游与「不应改变」清单的边界，仍以 **「目标结果」** 上文的 **不应改变** 为准。

## 产品口径（已定）

以下由董事会 **当轮定死**：后续实现（代码 / 技能 / 人设）须对齐；若与当前运行中的自动化路径冲突，**以本口径为准**推动改版（本 fork 不对上游默认行为默认同化）。

### 手动 pause 智能体

1. **语义**：操作者在 Board **手动 pause**（含 **manual pause**）表达的是 **停机意志**，**不构成** Paperclip 默认意义上的「待系统自动抢救的故障」。  
2. **期望行为**：在人类 **未复机（resume / unpause）** 之前，**等待人类即可**；**不得**仅凭「经办智能体处于 pause」触发 **stranded / recovery / 自动加长阻塞工单链** 作为**默认**路径（除非另有 **显式实例配置**或董事会 **另行授权**）。  
3. **落地选项（讨论顺序）**：**服务端门禁（改代码）** 作为最终硬闸；**技能** 约束编排侧不乱开 recovery / 不误判；**CEO 人设** 只做口径复述与优先级——**不得指望单靠人设或技能替代硬闸**。  

**复盘与张力记录：** [`探查/021-探查-CEO运行可观测性与活动面缺口.md`](探查/021-探查-CEO运行可观测性与活动面缺口.md) §4.6、§7。

以下为 fork / 本仓库实战中已识别的**已知风险与操作提醒**（与上文「开放问题」区分：此处偏**可陈述的坑与边界**，后者偏**仍待拍板的问题**）。

- Fork NTFS：勿使用 **tsx file-watch** 全仓库监听；日常用 **`scripts/start-paperclip-dev-external.ps1`** 或仓库根 **`pnpm dev`**（`vite build` 仍易挂，见 **`AGENTS.md`**）。  
- Hermes 外置化分支稳定性待观察。  
- routic agent 曾需人工取消 **暂停** 才能拉活（005 前 Cursor agent）。

- **Board 中英混排**：**ROU-20** 已归因（见 `探查/002-探查-控制台任务中英混排.md`）；若要做一致中文体验，需单独决策 Board i18n / 系统 creation 文案 / agent 模板。  

## 需求草案：CLI 流式输出

### 现象（吴衡侧）

在终端里跑 **Paperclip 心跳 / agent 执行**（例如通过 `paperclipai` 相关命令观察某次 `heartbeat-run`）时，体感是 **进程跑完后才一大段输出**，而不是像本地直接跑 `cursor` / `codebuddy` 那样逐 token / 逐行刷出来。

### 目标

- **能选**：至少有一种 **官方支持的用法**，在终端里 **尽量接近实时** 地看到子进程 stdout/stderr（可接受「按行」或「按 JSONL 事件」粒度，不强制 token 级）。  
- **可解释**：文档写清「为什么有时仍是一坨出来」（子进程缓冲、非 TTY、适配器解析策略）。  

### 现状（基于本仓库代码，便于评估）

1. **服务端（heartbeat → run）**  
   子进程 stdout/stderr 在 `adapter-utils` 的 `runChildProcess` 里按 **`data` 事件** 追加；`heartbeat` 里对每段 chunk 会 **`runLogStore.append`** 并 **`publishLiveEvent`**（`heartbeat.run.log`）。即：**落库与对前端的 live 通道已经是「有 chunk 就发」**，瓶颈往往不在这一层。

2. **终端 CLI（`cli/src/commands/heartbeat-run.ts`）**  
   - 用 **`while` + 约 200ms 轮询**：拉 `/api/heartbeat-runs/:id/events` 与 `/api/heartbeat-runs/:id/log`，不是长连 SSE。  
   - **未加 `--debug` 时**：stdout 会先拼进缓冲区，再 **按换行切行** 后交给各 adapter 的 `formatStdoutEvent` 做「好看」输出，**整行未到齐前不会打印**；stderr 仍会较快透出。  
   - **`--debug` 时**：stdout/stderr **按 chunk 原样** `printRawChunk`，最接近「流式」。

3. **外部 CLI（Cursor / CodeBuddy 等）**  
   若子进程发现 **stdout 不是 TTY**，库/运行时可能 **块缓冲**，Paperclip 只能收到「更大块」的 `data`，这是 **进程外不可完全控** 的部分。

### 可选实现方向（按改动面排序）

| 方向 | 内容 | 难度 | 说明 |
| --- | --- | --- | --- |
| A. 文档与开关 | 在 CLI / 开发文档中明确：**观察实时输出请带 `--debug`**；说明非 debug 为「按行美化」 | **低** | 零或极少代码 |
| B. 缩小轮询延迟 | 将 `POLL_INTERVAL_MS` 可配置或默认调小（如 50ms） | **低～中** | 更吃 CPU/请求数 |
| C. 非 debug 也支持「原始 stdout 流」 | 增加例如 `--raw-stdout`，跳过 `formatStdoutEvent`，但仍按 chunk 写终端 | **中** | 与 debug 类似，语义更清晰 |
| D. CLI 改 SSE / WebSocket | 订阅服务端已有 live 事件，去掉轮询 | **中～高** | 要接 API、鉴权、重连、与现有 `events` 分页对齐 |
| E. 子进程 PTY + 全适配器验收 | 伪终端骗过子进程行缓冲；各适配器验证 | **高** | 跨平台、维护成本高 |

### 难度结论与是否搁置

- **「像 Board 里那样完整结构化 + 终端里又完全实时」**：**偏高**（格式化与缓冲天然矛盾，外加第三方 CLI 缓冲行为）。  
- **「能实时看到子进程在吐什么」**：**不高**——**已有 `--debug`**；若仍嫌不够，再做 **C（独立 `--raw-stdout`）** 或 **B** 即可。  
- **D / E**：建议 **暂时搁置**，除非产品明确要把「终端体验」做成一等公民并愿意付维护成本。

### 建议的下一步（本 fork / routic）

1. 先在本需求下记一笔：**验收路径 = 带 `--debug` 跑一次心跳 CLI**，确认是否已满足「流式」预期。  
2. 若仍不满意：开 **小改动**（B 或 C），不要一上来 SSE/PTY。  
3. **D / E** 进入 `docs/项目计划` 的 backlog 仅作备选，**不排进当前 sprint**。

## 第一轮复盘后需求扩展（2026-05-15）

基于 routic/test-co 实战 3 天推进 + 4 公司 27 issue 验证，**原"范围外"的"新 adapter 开发"不再成立**——CodeBuddy 适配器的 spawn-and-forget 模式是 66 僵尸进程 + 75% 失败的根因之一，必须纳入范围。

### 新增范围

#### R1：CodeBuddy 适配器重写

当前适配器核心缺陷：spawn CLI 进程后不追踪、不回收、不管理生命周期。

| 子需求 | 说明 |
|--------|------|
| PID 树追踪 | Windows 下通过 WMI/Get-CimInstance 建立父子关系树，替代 `resolveProcessGroupId`（当前返回 null） |
| 进程池 + TTL | Pool(N) 个热 CLI 进程，TTL 30min 或 N run 后强制回收重建；避免每次心跳冷启动 |
| 熔断超时 | maxTurns 熔断 + 超时自动 failed + 进程 kill，不再依赖人类手动 done |
| 状态可观测 | 适配器层上报 run status + PID 列表 + CPU/内存 |

#### R2：进程兜底回收

| 子需求 | 说明 |
|--------|------|
| 临时脚本 | `taskkill /F /T /PID` 定期清理僵尸（立即可用） |
| 全局扫描器 | 每 5 分钟对比 DB 与 OS 进程，清理漏网之鱼 |
| PID 注册表 | `run_process_pids` 表，启动注册、退出反注册 |

#### R3：多 Agent 委派链验证

test-co 的 CTO/Code 从未收到过 run（TES-1 只走 CEO）。需要验证 CEO→CTO→Code 三级编排闭环，前提是先提高单 agent 成功率。

#### R4：Agent 失败根因分析

routic 12 个 done issue 中只有 3 个 run 真正 succeeded（成功率 25%）。需系统分析：为什么 agent 频繁卡死/跑偏/无限循环？区分模型能力问题 vs 适配器缺陷 vs 编排设计问题。

#### R5：CLI 并发能力验证

3 agent × heartbeat 10min × maxConcurrent 10 = 峰值 30 CLI 同时跑。需实测：进程池上限多少合适？排队机制能否防止进程洪峰？

#### R6：操作文档完善

当前"怎么跑起来"链路（issue→checkout→CLI→心跳）散落在代码中，需产出独立操作手册。

### 更新后的范围边界

**范围内（新增用 + 标记）**：
- 理解 Paperclip 完整生命周期
- routic 公司的 agent 心跳验证
- 端到端执行一次完整任务
- 僵尸 run 诊断与清理流程
- 本地开发环境稳定性
- **+ CodeBuddy 适配器重写（PID追踪 + 进程池 + 熔断超时）**
- **+ 进程兜底回收（临时脚本 → 正式方案）**
- **+ 多 agent 委派链验证**
- **+ Agent 失败根因分析**
- **+ CLI 并发能力验证**

**范围外（不变）**：
- Paperclip 上游新功能开发
- Cloud 部署 / 多用户认证
- 生产环境配置
- 向 upstream 提交 PR
