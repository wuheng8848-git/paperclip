# 047 · 适配器原始记录与异步解析器拆分 — 技术设计

**日期：** 2026-05-18  
**对应执行单：** [`../执行/047-适配器_stdout解析失败与结构化摘要.md`](../执行/047-适配器_stdout解析失败与结构化摘要.md)  
**长期需求（母本）：** [`../长期需求/27 适配器执行与异步解析器拆分-长期需求 2026-05-18.md`](../长期需求/27%20适配器执行与异步解析器拆分-长期需求%202026-05-18.md)

## 1. 文档目的

把 **「子进程已结束且原始输出已持久化」** 与 **「结构化字段（usage、摘要、终态）已生成」** 在控制面 **拆开**，使 **解析失败** 不再等价于 **本次执行失败**，并为 **跨 OS / 跨 CLI 输出形态 / 超大 stdout** 提供可演进管道。

本文是 **实现期契约草案**；字段名、表结构以实现 PR 与迁移为准。

---

## 2. 粒度辨析（避免照搬 IDE 会话模型）

| 概念 | Jarvis 记忆管线（参照） | Paperclip 本文范围 |
| --- | --- | --- |
| 大包来源 | **一个 IDE 会话**内含多轮对话，JSONL 体量可极大 | **单次 `heartbeat run`**：一次适配器拉起；体积主要来自 **stdout/NDJSON** 及 CLI **内部多圈**（`num_turns`） |
| 编排级「多段」 | 同日多 session 合并、按日聚合 | **同一 `issue` 下多条 run**（接续、重试、排队）：**叙事在工单/run 链**，不把多条 run 的 raw **糊成单条 blob** |
| 借鉴什么 | **分层、分路径、分片+manifest、管线状态** | **不默认**为每趟 run 复制「LLM 转日报」全链路 |

---

## 3. 职责边界

### 3.1 适配器（现有 `packages/adapters/*/execute` 演进方向）

**必须：**

- 组装命令、环境、stdin/提示词注入；
- 拉起子进程，按现有契约写 **日志流**（NDJSON 等）；
- 将 **原始输出** 以可重现形式 **落盘到控制面真源**（沿用现有 `logRef` / 扩展 spill 路径亦可）；
- 在 **进程生命周期结束** 时产出 **「执行阶段」** 结论：**退出码、信号、原始记录句柄、字节量**。

**不得再承担（逐步迁出）：**

- 同步阻塞的「整包 JSON 解析失败 → 整次 `AdapterExecutionResult` 失败」作为 **唯一终局**；
- 把 **超大 stdout** 的解析性失败与 **进程执行失败** 混在同一 `errorCode` 语义里。

### 3.2 解析器（新建独立模块，建议包名实现期定，如 `packages/cli-output-parsers` 或 `packages/adapter-utils` 子路径）

**必须：**

- **异步**（相对 run 结束事件）消费 **原始记录**；
- 按 **`adapterType` + 输出格式 / 版本** 选择 **解析策略**（纯函数 + fixture 单测）；
- 产出 **结构化摘要**：至少 **`usage`、`num_turns`、终结 `result` 片段、session 指针** 等控制面已有消费字段；失败时产出 **解析错误码 + 可诊断截断**，**不删原始**。

**不得：**

- 拉起进程或改写适配器 env/cwd 契约。

### 3.3 控制面（`server/`）

**必须：**

- 将 **run** 的 **执行完成** 与 **解析完成** 分层展示与登记；
- 解析任务 **可重试、可换策略**（幂等、显式 `force` 重建对齐 Jarvis `pipeline-state` / `--force` 思路）；
- **issue 级聚合**（多 run 用量合计、run 链）放在 **台账/查询层**，不合并原始字节。

---

## 4. 状态模型（草案）

建议 **同一 `heartbeat_runs` 行** 扩展语义（或 **旁挂 `run_parse_jobs`** 表 —— 实现期二选一）：

| 阶段 | 含义 | 与现有 `status`/`errorCode` 关系 |
| --- | --- | --- |
| **执行完成** | 子进程回收 + 原始记录已提交 | 与今日 **`failed/success`** 中的 **进程级** 语义对齐；**不因同步 parse 失败而标 failed** |
| **解析中** | 原始已可读，解析任务queued/running | **新增**或沿用扩展字段（如 `parseStatus`） |
| **解析失败** | 策略无法产出结构化字段 | **独立 error 域**（如 `parseErrorCode`），**不覆盖** 执行阶段成功事实 |
| **解析完成** | 结构化字段写入 | `resultJson`/usage 等由解析管线写入 |

**原则：** 观测与排障 UI **两层**：「跑得怎样」vs「账算出来没有」。

---

## 5. 体量与降级策略（对齐 047 C1–C2）

1. **阈值分级**（具体数字实现期调参）：  
   - **小**：单 pass 解析或轻量流解析；  
   - **中**：截断入口 + 仅解析尾部 `result` 帧；  
   - **大**：**分片** + **`manifest`**（块数、哈希、顺序），分块解析 / 渐进组装。  
2. **原始优先**：任一楼层失败，**原始记录仍可取证**，与长期需求 **§3.2「原始记录一等公民」** 一致。  
3. **与 047 交付物关系**：本设计不替代 **047** 中具体 **CodeBuddy / spill** 补丁，但 **定义** 这些补丁所服务的高层拆分。

---

## 6. 分期落地（降低一次性风险）

| 阶段 | 内容 | 验收 |
| --- | --- | --- |
| **M1** | 控制面：**执行成功 + 解析失败** 可区分登记；适配器：**原始已落盘** 即可视为执行路径成功（解析异步补写） | 样例 run：人工注入 parse 失败，**执行位仍为完成** |
| **M2** | 抽取 **CodeBuddy（及 047 范围适配器）** 内联 `parse*` 至 **解析器包**，适配器变薄 | 单测 fixture 覆盖数组 / JSONL / 截断 |
| **M3** | 分片 + manifest + 重试队列；可选 **issue 级用量聚合 API** | 超大样例；与 **`015`** 计量口径对账（若涉及） |

---

## 7. 测试矩阵

- **单元**：解析策略 **fixture**（Windows 换行、乱码边界、半包 JSON、仅尾帧 result）。  
- **集成**：mock 子进程 + 真实 **logRef** 读写；解析任务重跑 **幂等**。  
- **回归**：现有 **`codebuddy_parse_error`** 场景 → 期望迁为 **解析失败** 而非 **整 run 失败**（需与产品确认 UI 文案）。

---

## 8. 风险与开放问题

- **幂等与重复解析**：同一 run 多次解析 **以最新策略覆盖** 或 **版本号** 留痕，避免账单双计（与计费表联动时单独立项）。  
- **延时**：异步解析导致 **看板短窗内 usage 为空** —— UI 需 **解析中** 占位（或读 CLI 直报 usage 的中间态，实现期取舍）。  
- **安全**：原始记录可能含用户路径与密钥片段 —— **权限模型**与今日 log 访问一致，不新开外泄面。
