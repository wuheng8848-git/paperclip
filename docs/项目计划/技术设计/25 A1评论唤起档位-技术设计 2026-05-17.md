## 1. 要不要单独技术设计

**要。** 理由：牵涉 **契约（shared）**、**多条 HTTP 唤起入口**、**`heartbeat` run 准备全链**、与 **042 / 045 / 046 / 050** 的交界；若不先把 **枚举语义、覆盖优先级、合并规则、分期范围** 写死，容易半成品或返工。

---

## 2. 目标与分期（建议）

### 2.1 一期（服务端可控、验收成本低）

在 **`issue_commented` / `issue_reopened_via_comment` /（可选）`issue_comment_mentioned`** 路径上，实现 **档位进入 `contextSnapshot`**，并在 **`heartbeat` 准备 run** 时施加：

| 能力 | 一期是否约束 | 说明 |
| --- | --- | --- |
| **`paperclipWake` 评论内联体量** | ✅ | 已有常量上限基础上，按档收紧或放宽（或低档只传 id、不传 body）。 |
| **公司 `paperclipRuntimeSkills` 注入** | ✅ | 低档：仅 `required`/白名单；高档：现状等价。 |
| **issue 线程 `@skill` 提及合并进 config** | ✅ | 低档：忽略或仅允许显式 mention 的子集（需规则）。 |
| **`paperclipTaskMarkdown` / continuation 摘要** | ✅ 可选 | 低档可省略或极短。 |
| **「禁止改仓」工具白名单** | ⚠️ 视适配器 | 仅服务端少灌字 **不等于** CLI 不能写盘；真硬闸要适配器/CLI 能力对齐（见 §6）。 |
| **API 调用（ Paperclip MCP / 适配器工具）** | ⚠️ 二期 | 与「工具」边界强相关，建议与 **046** 合订后再做硬语义。 |

### 2.2 二期

- 与 **042** 的 **effectiveTrigger / 合并** 一致：多评论 batch 时档位如何归并。
- 与 **050** 的 **预览与展示**。
- **推理档位**（**046 / 24§B5**）与档位联动字段。

## 3. 档位枚举（产品草案 · 实现须冻结命名）

与母本 §A1 对齐，建议 **有序枚举** `commentWakeTier`（数值或 lexicographic 便于 `max` 归并）：

| 档位 | 代号（示例） | 意图（给操作者看的） |
| --- | --- | --- |
| 0 | `receipt_only` | 仅回执：极小上下文，可不拉全线程正文。 |
| 1 | `read_thread` | 读线程即止：可给摘要级 thread，不扩张 Skill。 |
| 2 | `allow_api_context` | 允许为理解而拉 API/增补材料（仍可不启全 Skill）。 |
| 3 | `allow_full_skills` | 允许全 Skill（与现行为接近）。 |
| 4 | `allow_repo_write` | 允许改仓：默认满配 + 若章程限制 CEO 写代码则仍受 **046** 约束。 |

**默认档（已评审）：** **轻档为默认**（具体枚举名实现冻结；与 `26`「以实现为准」一致）。

---

## 4. 配置落点与覆盖优先级（**已评审**）

**工单覆盖存哪（已决）：** **`issues` 表顶层单独字段**（如 `commentWakeTier`，**不是**塞进 `executionPolicy` JSON、**也不是**挂在单条 `issue_comments` 行上）。语义：**这条工单**上的评论叫醒用哪一档，覆盖公司默认。

**本期不做：** 按 label 自动映射档位（见 **`26`**）。

**后续可渐进：** 标签规则、`executionPolicy.mode` 映射、`agent.role` 映射等 —— 与 **041** 分期一致。

**覆盖优先级建议（实现时写死）：**

1. **工单 `commentWakeTier`**（非空则赢）  
2. **公司 / 实例默认**（轻档）  
3. **@ 非经办** 路径单独更轻（见 **`26`**）  

（原列 2–5 里未上线的能力不要与 **1** 双真值并存。）

## 5. 服务端数据流（实现草图）

1. **解析** `resolveCommentWakeTier({ issue, agent, companyDefaults }) → tier`（`issue.commentWakeTier` 优先，其次公司默认，@ 路径单独规则见 **`26`**）。  
2. **`issues` 路由** `addWakeup`：把 **生效档位** 写入 `contextSnapshot`（字段名与 **`041`** 冻结一致，如 `commentWakeTier`）。  
3. **`enrichWakeContextSnapshot` / `mergeCoalescedContextSnapshot`**：多评 merge 时应用 **§7 规则**。  
4. **`buildPaperclipWakePayload` 前/内**：按 tier 裁剪 `comments`、是否附带 `continuationSummary` 全文等。  
5. **Skill 段**（`resolveRunScopedMentionedSkillKeys`、`listRuntimeSkillEntries`）：按 tier 过滤后再 `applyRunScopedMentionedSkillKeys` / 构造 `paperclipRuntimeSkills`。  
6. **活动 / run 元数据**（可选）：写入 `heartbeatRuns.contextSnapshot` 已含 tier，供 **050** 时间线。

---

## 6. 适配器与「硬闸门」边界

- **软闸门（推荐一期）：** 少注入、少 Skill → 降 token、降胡写概率。  
- **硬闸门：** 「不写仓」「不调某类工具」需 **各 `packages/adapters/*` / CLI** 支持（额外参数或 config）。未统一前，文案上应对操作者诚实：**低档 = 控制面尽力裁剪，不承诺 CLI 层绝对禁止**。

---

## 7. 与 042（多源合并）的交界 · **评审已定（见 `26`）**

**批处理 / defer** 合并多条评论进同一轮时：

- **不再**以「就高 / 就低」两档合并为主叙事。  
- **工程侧**对多条评论做 **摘要、去冗余、压反复** 后再注入 wake 载荷；**全量评论只在库中**，**不**指望模型逐条扫全线程。  
- **档位若仍需合成**（与 042 `effectiveTrigger` 对齐时）：在 **已摘要的意图** 上再定规则；细则在 **`041` / 042** 实现阶段写死。  
- **例外：** 人类显式钉死档位（**050 §E3**，本期未做 UI）——仍可按 **`26`** 单条规则处理。
