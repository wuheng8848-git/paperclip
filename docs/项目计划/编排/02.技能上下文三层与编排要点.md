## 1. 三套上下文：别混在一起谈「技能」

一次无头跑里，模型看到的「长文」经常来自 **三条独立管道**。编排若不讲清楚，就会叠床架屋、token 与心智一起爆。

### 第一道：回形针（控制面 · skill 管线）

- **发行时自带的技能包**：随适配器/运行时列出的 Paperclip 技能目录（谁标 `required` 谁默认可能进跑次）。  
- **公司维度登记的技能**：心跳把该公司的 `paperclipRuntimeSkills` 整表塞进配置 —— 含义是「**这家公司允许用哪些技能目录**」，**不等于**每一跑次都把每一段正文灌进模型。  
- **本跑次要带哪些（点名）**：  
  - 配置里 **`paperclipSkillSync.desiredSkills`**（显式），和/或  
  - 事务/评论里 **@ 技能** 合并进 desired，且始终与 **必选技能** 做并集。  
- **和「角色」的关系**：产品里未必有单独的「角色技能表」字段名；**实作上**常体现为 —— **某智能体配置 + 公司技能表 + 本跑 desired/@** 三层合流。编排口语可以叫「**角色这次该带什么**」，真值仍以 config 与服务端写入为准。

**一句话：** 回形针管的是 **「发现条目 → 按规则选中 → 适配器注入（挂目录或卷进 prompt）」**。

### 第二道：CLI / 工具自己那一套

- 各编码 CLI 自带的 **全局 skills 目录**、**默认 home**、**工具自己的发现顺序**。  
- Paperclip **不删除**这些规则，多半是起进程时 **换 home、同步目录、或只把「该出现的」链进去**。  
- **要「削弱」工具自带列表**：常用手段是 **独立 home**（例如专用 `CODEX_HOME` / 管理态 home）、**容器/沙箱只带最小文件树**、**工作树/仓库侧干净 clone**，让「用户本机攒的全局 skills」**根本进不了这次进程的搜索路径** —— 具体键名与适配器挂钩，不在此复述命令。

**一句话：** 这是 **工具生态自己的菜单**；和 Paperclip 选中技能 **可能重名、可能路径重叠**，要靠 **隔离环境** 控量，而不是假定「回形针关掉就只剩一条管」。

### 第三道：项目（仓库）自己带的说明资产

- **AGENTS.md、`./` 长说明**、各厂牌 **规则文件**（如部分工作流里的 rules）、**自维护技能清单或规程** 等。  
- 往往由 **CLI 或工作区策略自动读**，与 Paperclip 注入 **并行存在**。  
- 编排要警觉：**同一段规程不要在「公司技能」里再复制一份** —— 字面上两路进去就是双倍上下文。

**一句话：** 这是 **仓库/工作区约定**；管的是「在这个 repo 里怎么干活」，不是 Paperclip 公司库那一套。

## 2. 三套之间的关系（编排视角）

| 维度 | 第一道（回形针） | 第二道（CLI） | 第三道（项目） |
| --- | --- | --- | --- |
| 谁为主数据源 | 控制面 config、心跳、技能注册表 | 工具默认目录与 home | 仓库树内文件 |
| 典型失败模式 | desired / @ 过多、与 wake 重复灌 | 全局 skills 与本次注入叠在一起 | AGENTS 巨章 + Paperclip 再灌同款 |
| 编排杠杆 | **选对跑次技能集、去重、封顶（见 045）** | **独立 home / 容器 / 干净 worktree** | **节制重复注册、长文单一真源** |

## 3. Skill 组件：编排要盯的重点（先不谈 wake、评论、别的平面）

在 **暂不考虑** 其他组件（例如 wake JSON、全串评论、运行时服务面）的前提下，**只谈「技能组件」**，编排侧应长期盯住：

1. **本跑次的「选中集」是否收敛** —— 隐式（仅 required）vs 显式（`desiredSkills` ∪ @）；是否与角色/事务意图一致。  
2. **注入形态是否与适配器匹配** —— 目录 symlink 与 prompt bundle 两类不要 mentally 混成一种「都等于全文进去了」。  
3. **与第三道项目说明的去重** —— 同一规程不要两处各写一长段。  
4. **与第二道 CLI 全局技能的隔离** —— 跑次是否应绑定 **专用 home / 镜像内目录**，避免「用户本机全家桶」进入本次运行。

**落地缺口与产品演进**（占位链）：执行单 **045**（懒加载、上下文封顶）、探查 **025**。**2026-05-17**：045 首轮已在 **Qwen / CodeBuddy** stdin **`skill_note`** 与 **`shouldMinimizeAdapterRuntimeSkillNotes`** 上落地「轻档 + 续会话 → 单行指针」，**不等于** B1 全文懒加载或 B3 AGENTS 封顶已结案。

## 4. 基线、接续轮与「按需下发」（产品共识 · 技能主线）

- **默认发行技能包（含常见「8 包」叙事）**  
  **当前阶段可以不先动基线**（避免与「每轮下发多少」混谈），**不等于**以后永远不能换包、删包、收窄 `required`。若改基线，单独立项、单独验收。

- **懒加载在本主题的口径**（与「第一轮不给无头」无关）  
  重点在 **同一 issue 多轮接续**：**第二轮起**编排侧默认 **不再自动灌全套技能正文/全目录**；若存在 **第三、第四轮** 接力，**仍按编排裁剪按需下发**，而不是跑到后面又恢复「全家桶让模型自己选用」。

- **角色优先**  
  理想形态是 **按角色锁定小技能包**（参见 **§5** 中 **08 系列**）。若工程暂缺强类型「角色→技能」映射，可退化为：智能体侧 `desiredSkills`、事务 **@ 技能**、或后续旁路路由——**显式写清退路**，但不把「发全套」当目标态。

- **编排要的成功**  
  **系统侧按需下发**，且可回答验收题：**本轮上下文里到底挂了哪些技能、为什么**。  
  **反模式**：把公司库/可选表 **全套塞进会话**，再靠 **超长提示词** 规定「什么情况用这个、不要用哪个」——模型实际怎么吃进上下文**不可控**，提示词越多越难调试。
