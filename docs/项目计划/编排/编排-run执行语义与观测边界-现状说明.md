# 编排：run 执行语义与观测边界（现状说明）

**用途：** 把「**回形针编排平面**」里 **`heartbeat_run`（界面称 run）** 的抽象含义、与事务/进程/模型能力的关系、以及**看板观测**时易混的判断分一下层，避免把**展示**与**编排契约**绑死在一起。  
**真值：** 产品契约仍以 **`doc/`**、数据库契约与 **`长期需求/`** 正文为准；本文是**当下实现与协作用语**的归纳，不替代 SPEC。

---

## 1. 编排平面上的「run」是什么

- **run** 是控制面的一条**执行记录**：从进入可调度态到写入**终态**（成功、失败、取消、超时等）为止的生命周期锚点。
- 它回答的是「**这一次由 Paperclip 记账、可取消、可关联日志与费用字段的执行**」到了哪一步，**不是**「某模型有多少 token 窗口」的计量单位。

换言之：**run = 编排记账与状态机里的「一次执行」**；不要把「一次执行」默认等同为「只能干一件事」或「只能聊一轮」。

---

## 2. 与「多轮」、与「上下文窗口」的关系（分层）

| 层级 | 含义 |
| --- | --- |
| **编排 / run** | 一条执行生命周期；其转写可含多段 assistant / 工具 / 系统输出等。 |
| **对话轮次** | 适配器与模型交互的自然轮次，多数落在**同一条 run** 的持续输出里。 |
| **模型上下文窗口** | 供应商与模型族的能力上限；**不是** Paperclip 给 run 划的硬边界。 |

因此：**「一条 run」与「多轮交互」是兼容的**；**「一条 run」与「塞满 1M 窗口」之间没有一一对应关系**。

---

## 3. 「触及的事务」与 run：集合关系，不是流水线队列

运行详情中「**触及的事务**」应理解为：**与本 run 在数据上有关联的事务集合**（例如活动日志按 `runId` 关联到的 issue、以及运行快照里上下文指向的事务等）。

- **多条事务同时出现在列表里**，表示「这条 run 在链路上**碰过**这些事务」，**不**等价于「必须先收尾事务 A，才允许在界面逻辑上涉及事务 B」的严格队列语义。
- **某条事务在板上已是「完成」、另一条仍在进行**，与「这条 run 是否应立即被销毁」**不能**从该列表**直接**推导。

编排是否结束一条 run，以 **run 的终态与唤醒/续跑策略**为准，而不是以「单事务 close」为自动析构条件——那是另一个产品决策面。

---

## 4. 控制面终态 vs 适配器进程

- **run 进入终态** = Paperclip 侧认为**本次执行单元**已结算；与**适配器 OS 进程**是否仍存在，不是同一个概念（存在续跑、丢失检测、再次唤醒等分支）。
- 若观察到「进程已异常」与「界面仍像进行中」并存，应先区分：是**状态机/存活探测**类问题，还是**观测层信息滞后**（见下节）。

---

## 5. 观测边界：界面「进行中」与 API 终态

看板是**观测层**：同一 `runId` 的请求可能来自**列表**与**详情**等不同入口，客户端还会有缓存与刷新节奏差异。

可能出现的**短时不一致类型**包括：**详情缓存仍停留在旧的「进行中」快照，而列表已反映服务端终态**。这类问题属于**观测新鲜度**，修的是「别让旧快照覆盖较新的列表结果」，**不改变**「一条 run 应服务几条事务、何时结算」的编排语义。

排障时建议：**以服务端对应该 `runId` 的 `status` / `finishedAt` / `updatedAt` 为仲裁**；界面若与之一致，再讨论进程与续跑。

## 7. 编排张力与一种可选拓扑（讨论稿）

本节不是既成 SPEC，而是把常见**产品直觉**与**当下实现可能允许的形态**对齐用语，便于以后开工单或写 **`长期需求/`** 时引用。

### 7.1 风险：上下文未收紧时的「第二条事务」

在**没有把「一次执行携带的上下文」收紧到可预期范围**之前，若同一条 run 在事实行为上连续承载**多条事务**相关的工作（评论、checkout、工具侧状态等），出现**第二条事务更容易失败或跑偏**是可预期的工程风险：包括窗口与提示词挤占、会话续用带来的错误焦点、以及活动日志上「触及多条」与执行者心智模型不一致等。

这类问题应在 **上下文策略 + 唤醒/续跑边界** 上治理；单靠 UI 或单靠「转写更长」解决不了。

### 7.2 可选原则：「一事务 → 一条 run → 正常终态」

若坚持 **执行面与事务（issue）在编排上分离**，一种**清晰、易记账、易沟通**的拓扑是：**以单条事务为一次执行会话的边界**——事务收敛到完成（或明确失败退出）时，**对应 run 正常结束**；下一条事务由**新的唤醒/新的 run** 承接。

可以将它理解为一种**常见编排模式**（并非断言当前默认即如此；实现是否收紧到「一单一线」要以 **`doc/` / 需求说明 / 执行单** 拍板为准）。

若日后采纳，需要在产品层明确：**何谓「事务完成」触发终态**、与 **continuation / 续跑 / 同会话多轮** 的优先级关系，避免与适配器「长会话」能力打架。

---

**归档意图：** 当讨论「为什么一条 run 会挂多条事务」「run 结束了为什么还能看很多轮」时，优先用**本文前半（§1–§6）分层语言**对齐语义，再下钻到具体工单或 API 取证。若讨论**应否一单一线**或**多事务同 run 的风险**，引用 **§7** 并另立需求/执行单，避免口头「应该」与实现真值长期漂移。
