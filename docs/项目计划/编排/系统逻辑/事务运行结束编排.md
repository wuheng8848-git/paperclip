# 事务运行结束编排（薄总览）

**状态：** 产品 + 技术定稿（讨论稿，2026-05-19）  
**厚文档：** [`事务运行结束编排/关单后运行结案对账.md`](事务运行结束编排/关单后运行结案对账.md)

---

## 1. 这一编排域解决什么问题

回答三件事：

1. **一张活单（issue）什么时候算在编排上「干完了」？**
2. **对应那一趟 run 什么时候算「记账结束」？**
3. **若 CLI/适配器还没退出，控制面该不该等、该不该杀、run 标什么态？**

读者不必先打开 `server/src/services/heartbeat.ts`。

---

## 2. 产品原则（定稿）

| # | 原则 | 白话 |
| --- | --- | --- |
| P1 | **单事务单 run（主路径）** | 一趟 run 的快照里只有一个 `issueId`；经办关单后，应由 **适配器自己退出**，控制面 `executeRun` **一直等到** `adapter.execute()` 返回。 |
| P2 | **关单即 run 应收尾** | 事务进入 `done` / `cancelled` 后，这趟 run 在控制面上应视为「该干的活已结案」；**不是**长期挂着「进行中」等别的机制来擦屁股。 |
| P3 | **优雅退出优先** | 标 `done` **不**立刻 cancel run；给 CLI 收尾（含适配器 `terminalResultCleanup` 宽限）。 |
| P4 | **对账是兜底，不是主刀** | 「关单后运行结案对账」只在 **进程已死 / 不在内存执行表** 等条件下收库；**活着且仍在 executeRun 里则跳过**。 |
| P5 | **结案不是取消** | 兜底收 run 时标 **已完成（succeeded）**，不用 `cancelled` +「为对账已取消」文案。 |
| P6 | **不配合退出是产品设计** | 标了 done 但 CLI 长期不退：属执行层问题；超长兜底可杀 **这一趟 run 登记的进程树**，仍按 **已完成** 记账，**不算**运营异常。 |
| P7 | **并发 run ≠ 抢同一事务** | `maxConcurrentRuns` 限制的是同一智能体同时几趟 **running**；执行锁 `executionRunId` 防止两趟 run 同时占一张活单。实践上 Routic 单智能体常用 **1**。 |
| P8 | **不画蛇添足** | 不做「单 run 多事务」专项检测；主路径 + 对账兜底已覆盖。 |

## 3. 生命周期（主路径）

```text
唤醒（带 issueId）
  → 排队 run（queued）
  → claim → running，打上 executionRunId（执行锁）
  → 适配器 spawn CLI（登记 processPid / processGroupId）
  → 经办 PATCH 事务 done（或 cancelled）
  → CLI 输出终态 result → 适配器 execute 返回
  → executeRun：run → succeeded，releaseIssueExecutionAndPromote，封日志
```

**分支（仅 cancelled 事务）：** 改状态为 `cancelled` 时，issue 路由可 **立刻** `cancelRun`（用户明确要停）。

## 5. 代码地图（只看目录级）

| 模块 | 路径 |
| --- | --- |
| 心跳主循环、executeRun、对账 | `server/src/services/heartbeat.ts` |
| 不变式与取消原因常量 | `server/src/services/orchestration-invariants.ts` |
| 改事务状态、cancelled 时 cancel | `server/src/routes/issues.ts` |
| 本地子进程 spawn / 进程组 | `packages/adapter-utils/src/server-utils.ts` |
| 按 run 杀进程 | `heartbeat.ts` → `terminateHeartbeatRunProcess` → `local-service-supervisor.ts` |

调度顺序（`server/src/index.ts` 心跳 tick）：**关单后结案对账** 宜在 tick 前段；**孤儿回收** 若保留，不得在对账前把「应等待优雅退出」的 run 标成 failed/cancelled。
