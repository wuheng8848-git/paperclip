## 2. 现象（操作者原话摘要）

- **CEO 卡住与否**，在没有其它入口时 **完全不可知**。  
- **活动页 / 活动流** 对人类 **没有可用的连续观察面**（粒度、因果、当前 run 状态难对齐），无法替代「我在盯一条前台 run」的心智模型。

> 本条 **不断言** 技术上「没有任何日志」——而是强调：**默认看板路径** 对 **董事会式人类** 不构成 **可消费的运行镜像**。

## 5. 验证点四：ROU-59 —— 显式 recovery 工单、系统通知与「CEO 为何先看源单」

**人类困惑：** recovery 单 **拍给 CEO**，为什么唤醒后 **先去盯 ROU-59（源单）** 而不是 **先打开 recovery 本体**？

### 5.1 系统通知原文（照录）

（英文 UI，照录）

> Recover stalled issue **ROU-59**  
> Paperclip exhausted automatic recovery for an assigned issue and created this explicit recovery task.
>
> **Source**  
> Source issue: **ROU-59**  
> Previous source status: **in_progress**  
> Latest retry run: **fa219c15-a794-48a2-a52e-79915ee7d7ff**  
> Latest retry status: **cancelled**  
> Detected invariant: **stranded_assigned_issue**  
> Retry reason: **issue_continuation_needed**  
> Failure: Latest retry failure details were withheld from the issue thread; inspect the linked run for evidence.
>
> **Ownership**  
> Selected owner: the first invokable manager/creator/executive candidate with budget available.
>
> **Required Action**  
> Inspect the latest run and source issue state.  
> Fix the runtime/adapter problem, reassign the source issue, or convert the source issue into a clear manual-review state.  
> When the source issue has a live execution path or has been intentionally resolved, mark this recovery issue done.

### 5.2 控制面事实（为何不要求「必须先打开 recovery 工单」）

1. **`tickTimers` → `enqueueWakeup(..., source: "timer")`** 时，`contextSnapshot` **通常不带 `issueId`**（仅有 scheduler 元数据，如 `interval_elapsed`）。见 **`server/src/services/heartbeat.ts`** 中 `tickTimers` 与 `executeRun` 开头：`issueId` 从快照读出，**为空则 `issueContext` 为空**，适配器侧 **没有「唯一当前工单」**。  
2. Timer 闸门 **`agentHasRunnableAssignedIssue`**：只要经办名下存在 **非 `backlog` / `done` / `cancelled`** 的工单即可唤醒 —— **`blocked` 仍算**（HB-007 语义）。故 CEO 名下 **同时挂源单 + recovery 单** 时，timer **照样触发**，但 **不在服务端替你选好先看哪张**。  
3. **`deriveTaskKeyWithHeartbeatFallback`**：timer 醒会用 **`HEARTBEAT_TASK_KEY`** 做会话续跑键 —— **Still not** 绑定具体 `issueId`。  
4. **通知正文 Required Action** 第一句就是 **Inspect the latest run and source issue state** —— 产品与文案 **明确包含「先看源单状态」**，与 CEO **先打开 ROU-59** **不矛盾**。  
5. **启发式**：源单 **阻塞 + 高优先** 时，模型优先救火 **症状现场（ROU-59）** 而非 **元工单（recovery）**，在 **未强制排序** 的前提下是常见行为。

**推论（弱）：** 「派给 CEO 的是 recovery」是 **所有权/收口责任**；**心跳第一句去哪张 issue** 在 **timer 路径上更像模型 + 工具自选**，**不等于** 违背派单，除非产品明确要求 **recovery 永远先于 source**（当前代码未见该硬规则）。

### 5.3 待钉

- **`GET /api/issues/ROU-59`** 与 **对应 `stranded_issue_recovery` 工单**（slug 由实例为准）：二者 `assignee`、`status`、`blockedBy` 是否在 UI 上与人类预期一致。  
- **`fa219c15-…`**：**cancel** 根因（适配器 / 人类 / 预算 / 依赖）。  
- **体验欠账汇总：** **[`022-探查-ROU-59体验问题清单-格式计划任务与通知汉化.md`](022-探查-ROU-59体验问题清单-格式计划任务与通知汉化.md)**（Qwen 格式、计划任务 UI、系统通知汉化）。

## 7. 治理张力（董事户口径）：手动 pause 智能体 vs 系统自动造阻塞 / recovery

**已定产品口径（硬）：** **人类手动 pause ≠ 系统故障**，**待人复机即可**；**默认不得**仅凭 pause 拉长 stranded/recovery/阻塞链 —— 全文见 **[`01.项目需求说明.md`](../01.项目需求说明.md)**「**产品口径（已定） → 手动 pause 智能体**」。  
本节下文保留 **案发复盘、体感与实现缺口**，供对齐代码 / 技能 / 人设时查阅。

### 7.1 操作权衡（为何「先停人」）

- **动机之一**：在特定局面下，故意 **不让自动化 recovery 在错误前提下层层加码**（口语所谓「别让死人修死人」）；暂停 **执行者** 是最直接的止血。  
- **推广**：即便不是上述动机，也会有 **别的理由** 要把推进停下来。  
- **路径对比**：理想化顺序若是「停活动 → 停工单 → 停智能体」，在现实里 **活动与工单都可能是多个**，收口成本高；**直接 pause 智能体往往最快**。  

### 7.2 张力陈述（权利 / 合法性追问）

当 **董事会已经手动 pause 执行工程师**（本案见 §4.6：`manual pause`）时，系统侧仍可能：

- 认定 **stranded / continuation** 不变量；  
- **生成 recovery 工单**、写 **阻塞关系**、推 **系统通知**；  
- 迫使董事会再在 **另一套叙事**（recovery / blocked）里接盘。

**董事会追问（产品层）：** 在 **人类已明示停机意志**（pause agent）的前提下，**控制面自动创建阻塞类后果单** 的 **正当性边界** 在哪里？  
换言之：**pause 应不应该构成「暂停 stranded 救济 / 延迟造 recovery」的门控？** 当前实现若 **不构成**，就会出现「人类最快止血 → 系统反过来加长工单链」的 **体感荒诞** —— 这与技术对错无关，是 **治理语义** 是否对齐的问题。

### 7.3 与本案证据的锚点

- §4.6 已记录：**ROU-59** 卡住与 **开发工程师 manual pause**、**CTO 亦暂停** 同日并存；其后 **ROU-64** 等 recovery 叙事展开 —— **可用作讨论默认策略时的实例锚点**，**不等于** 全局结论。

### 7.4 待产品回答（仅列问题）

1. **`manual pause`** 是否应对 **`stranded_assigned_issue` / recovery issue 创建** 产生 **抑制、延迟或需董事会二次确认**？  
2. **多个工单 / 多条活动** 场景下，如何把 **「停智能体」** 映射为 **人类可理解的单一停机语义**，避免系统侧 **并行再发明一套阻塞图**？  

### 7.5 董事会体感：「假问题」一路阻塞 + CEO 首位复苏时的扫街成本

**误报与放大：** 自动化 stranded/recovery 若在 **人类认定并非故障** 的前提下仍启动（或因 pause 等治理动作被误读为「待救」），就会把 **本非问题** 的事 **制度化问题**，再通过 **blocked / recovery / 通知** **一路加长链条** —— 对人类而言等于 **愚蠢的摩擦**，与是否最终实现收口无关。

**CEO 亦被 pause、且首位恢复的是 CEO 时：**  
经办队列里往往 **多条阻塞类工单**；timer 路径 **不在快照绑定 `issueId`**（§5.2），模型需 **自行扫盘**。结果是：**未必能按因果对齐「线头第一条」**，更像 **沿优先级/启发式乱序救火**，与董事会心里的 **单一根因叙事** 脱节。

**待收敛（仍属 §7.4 同类）：** 是否需要 **「暂停治理窗口」**（全员 pause 或董事会停机）下 **抑制 recovery 噪声**，以及复苏后 **强制暴露「推荐第一条」**（根因单 / recovery 源单）—— **产品未定**，此处只记张力。

### 7.6 落地路径（对齐已定口径）

| 层级 | 作用 | 局限 |
| --- | --- | --- |
| **改代码（服务端 / heartbeat / recovery）** | **`manual pause` 参与门禁**：pause 期间 **抑制或延迟** stranded/recovery 创建；复机后再允许自动化 | **唯一能「定死」行为**；须防误伤「pause 以外真故障」 |
| **改技能** | 约束助手 **不把 pause 当故障**、不乱触发补救剧本 | 挡不住控制面自动路径 |
| **改 CEO 人设** | 复述董事会停机语义、减少无效救火叙事 | **不能代替硬闸** |
