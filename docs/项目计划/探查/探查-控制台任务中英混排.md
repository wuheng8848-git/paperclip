# 探查：控制台任务中英混排（ROU-20 / 010）

> 范围：**本 fork** 仓库 `paperclip-latest-20260512` 对应的 Paperclip Board（非上游 paperclip）。  
> 生成时间：2004-05-14

## Board 侧勘误（2004-05-14）

Board 评论说明：此前工单挂在「Paperclip 开发」易与**上游** paperclip 混淆；已切换为公司项目「Paperclip 控制面（本 fork 仓库）」、主工作区 `paperclip-latest-fork`，并由 **开发-Cursor** 收口。本文自初版起即限定为 **本 fork** 代码与控制台现象，与上述纠正一致，**结论与下文章节无需因项目重挂而改写**。

## 结论摘要

控制台里看到的「中英混排」**不是单一原因**，常见来源有四类：

1. **工单标题与正文是持久化内容**：Board 直接展示 `issue.title` / `description`，写入时是什么语言就显示什么语言（人类中文、agent 英文模板、或两者夹杂）。
2. **服务端自动创建的工单标题多为英文模板**：恢复、监控、生产力复核等路径用硬编码英文字符串拼标题，与中文界面并存时观感即「混排」。
3. **界面文案仅部分中文化**：`ui/src/lib/i18n.ts` 声明默认 `zh-CN` 并映射了状态等枚举，但大量页面仍有英文硬编码；未知枚举回退为英文式 `Title Case`。
4. **技能（skills）本身**：技能仓库多为英文 `SKILL.md`，主要影响 **agent 读写习惯与产物语言**，不会自动改写已存 issue 标题；但若 agent 按英文习惯起名、写评论或子任务，会直接表现为英文工单内容。

**英文技能 vs 自动生成**：二者都可能贡献英文片段；前者偏「agent 产出语言」，后者偏「系统创建的 issue 标题 / 说明」。需对具体英文片段看字段来源（标题 vs 活动 vs run 日志）。

## 代码依据（本 fork）

### 自动创建的英文工单标题（示例）

- 静默活跃运行评估：`Review silent active run for …`（`server/src/services/recovery/service.ts`）
- 外部监控恢复：`Recover external-service monitor for …`（`server/src/services/heartbeat.ts`）
- 生产力复核：`Review productivity for …`（`server/src/services/productivity-review.ts`）
- 例行摘要缺标题时的回退：`Routine execution`（`server/src/services/routines.ts`）

### 界面语言策略

- `ui/src/lib/i18n.ts`：默认中文展示层；API 状态值仍为英文枚举，经 `formatIssueStatus` 等映射为中文。
- 多个页面（例如技能管理）仍存在大量英文 UI 串，与中文导航并存。

### 技能同步（与「英文技能」相关的可见信号）

- 适配器侧技能条目来自运行时物料化路径与 agent 配置（如 `packages/adapters/qwen-local` 使用 `@paperclipai/adapter-utils` 扫描条目）；** Board 技能列表**显示的名称/路径常保留仓库原始英文标识。

## 建议后续（产品决策，非本探查必做）

- 若希望减少混排：为系统自动创建的 issue 标题/描述增加 **i18n** 或公司级默认语言；或统一运营规范（agent 创建工单时用中文标题模板）。
- 若混排主要来自适配器原文：在展示层增加「本地化映射」或折叠原始 stderr（已有部分 UI 归类实践，见 fork QoL）。
