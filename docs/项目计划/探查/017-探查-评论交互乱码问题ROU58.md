# 探查报告：ROU-58 评论/交互乱码问题

> 日期：2026-05-16
> 性质：只读探查，不修改代码、数据库、issue 状态
> 关联文档：`探查-评论乱码问题真因.md`（已有报告侧重 stdout 编码路径）

---

## 1. 问题现象

ROU-58 的 `GET /api/heartbeat-runs/c63804b9-99d7-4da8-bcb8-80bcb7305cc2` 返回数据中：

- ✅ `resultJson.result` 和 `summary` 是正常中文
- ❌ agent 创建的评论（id `479f333f-eaf9-4548-bf49-78499aa3594a`、`0434ebbf-d146-483b-be7f-5d9878f4ac2f`）和 `ask_user_questions` interaction payload 已乱码

**关键区分**：这不是"agent stdout 被 Paperclip 服务端读坏"，而是"agent 在运行中自行用 curl 写回 API 时，中文 JSON 在命令行链路中损坏"。

---

## 2. 两条数据路径的对比

| 路径 | 数据流 | 结果 | 原因 |
|------|--------|------|------|
| **A. stdout → resultJson** | Qwen `-o stream-json` → Paperclip `spawn` 子进程 → `parseQwenStreamJsonOutput()` → 写入 DB | ✅ 正常 | Qwen 输出 UTF-8 JSON，Node.js spawn 读取 UTF-8 Buffer，`JSON.parse()` 成功 |
| **B. agent curl → 评论/interaction** | agent 运行中执行 `curl -d '{"comment":"中文"}'` → Windows 控制台/管道 → Paperclip API → DB | ❌ 乱码 | shell 内联 JSON 经过 Windows 代码页（CP936/GBK）→ curl → HTTP → 服务端接收时已损坏 |

**结论**：ROU-58 乱码的真因是路径 B，不是路径 A。已有探查文档侧重路径 A（stdout 编码），但 ROU-58 证据指向路径 B。

---

## 3. Agent 技能/Adapter 中涉及写评论的位置

### 3.1 paperclip 技能 `SKILL.md`

路径：`~/.qwen/skills/paperclip/SKILL.md`

**正面约束**（文档中已写）：
> 多行 Markdown 评论 **禁止**手写一行 JSON 挤在一起：用仓库 `scripts/paperclip-issue-update.sh` 或等价 `jq --arg`。

**反面示例**（文档中大量存在）：
所有 API 调用示例均使用 `curl -d '{"key":"值"}'` 内联 JSON 写法，包括：
- `POST /api/issues/{issueId}/interactions` 示例
- `PATCH /api/issues/{id}` 示例（含 `comment` 字段）
- `POST /api/companies/{companyId}/approvals` 示例

**矛盾**：SKILL.md 告诉 agent "不要手挤单行 JSON"，但给出的示例全是内联 `-d` 写法，agent 会照着示例做。

### 3.2 默认 prompt 模板

路径：`packages/adapter-utils/src/server-utils.ts` §149

`DEFAULT_PAPERCLIP_AGENT_PROMPT_TEMPLATE` 内容：
- ✅ 提到 `"Leave durable progress in comments..."`
- ✅ 提到 `"update the issue to a clear final disposition"`
- ❌ **没有教 agent 如何安全地发送含中文的 HTTP 请求**
- ❌ 没有提到编码注意事项

### 3.3 Qwen adapter API 访问注

路径：`packages/adapters/qwen-local/src/server/execute.ts` — `renderPaperclipApiAccessNote()`

已提供 **PowerShell 安全示例**：
```powershell
Invoke-RestMethod -Method Patch -Uri "$env:PAPERCLIP_API_URL/api/issues/$env:PAPERCLIP_TASK_ID" `
  -Headers @{ ... } -ContentType "application/json" `
  -Body (@{ status = "done"; comment = "Completed in this run." } | ConvertTo-Json)
```

✅ 这条路径是安全的（`ConvertTo-Json` 输出 UTF-8）。但 agent **可能不照做**——SKILL.md 里的 curl 示例更醒目、数量更多。

### 3.4 `scripts/paperclip-issue-update.sh`

SKILL.md 提到此脚本作为推荐写法，但**当前项目快照中不存在此文件**。即使存在，在 Windows 环境下也需要 Git Bash 执行，增加了链路复杂度。

## 4. 服务端编码处理代码

路径：`packages/adapter-utils/src/server-utils.ts` §2107-2114

```typescript
const decodeChunk = (chunk: unknown): string => {
  if (typeof chunk === "string") return chunk;
  if (Buffer.isBuffer(chunk)) {
    return chunk.toString("utf8");
  }
  return String(chunk);
};
```

这段代码负责读取子进程 stdout/stderr。注释中提到：
> Try UTF-8 first; if the bytes don't round-trip, fall back to UTF-8 with replacement characters — this is a best-effort guard against mojibake until a proper iconv-lite integration is added.

**实际行为**：只做 UTF-8 转换，没有 GBK 回退。但对 ROU-58 而言，这条路径不是问题所在（Qwen stream-json 本身是 UTF-8）。

---

## 5. 根因总结

| 维度 | 判断 |
|------|------|
| **直接原因** | agent 在 heartbeat 运行中用 `curl -d '{...中文...}'` 内联 JSON 调用 Paperclip API，Windows 环境下编码损坏 |
| **根本原因** | paperclip SKILL.md 的 API 示例全是内联 `-d` 写法，agent 照着做；且没有明确警告"禁止 shell 内联中文 JSON" |
| **为什么 stdout 路径正常** | Qwen `-o stream-json` 输出纯 UTF-8 JSON，Node.js spawn 直接读 Buffer 后 `JSON.parse()`，不经过控制台 |
| **为什么其他评论正常** | 用户手动输入通过 HTTP JSON body（UTF-8）；服务端代码直接生成的评论也是 UTF-8 |

---

## 6. 止血方案

### 方案 A（推荐首选）：修正 paperclip SKILL.md 的 curl 示例

**改动范围**：`~/.qwen/skills/paperclip/SKILL.md` + `references/` 下所有含 curl 示例的文件

**改动内容**：

1. 在 SKILL.md "鉴权与环境变量"段落后追加一条**编码规则**：
   ```
   **中文编码规则：** 当请求体包含中文时，禁止使用 `curl -d '{"comment":"中文"}'` 内联写法。必须：
   - 方式 1（PowerShell）：`$body = @{ comment = "中文" } | ConvertTo-Json; Invoke-RestMethod ... -Body $body`
   - 方式 2（文件）：先生成 UTF-8 JSON 文件，再用 `curl --data-binary @file`
   ```

2. 将所有现有 `curl -d '{...中文...}'` 示例替换为上述安全写法。

**风险**：低。只改技能文档，不改代码。下一轮 heartbeat agent 加载技能即生效。

### 方案 B：在默认 prompt 模板中增加编码指令

在 `DEFAULT_PAPERCLIP_AGENT_PROMPT_TEMPLATE` 末尾追加：
```
- When calling Paperclip API with Chinese text, NEVER use `curl -d` with inline JSON. Use PowerShell `Invoke-RestMethod -Body (@{...} | ConvertTo-Json)` or write JSON to a UTF-8 file first and use `curl --data-binary @file`.
```

**风险**：低。影响所有 agent，但指令只约束中文场景。

### 方案 C（兜底）：服务端引入 iconv-lite GBK 回退

在 `decodeChunk` 中加入 GBK 检测回退逻辑。

**风险**：中。需要引入新依赖，影响所有 adapter 的 stdout 读取路径，需测试验证。

---

## 7. 推荐执行顺序

1. **立即**：方案 A（改 SKILL.md curl 示例）+ 方案 B（prompt 模板追加指令）
2. **验证**：触发一次 Qwen agent heartbeat，让其写含中文的评论，检查 UI 是否正常显示
3. **长期**：方案 C（服务端编码防御，作为兜底）；创建 `scripts/paperclip-issue-update.ps1`（PowerShell 版安全脚本）
