# 探查：回形针任务运作机制（ROU-49 时间线）

## 结论摘要

ROU-49 不是执行失败，而是 **任务执行成功后未被 Paperclip 识别为正式收口**。

本次链路里，Qwen 一共跑了 4 次 heartbeat run：

1. 第一次由人类指派触发。
2. 第二、三次由 Paperclip 的「运行活性接续」自动触发。
3. 第四次由 Paperclip 的「成功运行收口纠偏」自动触发。
4. 第四次里 Qwen 在评论中写了 `最终处置: done`，但没有完成 Paperclip 可识别的正式状态变更。
5. Paperclip 因此升级为 recovery，把 ROU-49 置为 `blocked`，并创建 ROU-53。

关键判断：**后续不是人类催促 Qwen 继续做，而是 Paperclip 的自动接续与收口纠偏机制在催促。**

## 关键英文口径翻译

| 原字段 / 原文 | 中文口径 | 含义 |
|---|---|---|
| `assignment` | 分配触发 | 人类或系统把 issue 分配给 agent 后，触发一次唤醒。 |
| `issue_assigned` | 事务已分配 | 分配动作对应的唤醒原因。 |
| `automation` | 自动化触发 | 不是人类点击，而是 Paperclip 后台机制触发。 |
| `run_liveness_continuation` | 运行活性接续 | 上一次 run 成功结束，但系统判断任务还没形成足够的完成证据，于是继续唤醒。 |
| `plan_only` | 仅计划态 | 系统判断 run 更像是在描述下一步计划，而不是已经完成可验收动作。 |
| `Run described runnable future work without concrete action evidence` | 运行描述了可继续执行的未来工作，但缺少具体行动证据 | `plan_only` 的解释文本。 |
| `modelProfile: cheap` | 使用便宜模型档 | 自动接续时使用了 cheap profile。此处容易受模型配置错误影响。 |
| `continuationAttempt` | 接续次数 | 第几次自动接续。ROU-49 中出现 1/2 与 2/2。 |
| `finish_successful_run_handoff` | 成功运行后的收口交接 | run 成功，但 issue 仍未正式处置，系统要求 agent 补上 `done` / `blocked` / `in_review` 等正式收口。 |
| `successful_run_missing_state` | 成功运行缺少状态收口 | 系统认为 run 成功，但 issue 状态仍没有有效终局或下一步。 |
| `clear_next_step` | 缺少明确下一步 | Paperclip 无法判断该事务该结束、待审、阻塞、委派，还是继续。 |
| `successful_run_handoff_required` | 需要成功运行收口交接 | 活动日志事件，表示系统启动 corrective handoff。 |
| `successful_run_handoff_escalated` | 成功运行收口交接升级 | corrective handoff 仍未解决，系统升级为 recovery / blocker。 |
| `Recover missing next step` | 恢复缺失下一步 | 系统自动创建的 recovery issue，用来要求人或 agent 补齐收口。 |

## 每次 Qwen 完成后的系统动作

这里区分两件事：

- **Qwen 自己完成了什么**：run 的 `result_json.result` 与 issue comment。
- **Paperclip 之后怎么判定**：是否生成下一次 wakeup、是否升级 recovery、是否改变 issue 状态。

### Run 1 结束后：系统判定为「仍需接续」

```text
run: 124fd8b4...
结束时间：2026-05-16 04:19:47
run event: run succeeded
exitCode: 0
```

Qwen 的完成动作：

- 产出《ROU-49 实现完成总结》。
- 声称已完成优先级字段、数据库迁移、优先级计算引擎等实现。
- 在 issue 评论中留下长总结。

Paperclip 的后续动作：

```text
下一次 wakeup requested_at: 2026-05-16 04:19:47
source: automation
reason: run_liveness_continuation
livenessState: plan_only
livenessReason: Run described runnable future work without concrete action evidence
continuationAttempt: 1
maxContinuationAttempts: 2
modelProfile: cheap
```

中文解释：Paperclip 认为 Qwen 虽然报告完成了一部分，但文本中仍出现「接下来要更新 API」这类可继续执行信号；系统判定它更像“还有活要干”，于是自动排队 Run 2。

### Run 2 结束后：系统再次判定为「仍需接续」

```text
run: fa6a7d64...
结束时间：2026-05-16 06:05:17
run event: run succeeded
exitCode: 0
```

Qwen 的完成动作：

- 再次产出《ROU-49 完成总结》。
- 声称优先级引擎、数据库层、API、前端组件已完成。
- 提到修复了 blocker 推断误判。
- 在 issue 评论中记录阶段性总结。

Paperclip 的后续动作：

```text
下一次 wakeup requested_at: 2026-05-16 06:05:18
source: automation
reason: run_liveness_continuation
livenessState: plan_only
livenessReason: Run described runnable future work without concrete action evidence
continuationAttempt: 2
maxContinuationAttempts: 2
modelProfile: cheap
```

中文解释：Paperclip 仍然没有把任务视为已正式收敛，而是使用第二次，也是最后一次自动接续额度，排队 Run 3。

### Run 3 结束后：系统不再接续实现，改为要求正式收口

```text
run: cad1cde7...
结束时间：2026-05-16 06:28:50
run event: run succeeded
exitCode: 0
```

Qwen 的完成动作：

- 声称 ROU-49 实现完整且正常工作。
- 总结已有产物：优先级引擎、SQLite 迁移、API 端点、前端组件、排序规则验证。
- 评论中写了“状态：已完成”。

Paperclip 的后续动作：

```text
activity: issue.successful_run_handoff_required
sourceRunId: cad1cde7...
correctiveRunId: 607adc55...
handoffReason: successful_run_missing_state
missingDisposition: clear_next_step
```

系统评论：

```text
Paperclip needs a disposition before this issue can continue.
```

中文解释：这时 Paperclip 不再继续普通实现接续，而是发现一个状态机问题：run 成功了，但 ROU-49 仍是 `in_progress`，没有被正式标为 `done`、`in_review`、`blocked`、`cancelled`，也没有有效委派或继续路径。因此它启动 Run 4，专门要求 Qwen 补正式处置。

### Run 4 结束后：Qwen 写了 done，但系统仍未识别为正式处置

```text
run: 607adc55...
结束时间：2026-05-16 06:29:33
run event: run succeeded
exitCode: 0
reason: finish_successful_run_handoff
```

Qwen 的完成动作：

- 读取上次运行总结。
- 在评论中列出已完成项。
- 写出：

```text
最终处置: `done` — 所有验收标准已满足，无遗留工作。
```

Paperclip 的后续动作：

```text
activity: issue.successful_run_handoff_escalated
source: recovery.reconcile_successful_run_handoff_missing_state
previousStatus: in_progress
new status: blocked
blockerIssueIds: [ROU-53 的 UUID]
```

系统评论：

```text
Paperclip could not resolve this issue's missing disposition automatically.
The issue is blocked on a recovery owner.
```

中文解释：Qwen 在评论里写了 `done`，但没有触发 Paperclip 的正式状态变更动作。系统仍然认为缺少 disposition，于是创建 ROU-53，并让 ROU-53 反向阻塞 ROU-49。

## 关键机制拆解：完成后为什么还会继续

ROU-49 里出现了三种「完成后继续」：

| 阶段 | Qwen 视角 | Paperclip 视角 | 后果 |
|---|---|---|---|
| Run 1 后 | 我做了核心实现并总结 | 文本仍有未来可执行动作，像 `plan_only` | 自动接续 Run 2 |
| Run 2 后 | 我继续补完前端/API/验证 | 仍判定为 `plan_only`，还有接续额度 | 自动接续 Run 3 |
| Run 3 后 | 我确认已完成 | issue 仍无正式 disposition | 启动收口纠偏 Run 4 |
| Run 4 后 | 我在评论里写了 `done` | 评论不是正式状态动作 | 升级 recovery，创建 ROU-53 |

## 四次 run 的触发还原

### Run 1：人类指派触发

```text
source: assignment
reason: issue_assigned
requested_by_actor_type: user
requested_by_actor_id: local-board
```

中文解释：人类在 Board 上把 ROU-49 指派给 Qwen，Paperclip 因为“事务已分配”立即唤醒 Qwen。

### Run 2：第一次自动接续

```text
source: automation
reason: run_liveness_continuation
livenessState: plan_only
livenessReason: Run described runnable future work without concrete action evidence
continuationAttempt: 1
maxContinuationAttempts: 2
modelProfile: cheap
```

中文解释：Paperclip 认为 Run 1 虽然成功结束，但留下了“还要继续做”的信号，且缺少足够具体的完成证据，于是自动接续一次。

### Run 3：第二次自动接续

```text
source: automation
reason: run_liveness_continuation
livenessState: plan_only
continuationAttempt: 2
maxContinuationAttempts: 2
modelProfile: cheap
```

中文解释：Paperclip 仍然认为任务没有收敛，于是使用最后一次自动接续额度继续唤醒 Qwen。

### Run 4：成功运行后的收口纠偏

```text
source: automation
reason: finish_successful_run_handoff
instruction: Your previous run on ROU-49 succeeded, but the issue is still in `in_progress`...
```

中文解释：Paperclip 认为前一个 run 已成功，但 ROU-49 仍停留在 `in_progress`，所以强制发起一次“请补正式处置”的纠偏运行。

## 为什么评论里的 `done` 没有完成收口

Qwen 在最后一次评论中写了：

```text
最终处置: `done` — 所有验收标准已满足，无遗留工作。
```

但 Paperclip 的正式收口依赖可识别的 issue 状态变更或受控动作，而不是普通评论文本。评论可以作为证据，但不是状态机动作本身。

因此系统看到的是：

```text
run succeeded
issue status still in_progress
no valid disposition detected
```

于是进入 recovery：

```text
ROU-53 Recover missing next step ROU-49
ROU-53 blocks ROU-49
ROU-49 status = blocked
ROU-53 status = blocked
```

## 对当前系统的含义

### 1. Qwen CLI 已能长时间执行

ROU-49 连续 4 次 run 均为 `succeeded`。这说明 Qwen CLI / Qwen adapter 的基础执行链路已经恢复。

### 2. 自动接续会主动扩大执行时长

即使人类只做了一次指派，Paperclip 仍会根据 `plan_only` 判定自动接续。该机制能救回未完成任务，也可能在边界模糊时持续消耗模型额度。

### 3. cheap profile 是关键风险点

ROU-49 的自动接续使用了 `modelProfile: cheap`。如果 cheap profile 配错模型，自动接续会放大错误。本次 Qwen 401 问题就与错误 cheap 模型有关。

### 4. 收口协议需要被明确修复

当前缺口不是“agent 不会说 done”，而是“agent 说 done 后没有转成 Paperclip 的正式 disposition”。后续应修复以下至少一项：

- agent prompt 明确要求调用 Paperclip API / MCP 工具正式更新 issue 状态；
- Paperclip 对 corrective handoff 评论中的明确 `done` 做受控解析；
- recovery 不应默认派回同一个未能完成收口的执行 agent；
- UI 上应把 run liveness continuation、successful run handoff、recovery 的因果链显示成可读时间线。

## 判读速查

| 现象 | 更可能的解释 |
|---|---|
| `source=assignment` + `reason=issue_assigned` | 人类或系统分配事务后触发。 |
| `reason=run_liveness_continuation` | 自动接续，不是人类再次点击。 |
| `livenessState=plan_only` | 系统判断上次 run 像计划，缺少完成证据。 |
| `reason=finish_successful_run_handoff` | run 成功但 issue 没正式收口，系统要求补处置。 |
| agent 评论写了 `done`，issue 仍未 done | 评论不是正式 disposition，需检查是否调用状态变更动作。 |
| recovery issue 反过来 blocks 源 issue | 系统已放弃自动判断，把缺失下一步升级成人工/管理者收口问题。 |

## 建议收口动作

对 ROU-49 / ROU-53，建议明天人工处理：

1. 确认 ROU-49 产物是否真实可用。
2. 手动把 ROU-49 标记为 `done` 或明确改为其他正式状态。
3. 手动把 ROU-53 标记为 `done` 或 `cancelled`。
4. 另立 Paperclip 本体问题：成功运行后的评论处置未转为正式 disposition，且 recovery 默认派回同一 agent 会造成闭环失败。
