# 042 多源唤醒合并与归因 — 现状与落点（探查）

## 边界

- **本期实现**：`enqueueWakeup` 内**合并进已有 run**（同经办、同 scope 的 `queued` / `running` / `scheduled_retry`、以及「同链执行名+正在跑」的 coalesce）时，写入 **`contextSnapshot.effectiveTrigger`**；新唤醒入队前一律补 **`attachInitialEffectiveTriggerIfMissing`**，与行上 `invocationSource` 对齐。
- **未做**：`issues` 路由里 `addWakeup` 用 `Map(agentId:issueId)` **后者覆盖前者**的合并仍独立于心搏；若要与 A2「短时窗多源」完全对齐，需另单扩 issues 侧或统一入口。
- **未做**：活动日志逐条打出「谁吞谁」；auditor 现阶段靠 **`agent_wakeup_requests.status=coalesced`** + run 上 **`effectiveTrigger.absorbed`**。

## 合并规则（书面）

1. 优先级档位：`timer` &lt; 未知源 &lt; `automation` / `assignment` / `on_demand`。  
2. **高档盖住低档**；**同档**时 **incoming（后一次唤醒）** 为赢家，前一次记入 **`absorbed[]`**。  
3. **`effectiveTrigger.winningInvocation`** 表示当前「算谁叫醒的」；**`absorbed`** 为时间线可读的**被淘汰触发**（含 `absorbedAt`）。
