## 2. 评论唤起入口（HTTP / 控制面）

**主路径：** `server/src/routes/issues.ts` 在处理 **带 `commentBody` 的 issue 更新** 时，对经办人 `assigneeAgentId` 调用 `addWakeup`：

- `reason`：`issue_commented` 或 `issue_reopened_via_comment`（重开）。
- `contextSnapshot` 典型字段：`issueId`、`taskId`、`commentId`、`wakeCommentId`、`source`（`issue.comment` / `issue.comment.reopen`）、`wakeReason`、可选 `resumeIntent` / `interruptedRunId` 等。
- **未发现**与标签、角色模板、`executionPolicy`、事务类型挂钩的额外策略字段。

**并列路径（同属「评论相关」唤醒，但 reason 不同）：**

- `@mention` 非经办：`reason: issue_comment_mentioned`，`source: comment.mention`（同文件紧随其后循环）。
- 交互解析后续唤醒：`queueResolvedInteractionContinuationWakeup`，`reason: issue_commented`，`payload.mutation: interaction`（同文件靠前辅助函数）。

## 3. 队列与 `contextSnapshot` 归一

**`enqueueWakeup`**：`server/src/services/heartbeat.ts` 内 `async function enqueueWakeup`。

**`enrichWakeContextSnapshot`**（同文件）：合并 `wakeReason`、`issueId`、`taskKey`、**多条** `wakeCommentIds` / 最新 `wakeCommentId`，写入 `wakeSource`、`wakeTriggerDetail`；会删除需按 id 重建的 `paperclipWake` 缓存键，避免陈旧结构化载荷。**仍无档位字段。**

**延期评论唤醒**：同文件内将 `source: "deferred_comment_wake"` 等活动记入 **`deferred`** 晋升链；晋升时再次 `enrichWakeContextSnapshot`。**档位仍缺位。**

---

## 4. `paperclipWake` 与注入体量（与「轻量档」最接近的现有机制）

**`buildPaperclipWakePayload`**（`heartbeat.ts`）：按 `extractWakeCommentIds` 拉评论行，组装 `comments` 数组写入未来将序列化到适配器环境的载荷。

**硬上限（常量，非产品档位）：**

- `MAX_INLINE_WAKE_COMMENTS = 8`
- `MAX_INLINE_WAKE_COMMENT_BODY_CHARS = 4_000`
- `MAX_INLINE_WAKE_COMMENT_BODY_TOTAL_CHARS = 12_000`

超过则 `truncated` / `fallbackFetchNeeded`，**不会**按策略切换为「仅回执」或「禁 API」。

**Run 准备阶段**（`heartbeat.ts` 内为单次 run 构建 `context` 的大段逻辑）：在解析 workspace、model profile、secrets 之后：

- **`resolveRunScopedMentionedSkillKeys`**：从事务 **title、description、全线程评论** 中提取 `@skill` 提及，解析为 `skillKeys`，经 **`applyRunScopedMentionedSkillKeys`** 写入适配器 config 的 skill 同步偏好。
- **`companySkills.listRuntimeSkillEntries`**：公司级运行时 Skill **全量**进入 `runtimeConfig.paperclipRuntimeSkills`。

**含义：** 即便唤起原因是 `issue_commented`，当前仍走 **同一套** Skill 与 wake 载荷构建逻辑；**没有**「评论唤起 → 少灌 Skill / 少灌 thread」的开关。

## 5. `IssueExecutionPolicy` 与 041 需求差距

**类型定义：** `packages/shared/src/types/issue.ts` — `IssueExecutionPolicy`：`mode`、`commentRequired`、`stages[]`、`monitor?`。

**服务端：** `server/src/services/issue-execution-policy.ts` 负责阶段流转、monitor 元数据脱敏、与状态 patch 等。

**差距：** 共享类型与路由层 **均未**承载「评论唤起档位」或工具/API/改仓白名单；041 要求的「按 `executionPolicy` 选配」若要落地，需 **-schema 扩展 + 校验 + issues 路由/heartbeat 读取**，属 **破坏性契约增量**（需与 SPEC / 项目计划双向真值同步，见 `AGENTS.md` §5.9）。

## 6. 适配器侧

各 `packages/adapters/*-local/src/server/execute.ts`：从 `context` 读取 `wakeCommentId` / `commentId`，设置 `PAPERCLIP_WAKE_COMMENT_ID`；服务端还可传 `PAPERCLIP_WAKE_PAYLOAD_JSON`（Vitest 中有覆盖）。**无档位枚举或工具白名单由服务端下发**（工具能力仍主要由各 CLI/适配器与 config 决定）。
