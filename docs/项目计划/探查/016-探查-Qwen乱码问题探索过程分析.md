# Qwen 乱码问题探索过程分析

> 生成日期：2026-05-16
> 分析对象：Qwen 模型在探查评论乱码问题时的探索行为
> 原始对话：见 `探查-关于评论里面出现乱码的问题.md`

---

## 一、探索统计概览

| 指标 | 数值 |
|------|------|
| **总轮次** | 约 40+ 轮对话（含思考 + 工具调用） |
| **Grep 搜索次数** | 约 25+ 次 |
| **ReadFile 读取次数** | 约 15+ 次 |
| **思考/推理次数** | 约 20+ 次（每轮都有思维链） |
| **错误结论次数** | 至少 3 次（GBK→UTF-8 结论被用户否定后反复回到同一路径） |
| **重复搜索相同内容** | 至少 5 次（如 `addComment|commentBody` 搜了 3 遍） |
| **真正找到真相的轮次** | 最后一轮（读取 `server-utils.ts` 第 2096 行） |

---

## 二、探索路径时间线

### 第 1 阶段：表面判断（第 1-5 轮）
- **行为**：看到 `???` 直接判断为"编码问题"，给教科书式排查清单
- **问题**：没有动手搜索，只给方向性建议
- **用户反馈**："你去搜索"

### 第 2 阶段：盲目搜索（第 6-15 轮）
- **行为**：开始大量 grep，但搜索词过于宽泛
  - 搜 `prompt|server-utils|heartbeat|PATCH` → 117 个匹配
  - 搜 `comment.*body|body.*comment` → 227 个匹配
  - 搜 `addComment|commentBody|comment_body` → 9 个匹配（**重复搜了 3 遍**）
- **问题**：搜索词没有收敛性，每次都在同一层级打转

### 第 3 阶段：错误结论（第 16-20 轮）
- **行为**：得出"Windows 控制台 GBK 编码导致"结论
- **用户反馈**："补充上下文 agent 运行在本地 不在容器 容器里面只有数据库"
- **问题**：用户补充了关键上下文，但 Qwen **没有利用这个信息修正方向**，反而继续在同一条路径上搜索

### 第 4 阶段：反复打转（第 21-35 轮）
- **行为**：
  - 反复搜索 `addComment|commentBody`（第 4 遍、第 5 遍）
  - 反复思考"这条评论是怎么来的"
  - 提出多种假设（截断、SQL_ASCII、HTTP 编码、resultJson 序列化）但都不验证
  - **同一个思考模板重复出现至少 6 次**：
    ```
    "让我重新思考。用户说 agent 在本地跑，容器只有数据库..."
    "让我重新追踪这条评论的来源..."
    "关键问题是：为什么其他中文都能正常显示，唯独这条变成问号？"
    ```
- **问题**：思维陷入循环，没有从假设走向验证

### 第 5 阶段：接近真相（第 36-38 轮）
- **行为**：发现 `hasUnsafeTextProjectionDatabase()` 和 `safeForLegacyEncoding`
- **进展**：终于意识到 Paperclip 有编码感知逻辑
- **但**：仍然没有定位到 `server-utils.ts` 的 `String(chunk)`

### 第 6 阶段：找到真相（第 39-40+ 轮）
- **行为**：追踪到 `packages/adapter-utils/src/server-utils.ts` 第 2096 行
- **发现**：`const text = String(chunk);` — GBK Buffer 被当 UTF-8 解码
- **用户追问**："那为什么有的评论他能输出这个正常的中文"
- **最终回答**：区分了用户手动输入（API POST JSON）和 agent 自动生成（stdout 捕获）两条不同路径

## 三、为什么这么久才摸到真相？

### 原因 1：搜索策略低效（权重 40%）

**问题**：Qwen 用了"广撒网"策略，而不是"窄聚焦"策略。

| Qwen 的做法 | 应该的做法 |
|-------------|-----------|
| 搜 `comment.*body\|body.*comment` → 227 匹配 | 先确认评论写入 API 路由，再追写入函数 |
| 重复搜同一个词 5 遍 | 记录已搜索路径，避免重复 |
| 同时追 5 条假设 | 一次追一条，验证后排除 |

**根因**：Qwen 的 grep 工具没有限制搜索范围的能力（或 Qwen 没用好 `path` 参数），导致返回结果过多，无法从中提取有效线索。

### 原因 2：思维循环陷阱（权重 30%）

**问题**：Qwen 的思维链出现明显的"自我重复"模式：

```
Loop 1: "让我重新思考 → 可能原因 A/B/C → 让我搜索" → 没找到 → 
Loop 2: "让我重新思考 → 可能原因 A/B/C → 让我搜索" → 没找到 →
Loop 3: "让我重新思考 → 可能原因 A/B/C → 让我搜索" → ...
```

每次 Loop 的思考文本几乎一模一样，只是在排列组合不同的假设。**没有从上一轮学到任何东西**。

**根因**：Qwen 的会话上下文虽然保留了历史，但它的推理机制没有"排除法"意识——不会说"我已经排除了 A 和 B，现在只剩 C 需要验证"。

### 原因 3：没有利用用户反馈（权重 20%）

**关键节点**：用户说"agent 运行在本地 不在容器 容器里面只有数据库"

这是一个**高价值约束**，应该立刻缩小搜索范围到：
- Agent 进程的编码环境（Windows 本地）
- Agent → Paperclip 的数据传递路径（HTTP 或文件）

但 Qwen 的反应是："让我重新思考... 让我重新追踪..." 然后又回到同样的广撒网搜索。**用户反馈没有被转化为搜索策略的收敛**。

### 原因 4：验证缺失（权重 10%）

Qwen 提出了很多假设（截断、SQL_ASCII、HTTP 编码、resultJson 序列化），但**没有一个假设是通过验证确认的**，全都是"可能是..."然后继续搜下一个。

**正确做法**：提出假设 → 设计验证步骤 → 确认/排除 → 下一步。

## 四、是能力问题还是提示词问题？

### 能力问题（占 60%）

| 能力缺陷 | 表现 |
|---------|------|
| **搜索收敛能力弱** | 不会从宽泛搜索逐步收敛到具体文件 |
| **排除法意识缺失** | 不会记录"已排除 A，现在追 B" |
| **假设验证链断裂** | 提假设但不验证，直接跳到下一个假设 |
| **上下文利用不足** | 用户给的关键约束没有被转化为搜索策略 |
| **重复劳动** | 同一搜索词执行 5 遍，同一思考模板循环 6 次 |

### 提示词问题（占 40%）

| 提示词缺陷 | 影响 |
|-----------|------|
| **没有要求"先确认现象再追根因"** | Qwen 一上来就给排查清单，没有先确认"数据库里存的是问号还是正常中文" |
| **没有要求"记录已搜索路径"** | 导致重复搜索 |
| **没有要求"一次追一条路径"** | 同时追 5 条假设，每条都不深入 |
| **没有约束"思维链去重"** | 同样的思考模板循环出现 |

## 五、优化建议

### 对 Qwen 的提示词优化

```markdown
## 问题排查纪律

1. **先确认现象**：在追根因前，先确认事实（如：数据库里存的是什么？API 返回的是什么？）
2. **一次追一条路径**：不要同时追多个假设，选定一条后验证完再换
3. **记录已搜索路径**：避免重复搜索同一个词/同一个文件
4. **利用用户反馈收敛**：用户给的每个约束都应缩小搜索范围
5. **假设必须验证**：提出假设后，设计验证步骤，确认或排除后再继续
```

### 对用户的提示词优化

| 现在的说法 | 更好的说法 |
|-----------|-----------|
| "你去搜索" | "去搜索 `buildHeartbeatRunIssueComment` 函数，看它从哪里读取评论文本" |
| "你已经摸过一轮了，应该可以很快摸到真相" | "你已经读过 `heartbeat.ts` 和 `server-utils.ts`，真相在 stdout 编码处理那里，去确认 `String(chunk)` 的解码方式" |
| "说人话" | "用一句话说清楚根因和修复方案，不要列排查清单" |

**核心原则**：Qwen 在没有明确指引时会"广撒网 + 多假设并行"，这会浪费大量轮次。**给出具体搜索目标和验证方向**可以大幅缩短探索时间。

### 对 Qwen 能力的长期优化方向

1. **搜索策略记忆**：让 Qwen 在思考中维护一个"已搜索路径清单"
2. **排除法显式化**：要求 Qwen 在每轮思考开头说"已排除：A、B；待验证：C"
3. **验证驱动**：提假设后必须附带验证步骤，不能只说"可能是..."
4. **收敛阈值**：搜索超过 5 次没结果时，自动切换到"先确认现象"模式

## 六、结论

**Qwen 这么久才摸到真相，60% 是能力问题，40% 是提示词问题。**

### 能力问题的核心

不是"多轮工具执行"的问题（这是正常的），而是**多轮无效执行**的问题。Qwen 在 40+ 轮中，至少有 20 轮是重复搜索或循环思考。如果把有效搜索和无效搜索分开，真相应该在 **8-12 轮** 内就能找到。

### 提示词问题的核心

用户早期的提示词（"你去搜索"）过于开放，没有给搜索方向和收敛目标。后期的提示词（"你已经摸过一轮了"）虽然暗示了方向，但仍然没有给出具体路径。

### 最优协作模式

```
用户：给出具体搜索目标（如"去看 server-utils.ts 的 stdout 处理"）
Qwen：读取 → 定位问题 → 给出结论
```

而不是：

```
用户："你去搜索"
Qwen：广撒网 → 多假设 → 循环思考 → ...（40+ 轮）
```
