## 1. 调度入口（服务进程）

当 `config.heartbeatSchedulerEnabled` 为真时，`server/src/index.ts` 会：

1. **启动时**跑一串恢复：`promoteDueScheduledRetries`、`resumeQueuedRuns`、`reconcileStrandedAssignedIssues`、`reconcileIssueGraphLiveness`、`scanSilentActiveRuns`、`reconcileProductivityReviews`（失败记日志，不挡启动）。
2. 随后调用 **`heartbeat.tickTimers(new Date())`**。
3. 按 **`config.heartbeatSchedulerIntervalMs`** **周期重复**：同一套恢复 + **`tickTimers`**。

即：**timer 心跳与若干 reconcile 共用同一节拍**，具体间隔以配置为准。

---

## 2. `tickTimers` 在做什么（摘要）

实现：`server/src/services/heartbeat.ts` 内 `tickTimers`。

顺序大致为：

1. `reconcileTerminalIssueRunningRuns`（终端态 issue 仍挂 running run 的对账）。
2. 遍历库内 **agents**，对每个 agent：
   - `paused` / `terminated` / `pending_approval` → **跳过**。
   - **`agent.role` 须在实例实验配置 `timerHeartbeatEligibleAgentRoles` 集合内**，否则 **跳过**（与执行单 **007** 术语「Eligible」一致）。
   - `parseHeartbeatPolicy(agent)`：`enabled` 为假或 `intervalSec <= 0` → **跳过**。
   - 用 `lastHeartbeatAt ?? createdAt` 与 `now` 比较 interval；未到点 → **跳过**。
   - 调用 `enqueueWakeup(..., source: "timer", reason: "heartbeat_timer", ...)`；若返回 **无 run**（含 timer 闸门跳过），计入 **`skipped`**。
3. `tickDueIssueMonitors(now)`（issue monitor 定时触发），其 `checked` / `triggered` / `skipped` 并入返回值统计。

**返回值字段：** `checked`、`enqueued`、`skipped`、`terminalIssueRunsScanned`、`terminalIssueRunsReconciled` 等（以代码为准）。

## 3. 与「无经办不唤醒」闸门的关系

执行单 **[`../执行/007-心跳调度无issue不触发.md`](../执行/007-心跳调度无issue不触发.md)**：对 **`source === "timer"`** 在入队前要求存在 **可归因、且 status 非 backlog/done/cancelled** 的 issue；不满足则 **不写 `heartbeatRuns`**，并 **`agent_wakeup_requests` 留痕 skipped**。

本探查单 **只陈述**：`tickTimers` 路径最终会走到上述 `enqueueWakeup`；是否在闸门处跳过以运行库数据与测试为准。
