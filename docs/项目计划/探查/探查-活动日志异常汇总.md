## 2. 执行环境释放失败（`environment.lease_released` + `status: failed`）

与 **ROU-17**（ea9c971f…）同一波执行相关，`failureReason` 摘录为 Cursor 侧返回（英文原文）：

> You've hit your usage limit … Ultra … Spend Limit … monthly cycle ends on 5/14/2026.

### 探查结论：🔴 需跟进（计费/适配器）

**归因分析**（代码路径）：

1. agent 指定 **composer-2-fast**，走 Auto/Composer 池（非按 token 的 API 模型池），与「Ultra / Spend limit」报错 **不矛盾**。
2. 适配器返回 `adapter_failed` 后，heartbeat 收尾逻辑（`heartbeat.ts:7771-7772`）将 errorCode 设为 `latestRun?.errorCode ?? "cancelled"`，即透传适配器错误码。
3. Paperclip 侧的恢复系统随后检测到 ROU-17 无活跃执行路径 + 最近一次 run 不成功 → 触发 stranded recovery（见第 3 节）。

**三种可能根因（需对照 Cursor 账户页确认）**：

| 可能原因 | 说明 | 验证方式 |
| --- | --- | --- |
| Ultra 月度额度耗尽 | composer-2-fast 可能消耗 Ultra 订阅池配额 | 检查 Cursor 账户 Usage 页 |
| Spend Limit 触发 | 独立的计费上限（与 Ultra 订阅并行） | 检查 Cursor 账户 Billing → Spend limit |
| 适配器错误映射 | Cursor 侧实际错误不是额度，但适配器映射为 usage limit 文案 | 需对照原始错误响应与适配器错误码映射表 |

**建议动作**：
- 登录 Cursor 账户检查 Usage / Billing 页确认真实状态
- 如额度确实耗尽：等待 5/14 周期重置，或调整 Spend limit
- 如额度充足但持续报错：怀疑适配器错误映射，需排查 `cursor-local` adapter 的错误分类逻辑

## 3. 工单恢复对账（`issue.updated` + 恢复链）

| 标识 | 现象 | 关键字段 |
| --- | --- | --- |
| **ROU-17** | 曾被标 **blocked** | `latestRunErrorCode: adapter_failed`，`recoveryCause: stranded_assigned_issue`，阻断指向 ROU-18 |
| **ROU-18** | 恢复单对账 | `source: recovery.reconcile_stranded_recovery_issue`，`latestRunStatus: cancelled` |

### 探查结论：🟡 恢复链设计正确，存在可观测性改进空间

**完整恢复链路**（代码追踪）：

```
ROU-17 (in_progress, assigned 开发-Cursor)
  │  adapter_failed → run terminated
  │  no live execution path
  ▼
reconcileStrandedAssignedIssues()  [recovery/service.ts:1784]
  │  检测到 assigned issue 无活跃执行路径
  │  didAutomaticRecoveryFail() → true
  ▼
escalateStrandedAssignedIssue()  [recovery/service.ts:1677]
  │  ensureStrandedIssueRecoveryIssue() → 创建 ROU-18
  │  ROU-18 分配 CEO，阻断 ROU-17
  ▼
ROU-18 (todo → in_progress, assigned CEO)
  │  CEO agent 拉活执行
  │  run → cancelled（手动取消或执行失败）
  ▼
reconcileStrandedAssignedIssues()  [recovery/service.ts:1833]
  │  isStrandedIssueRecoveryIssue(ROU-18) && isUnsuccessfulTerminalRun
  ▼
escalateStrandedRecoveryIssueInPlace()  [recovery/service.ts:1596]
  │  嵌套恢复抑制（line 1685: nestedRecoverySuppressed）
  │  ROU-18 → blocked，不创建 ROU-19
  │  活动日志: source="recovery.reconcile_stranded_recovery_issue"
```

**设计正确性确认**：

- ✅ **无无限递归**：`isStrandedIssueRecoveryIssue()` 守卫阻止 recovery issue 再创建嵌套 recovery（`service.ts:1685`）
- ✅ **明确所有权**：CEO 角色自动获得恢复单指派（`resolveStrandedIssueRecoveryOwnerAgentId` 优先 CTO/CEO，`service.ts:1375-1409`）
- ✅ **Board 可见**：两个工单均 blocked，阻断关系可追溯

**可观测性改进建议**：

| 改进项 | 优先级 | 说明 |
| --- | --- | --- |
| 恢复单卡住告警 | 中 | 恢复单 blocked 超过 N 小时后，可通过 Board 或活动日志产生显式通知（当前仅在 Board 上显示 blocked 状态，无主动推送） |
| 恢复单 cancel 的根因记录 | 低 | ROU-18 cancelled 时，cancel 原因（手动 vs 超时 vs adapter 失败）未在活动日志中区分——`errorCode: cancelled` 对三种场景一致 |
| recovery sweep 频率可观测 | 低 | `reconcileStrandedAssignedIssues` 的调用频率和结果无仪表盘暴露 |

### 3a. 手动 cancel 与恢复链的交互

**关键行为**：当对 recovery issue（如 ROU-18）执行手动 cancel 时：
1. `cancelRunInternal` 将 run → cancelled，释放执行锁
2. 下次 `reconcileStrandedAssignedIssues` 检测到 recovery issue 无活跃执行路径
3. 触发 `escalateStrandedRecoveryIssueInPlace` → 标记 blocked，**不创建新恢复单**
4. 原始 issue（ROU-17）仍被 ROU-18 阻断 → **两条工单同时卡住**

**结论**：这是**设计意图**——避免无限恢复链。但意味着：手动 cancel 恢复单的 run 后，**必须手动处理**（取消 ROU-18 工单本身、或清掉 ROU-17 对 ROU-18 的阻断），否则两条工单永久 blocked。

**建议**：在恢复单被 blocked 时，向恢复单评论中追加明确的「下一步」操作指引（当前已有评论，但可更突出「需手动解决」）。
