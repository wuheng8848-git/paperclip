# 探查：回形针进程泄露 / CPU 爆表 / 数据库挂掉 — 当前假设

> 本文档记录**操作者当下的因果假设**，便于后续取证与对账；**不是**已验证结论，也**不要求**本文自洽成「最终根因报告」。

---

## 一句话总结

**数据库起不来 / 挂了** ← 用了**一键杀服务**（如 `pnpm dev:nuke` 及同类脚本）收尾；**为什么要杀** ← **回形针崩了、进程飙满、CPU 爆表**；**CPU 爆表** ← 开了很多 agent 有贡献，但**主因不在此**；**主因** ← **Codebuddy 无头模式**，叠加**当时写的适配器不合适**、**回形针里智能体配置用得不好**，**可能**再叠一点项目代码瑕疵。

---

## 假设明细（按链路顺序）

1. **数据库状态**：当前或当时出现的库不可用/挂掉，与**一键杀服务**的收尾方式有关（例如连带着内嵌 Postgres、锁文件、或外部库连接被打断后的残留状态等——细节待取证）。

2. **一键杀服务的动机**：并非 routine 操作，而是因为**回形针失控**：进程数量异常增多，**CPU 拉满**，必须先止损。

3. **CPU 爆表**：**很多 agent / 并行唤醒**有放大作用，但在本假设里**不是主要原因**。

4. **主要原因（本假设核心）**：**Codebuddy 无头模式**的行为 + **既有适配器与其实际不匹配** + **Paperclip 侧智能体配置使用不当**；**可能**还有少量**项目代码**边界未兜住，加重泄漏或重复拉起。

5. **本文档立场**：以上均为**假设**；后续若要做正式根因，应单独走日志、进程树、适配器与配置快照、复现路径，再更新结论。**不在此反驳假设本身。**
