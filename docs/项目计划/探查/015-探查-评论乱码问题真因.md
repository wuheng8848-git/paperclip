# 乱码问题真因诊断报告

> 生成日期：2026-05-16
> 问题来源：Paperclip 项目 UI 评论中出现 `???` 乱码
> 项目位置：`C:\Users\wuhen\code\paperclip-latest-20260512`

---

## 一、问题现象

用户在 Paperclip UI 中看到一条 agent 自动生成的评论显示为乱码：

```
?????????:?API?prompt??(??)+ ???????????(??)?????:?adapter-utils/src/server-utils.ts?prompt template????PATCH?????heartbeat????API????,??Qwen???????
```

**关键特征**：
- 其他中文评论正常显示（用户手动输入的评论、系统生成的标签等）
- 唯独这条 agent 运行后自动生成的评论变成问号
- 乱码内容可辨认出原文应该是"更新了 API prompt 模板"等工作摘要

---

## 二、数据路径（真相链路）

```
Agent stdout（GBK 编码）
    ↓
Paperclip 服务端通过 spawn 子进程捕获 stdout
    ↓
server-utils.ts 第 2096 行：const text = String(chunk)
    ↓  ← GBK Buffer 被当 UTF-8 解码 → 乱码
resultJson.summary / result / message 字段
    ↓
buildHeartbeatRunIssueComment() 提取评论文本
    ↓
issuesSvc.addComment() 写入数据库
    ↓
前端 UI 读取并渲染 → 显示 ???
```

---

## 三、根因

### 核心问题

**Paperclip 服务在读取 agent 输出到写入数据库之前，有转码/编码处理环节，没有考虑 Windows 环境兼容性。**

### 作者背景与盲区

| 维度 | Paperclip 作者环境 | 用户实际环境 |
|------|-------------------|-------------|
| 操作系统 | Linux/macOS | Windows |
| 控制台编码 | UTF-8（系统默认） | GBK（CP936，控制台默认） |
| 编码意识 | 不需要考虑（Linux 默认 UTF-8） | 需要处理 GBK ↔ UTF-8 转换 |

作者在英文/Linux 环境下编译和开发，系统默认就是 UTF-8，**不需要做任何编码转换**，所有文字、JSON、stdout 自然就是 UTF-8。所以代码里**没有显式处理编码兼容性**的逻辑。

### 数据路径上的编码陷阱

```
Agent (Qwen) 运行在 Windows
    ↓
stdout 输出中文（Windows 控制台编码 = GBK）
    ↓
Paperclip 服务读取 stdout（String(chunk)，Node.js 默认 UTF-8 解码）
    ↓  ← 这里 GBK 字节流被当 UTF-8 读，已经损坏
resultJson 字段写入数据库
    ↓
前端 UI 读取 → 显示 ???
```

### 为什么不是 agent 的问题

agent 是 Qwen CLI（Node.js 进程），它本身输出 UTF-8 没问题。但**在 Windows 控制台环境下**，如果进程继承了控制台的代码页设置（CP936/GBK），或者通过某种管道/重定向输出时编码不一致，就会出现 GBK → UTF-8 的解码错误。

### 为什么其他评论正常

| 评论类型 | 路径 | 编码 | 结果 |
|---------|------|------|------|
| 用户手动输入 | UI → API POST JSON | UTF-8（HTTP 默认） | ✅ |
| 系统生成标签 | 服务端代码直接生成 | UTF-8 | ✅ |
| **Agent 自动生成** | **stdout → Paperclip 读取 → DB** | **GBK → UTF-8 损坏** | **❌** |

---

## 四、关键上下文

### 不是 agent 的问题，是服务的问题

**不能因为用户在 Windows 就改 agent 让它适配**。正确的做法是：

**Paperclip 服务自身应该检测运行环境，处理所有可能导致乱码的编码逻辑。**

这是服务端的职责——作为跨平台应用，它应该：
1. 检测当前运行平台（Windows/Linux/macOS）
2. 检测控制台/进程的编码设置
3. 在读取 stdout/stderr、写入数据库之前，做正确的编码转换
4. 确保所有 JSON 解析、文本处理环节使用统一的 UTF-8

### 数据库编码

- PostgreSQL 初始化时使用 `--encoding=UTF8`（见 `index.ts` 的 `initdbFlags`）
- 数据库本身编码正确，问题在**写入前**的数据已经损坏

### Paperclip 已有编码感知（但不完整）

代码中存在 `hasUnsafeTextProjectionDatabase()` 函数，用于检测数据库 `server_encoding` 是否为 `SQL_ASCII`，并在读取时做安全截断。但这只影响**读取行为**，不处理**写入前的编码转换**。

---

## 五、修复方案（未实施）

### 修复方向：服务端自适应编码处理

**不是改 agent，是改 Paperclip 服务**。服务应该在启动时检测环境，然后在所有文字处理环节做适配：

```typescript
// 伪代码示例
function detectPlatformEncoding(): BufferEncoding {
  if (process.platform === 'win32') {
    // Windows 控制台可能是 GBK/CP936
    // 需要检测或强制转换为 UTF-8
    return 'utf8';
  }
  return 'utf8'; // Linux/macOS 默认就是 UTF-8
}

// 在读取 stdout 时
const text = process.platform === 'win32'
  ? iconv.decode(chunk, detectWindowsConsoleEncoding()) // GBK → UTF-8
  : String(chunk); // UTF-8 环境直接读
```

### 具体修复点

| 修复点 | 位置 | 动作 |
|--------|------|------|
| stdout/stderr 读取 | `server-utils.ts` spawn 子进程监听 | 检测平台编码，必要时做 iconv 转换 |
| JSON 解析 | `qwen-local/execute.ts` parseQwenStreamJsonOutput | 确保 JSON.parse 前字符串是 UTF-8 |
| 数据库写入 | `heartbeat.ts` buildHeartbeatRunIssueComment → addComment | 写入前验证/转换编码 |
| 启动检测 | `server/src/index.ts` 启动横幅 | 输出当前平台、编码环境到日志 |

### 需要的依赖

- `iconv-lite`：用于 GBK ↔ UTF-8 编码转换
- Windows 控制台编码检测逻辑（`chcp` 命令或 `GetConsoleCP` API）

---

## 六、结论

**这不是 i18n 翻译问题**，而是**存储/传输层面的编码问题**。

**真因**：Agent 在 Windows 本地运行时，stdout 输出 GBK 编码的中文，Paperclip 服务端通过 `String(chunk)` 按 UTF-8 解码，导致乱码写入数据库。

**影响范围**：仅影响 agent 自动生成的评论，不影响用户手动输入的评论和系统 UI 标签。
