## CLI 技能读取：显式 / 隐式、与「无头」（补充探查）

本节回答：**配置上「显式 / 隐式」指什么**；**是否仅 Codex**；**无头是否换一条读技能链路**。

### 1. 枚举「有哪些技能」——各适配器共用

- **`readPaperclipRuntimeSkillEntries(config, moduleDir)`**（`packages/adapter-utils/src/server-utils.ts`）  
  - 若 `config.paperclipRuntimeSkills` 有有效条目 → **只用配置条目**（公司/心跳注入的那张表）。  
  - 否则 → **`listPaperclipSkillEntries`**：扫适配器包旁 Paperclip 技能根目录（bundled），子目录即技能，并带 `required` 等元数据。

### 2. 「显式 / 隐式」——仍由 `paperclipSkillSync` 一把梭

- **`readPaperclipSkillSyncPreference`**：`explicit === true` 当且仅当 `paperclipSkillSync` 对象上 **存在自有属性** `desiredSkills`（`hasOwnProperty`），即使用户写成空数组也算显式。  
- **`resolvePaperclipDesiredSkillNames`**：  
  - **隐式**（`explicit === false`）：只返回清单里 **`required: true`** 的 key（外加去重）。  
  - **显式**：**required ∪ `desiredSkills`**（经 `canonicalizeDesiredPaperclipSkillReference` 归一成 canonical key）。

### 3. 不是 Codex 专利

- Codex：`resolveCodexDesiredSkillNames` → 直接转发 `resolvePaperclipDesiredSkillNames`。  
- Claude：`resolveClaudeDesiredSkillNames` → **同样转发**；`execute` 里对 `readPaperclipRuntimeSkillEntries` 过滤后再 `prepareClaudePromptBundle`（把选中技能的 `SKILL.md` 等卷进提示侧资产，而非 Codex 式 symlink 一家）。  
- Cursor / Gemini / OpenCode / Grok / Qwen / Codebuddy / PI / ACP 等：`grep` 可见均在各包 `execute.ts` / `skills.ts` 调用 **`readPaperclipRuntimeSkillEntries` + `resolvePaperclipDesiredSkillNames`（或薄封装）**；差异在**注入方式**（symlink、runtime 目录、拼 prompt、同步远端），不在「显式/隐式」判定。

### 4. 「无头」不换读技能管线

- 仓库里 **headless** 多指 **非交互、stdin / `--print` / 跳权限** 等 CLI 形态；**同一条 `execute`** 内仍先 **`readPaperclipRuntimeSkillEntries` → resolve desired → 各适配器注入**。  
- 未发现单独「无头专用」技能发现函数；若指其他脚本/产品入口，需另对那条命令取证。

### 5. 易混点（与 045 相关）

- **`ensureCodexSkillsInjected` 的默认参数**：仅当调用方**不传** `desiredSkillNames` 时，helper 内才用「`skillsEntries` 全 key」；**Adapter `execute` 正常路径已传入 resolve 后的列表**，行为以 **§2 显式/隐式** 为准。  
- **全文灌入**：当前多为「选中技能整目录/整 `SKILL.md` 进入子进程可见范围或 system 侧」；**按章节懒加载**属于 045 拟议增量，**正文级懒加载**仍未落地。**stdin 侧**已由 `shouldMinimizeAdapterRuntimeSkillNotes` + Qwen/CodeBuddy `skill_note` 做首包裁剪（见上表末行）。
