---
title: 执行工作区和运行时服务
summary: 项目运行时配置、执行工作区和事务运行如何协同工作
---

本文档记录了 Paperclip 中项目、执行工作区和事务运行的预期运行时模型。

Paperclip 现在将其呈现为工作区命令模型：

- `服务` 是保持受监督的长运行命令。
- `作业` 是运行一次并退出的单次命令。
- 原始运行时 JSON 仍可用于高级配置，但它不再是主要的心智模型。

## 项目运行时配置

你可以在项目工作区本身上定义如何运行项目。

- 项目工作区运行时配置描述该项目检出可用的服务和作业。
- 这是子执行工作区可能继承的默认运行时配置。
- 定义配置本身不会启动任何内容。

## 手动运行时控制

工作区命令从 UI 手动控制。

- 项目工作区服务从项目工作区 UI 启动和停止，项目作业可以在那里按需运行。
- 执行工作区服务从执行工作区 UI 启动和停止，执行工作区作业可以在那里按需运行。
- Paperclip 不会作为事务执行的一部分自动启动或停止这些工作区服务。
- Paperclip 也不会在服务器启动时自动重启工作区服务。

## 执行工作区继承

执行工作区将代码和运行时状态与项目主要工作区隔离。

- 隔离的执行工作区有自己的检出路径、分支和本地运行时实例。
- 运行时配置可能默认从链接的项目工作区继承。
- 执行工作区可以用自己的工作区特定设置覆盖该运行时配置。
- 继承的配置回答"存在哪些命令以及如何运行它们"，但任何正在运行的服务进程仍然特定于该执行工作区。

## 事务和执行工作区

事务附加到执行工作区行为，而不是自动运行时管理。

- 当你选择隔离工作区模式时，事务可能会创建一个新的执行工作区。
- 当你选择重用时，事务可能会重用现有的执行工作区。
- 多个事务可能有意共享一个执行工作区，以便它们可以针对同一分支和正在运行的运行时服务进行操作。
- 分配或运行事务不会自动启动或停止该工作区的工作区服务。

## 执行工作区生命周期

执行工作区是持久的，直到人类关闭它们。

- UI 可以归档执行工作区。
- 关闭执行工作区会停止其运行时服务，并在允许时清理其工作区工件。
- 指向项目主要检出的共享工作区在清理期间比一次性隔离工作区受到更保守的处理。

## 心跳运行期间的已解析工作区逻辑

心跳仍然为运行解析工作区，但这是关于代码位置和会话连续性的，而不是运行时服务控制。

1. 心跳为运行解析基础工作区。
2. Paperclip 实现有效的执行工作区，包括在需要时创建或重用工作树。
3. Paperclip 持久化执行工作区元数据，如路径、引用和预配置设置。
4. 心跳将解析的代码工作区传递给智能体运行。
5. 工作区运行时服务保持手动 UI 管理的控制，而不是自动心跳管理的服务。

## 当前实现保证

在当前实现中：

- 项目工作区命令配置是执行工作区 UI 控制的后备。
- 执行工作区运行时覆盖存储在执行工作区上。
- 心跳运行不会自动启动工作区服务。
- 服务器启动不会自动重启工作区服务。
