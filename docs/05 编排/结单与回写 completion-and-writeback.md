# 结单与回写

**定位：** run 跑完后 **怎么算交差**——工人回写什么、董事会怎么验、**run 成功 ≠ 事务完成**。

验 run 的完整 SOP（环境、running 中、API 路径）仍见 `docs/项目计划/最佳实践/024-…`；本文收 **编排闭环** 必做项。

---

## 1. 两层分开验

| 层 | 看什么 | 人话 |
| --- | --- | --- |
| **run** | `succeeded` / `failed` / `cancelled` | 这趟 CLI 是否正常跑完 |
| **issue** | `done` / `blocked` / … | 业务是否交付 |

**run succeeded  alone 不够**——还要事务状态 + 回写 + 董事会拍板（009 §6）。

---

## 2. 结单四件套（现阶段）

1. **文档/domain 回写** — 该更新的 `docs/` 或任务指定路径有改动或指针。  
2. **事务评论** — 可核验的 **中文摘要**（别贴整段 NDJSON）。  
3. **run 终态** — 预期 **succeeded**；事务已 done 时 run 不应莫名 **cancelled**（关单对账回归点）。  
4. **董事会 `done`** — 人确认完成定义满足。

与 [派单与能力表](派单与能力表%20dispatch-and-roles.md) §5 一致。

---

## 3. 工人回写写什么

| 要有 | 不要 |
| --- | --- |
| 做了什么、改哪些路径 | 整页日志 |
| blocked 原因 + 建议下一步 |  silently 失败 |
| 若不能改 issue 状态，评论里说明 | 假 done |

issue 描述里应事先写清 **完成定义**（ROU-6 类：做到哪算过）。

---

## 4. 董事会验 run（最小闸门）

**环境：** `GET /api/health` ok；端口/公司对（如 4100）。

**跑完后必看：**

| # | 通过标准 |
| --- | --- |
| F1 | run **`succeeded`**（测失败路径则预期 `failed` + 可读 error） |
| F2 | `exitCode` 成功路径通常 **0** |
| F3 | 运行清单有日志、`logBytes` > 0 |
| F4 | 事务 **done**（或 agreed blocked） |
| F5 | 评论里有交付摘要或路径 |
| F6 | issue done 时 run 非「对账 cancel」类文案 |

**UI：** 工作 → **运行清单**（勿写「编排注入」）。

---

## 5. 探查收口一句话（写进执行单）

> 于 {时间} 对 {ROU-xx}：run {id} **succeeded**，事务 **done**，评论已摘要，文档已回写 {路径}。
