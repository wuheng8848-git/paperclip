# 编排改进与演进

**定位：** 编排 **怎么越用越好**——先跑顺一条闭环，再单开自动化；改产品批次时怎么收口。**不是**代码目录地图。

---

## 1. 阶段顺序（强制）

```text
阶段 0：关默认自动化 + 汉化 + 适配器跑稳（见 实例开关）
    ↓
阶段 1：单事务 P0 闭环（派 → run → 回写 → done）
    ↓
阶段 2：按痛点单开一条 env / 实验项 + 48h 观察
    ↓
阶段 3：批次改版（评论唤起、关单对账…）走固定套路
```

路线图详述：[参考/路线图一人公司](参考/路线图一人公司%20roadmap-to-one-person-company.md)

---

## 2. 单开自动化时（摘自阶段零）

1. 记录 **重复人工步骤**。  
2. **只开一条** 控件（见 [编排控件一览](编排控件一览%20orchestration-controls.md)）。  
3. [结单与回写](结单与回写%20completion-and-writeback.md) 验一条事务。  
4. 填 [实例开关](实例开关与阶段零%20instance-switches.md) §5 启用表。

---

## 3. 改版批次固定套路（042～053 类）

| 步 | 做什么 |
| --- | --- |
| 探查 | 现象 + DB/API 证据 → `探查/` |
| 设计 | 对齐长期需求母本（如 **24 评论唤起**） |
| 评审 | 董事会 / CTO 拍板范围 |
| 执行 | `执行/` 单 + 代码 |
| 回写 | 母本「实现与需求对齐」表 + 验证证据 |

**单条执行单结案 ≠ 整轮结案**——批次全完要在母本 **24** 填对齐表。

源：最佳实践 016

---

## 4. 怎么判断「编排变好了」

| 信号 | 说明 |
| --- | --- |
| P0 事务稳定 done | ROU-6 类不再反复 cancel |
| 衍伸单减少 | 无因 recovery 冒出一串子单 |
| 人工步骤可命名 | 能说出「想自动化的那一步」 |
| 运行清单可读 | 中文、能看懂谁在跑、为何 wake |

个案长表仍在 `探查/`；不要把手册写成个案堆叠。
