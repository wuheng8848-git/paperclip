# 实践：董事会派单协议

本文固化一种工作方式：**对话框里的董事会负责判断，Paperclip 负责执行控制，执行智能体只做边界清楚的工作**。

它不是产品功能，也不是自动化实现；它是当前阶段防止执行面失控、重复派单、错误自愈循环的人工协议。

---

## 1. 角色边界

### 董事会（人类 + 对话框里的高级脑）

负责：

- 判断问题是否值得进入 Paperclip。
- 判断当前是讨论、探查、实现、验证，还是先不动。
- 拆清任务边界、执行者、完成定义和失败接管。
- 观察执行时间线，判断是否真正完成。
- 决定是否人工标记完成、阻塞、继续或归档。

不负责：

- 把所有想法都直接派成事务。
- 让执行智能体自行决定大路线。
- 在控制面 bug 未修复前，把控制面修复交给同一个出错执行者自救。

### Paperclip（执行控制面）

负责：

- 保存事务、执行者、心跳、运行记录、活动时间线。
- 唤醒执行智能体，提供可观察的执行证据。
- 暴露运行中、阻塞、缺少处置、重复唤醒等状态。

不负责：

- 替董事会决定战略优先级。
- 在任务边界不清时自动扩写出一串子任务。
- 把“执行者说完成了”直接等同于系统完成。

### 执行智能体（如 Qwen）

负责：

- 执行明确、窄范围、可验收的任务。
- 输出证据、结论、改动范围和下一步建议。
- 如果无法修改 Paperclip 状态，在评论里明确建议处置。

不负责：

- 自行扩大任务范围。
- 自行创建后续子任务。
- 自行决定项目路线或优先级。
- 在自己刚发生适配器、密钥、状态收口错误后继续修复自己。

## 2. 派单前必须回答的八项

任何事务进入 Paperclip 前，董事会先用下面八项判断。说不清就不要派单。

```text
是否进入 Paperclip：
目标：
执行者：
任务类型：
任务边界：
禁止事项：
完成定义：
失败接管：
观察点：
```

字段口径：

- **是否进入 Paperclip**：进入 / 不进入 / 先讨论 / 先本地探查。
- **目标**：一句话说明希望得到什么结果。
- **执行者**：Qwen / Codex / 人工 / 暂不指定。
- **任务类型**：探查 / 实现 / 验证 / 文档 / 运维 / 审计。
- **任务边界**：允许读哪些、改哪些、交付什么。
- **禁止事项**：不得新建事务、不得改某目录、不得重启服务、不得读无关项目等。
- **完成定义**：什么证据出现后算完成。
- **失败接管**：失败后由谁判断，是否允许自动恢复，是否需要人工。
- **观察点**：活动时间线、run 日志、最终评论、git diff、UI 状态等。

## 3. 什么时候派给 Paperclip

适合派单：

- 目标清楚，边界能写进事务。
- 可以由执行者独立完成一轮。
- 结果能通过代码 diff、文档、日志、UI 或 API 验证。
- 失败不会破坏当前控制面稳定性。
- 需要留下可追踪运行记录。

不适合派单：

- 还在讨论系统定位、角色职责、优先级。
- 需要高级判断而不是执行。
- 任务本身是在修 Paperclip 调度、心跳、状态收口、适配器认证等控制面关键路径。
- 执行者刚在同一链路上出错，继续派给它会形成“失败者修复失败者”的循环。
- 任务需要人类当场拍板，例如是否新建公司、是否迁入 `routic`、是否扩大执行范围。

灰区处理：

- 先在董事会讨论。
- 必要时由董事会本地只读探查。
- 如果仍需执行，再开一个窄范围 Paperclip 事务。

---

## 4. 派给谁

### Qwen

适合：

- 窄范围实现。
- 明确文件范围的汉化、适配器小修、UI 小改。
- 有清晰完成定义的探查。

必须限制：

- 一次只做一个小单。
- 不允许自行新建后续事务。
- 不允许自行扩大到全仓扫描，除非任务明确要求。
- 完成后必须给出处置建议：建议标完成 / 建议继续 / 建议阻塞 / 需要人工判断。

### Codex

适合：

- 董事会讨论。
- CTO / 审计脑 / 异常恢复判断。
- 跨文档、跨代码、跨事务的根因分析。
- 任务拆解、验收、优先级判断。

必须限制：

- 不作为普通开发主力。
- 不默认接所有心跳。
- 只在高价值、低频、需要判断的节点介入。

### 人工

适合：

- 标记事务最终状态。
- 授权密钥、公司、正式环境迁移。
- 判断执行者输出是否符合真实业务目标。
- 控制面处于不可信状态时的最终收口。

## 5. Paperclip 事务模板

派给执行者的事务正文建议使用下面模板。

```text
【任务类型】
探查 / 实现 / 验证 / 文档 / 运维 / 审计

【目标】
...

【范围】
- 允许读取：
- 允许修改：
- 不要触碰：

【禁止事项】
- 不得自行新建子任务。
- 不得扩大范围。
- 不得把未验证结论写成事实。
- 不得在未授权时推送远端或创建 PR。

【完成定义】
- ...

【收口要求】
完成后在评论中明确写出：
- 结论：
- 改动/证据：
- 验证：
- 建议处置：标完成 / 继续 / 阻塞 / 人工判断

如果无法调用 Paperclip API 修改事务状态，只写建议处置，不要反复自我唤醒。
```

## 6. 执行后的观察流程

董事会按固定顺序验收：

1. 看 Paperclip 活动时间线：是否重复唤醒、是否新建了意外事务、是否出现 recovery。
2. 看最终评论：是否给出明确结论、证据、验证和建议处置。
3. 看 run 日志：是否真的执行了任务，而不是只读文档或空跑。
4. 看 git diff / UI / API：是否符合完成定义。
5. 决定事务状态：完成、继续、阻塞、取消或人工判断。

执行者“运行成功”不等于事务完成。只有完成定义满足，并且董事会验收通过，才算完成。

## 7. 失败与恢复规则

### 普通失败

例如测试失败、编译失败、文件冲突：

- 允许同一执行者继续一次，前提是失败原因清楚。
- 第二次仍失败，转董事会判断。

### 控制面失败

例如：

- 适配器密钥、模型、端点错误。
- Paperclip API 调错地址。
- 执行者无法回写事务状态。
- recovery 反复创建或反复唤醒。
- 心跳 run 卡死、僵尸 run、状态与活动不一致。

处理规则：

- 不让同一个失败执行者继续自动修复自己。
- 董事会先本地探查或由高级脑判断。
- 需要实现时，拆成明确修复单，再决定执行者。
- 修复前可以人工收口，避免系统继续自我增殖。

### 缺少处置状态

如果执行者完成了实际工作，但没有能力把事务从执行中改为完成：

- 不立即判定执行失败。
- 先看最终评论是否有明确建议处置。
- 若证据充分，由人工或董事会标记完成。
- 同时记录为控制面改进需求：让 Paperclip 从最终评论中识别“建议标完成”，并弹出确认。

---

## 8. 自动化的边界

短期只自动化观察，不自动化最终裁决。

可以自动化：

- 汇总运行中 run。
- 汇总配置了心跳的 agent。
- 提醒缺少处置状态。
- 提醒重复唤醒或 recovery 循环。
- 提醒某执行者连续失败。

不应立即自动化：

- 自动把事务标完成。
- 自动让执行者无限续跑。
- 自动创建后续子任务。
- 自动把测试环境配置迁到 `routic`。
- 自动新建公司或改变公司边界。

---

## 9. 当前阶段的默认工作流

```text
董事会讨论
  -> 判断是否派单
  -> 写清八项
  -> Paperclip 派给合适执行者
  -> 执行者运行
  -> 董事会观察时间线和证据
  -> 人工或董事会收口状态
  -> 必要时再拆下一单
```

默认原则：

- 先讨论，后派单。
- 先边界，后执行。
- 先观察，后自动化。
- 先人工收口，后机器裁决。
- 便宜模型做窄执行，高级模型做低频判断。

---

## 10. 反模式

避免以下做法：

- 看到问题就直接派给执行者。
- 执行者完成后自动让它分配下一个任务。
- 让 Qwen 自己决定是否拆子任务。
- 让出错的 Qwen 去修 Qwen 适配器或 Paperclip 收口协议。
- 因为 Paperclip 有心跳，就把心跳当成优先级判断。
- 因为事务有主子嵌套，就假设它有可靠顺序执行能力。
- 因为 run 成功，就假设事务已经完成。
- 因为文档很多，就让执行者全量阅读再自由发挥。

---

## 11. 后续可产品化的点

这些是未来可以进 Paperclip 的能力，不是当前协议的前置条件：

- 新建事务时内置“董事会派单模板”。
- 创建任务前由轻量模型按技能表和角色表推荐执行者。
- 事务完成后从最终评论中抽取“建议处置”，供人类一键确认。
- 心跳任务观测面显示哪些 agent 开启心跳、间隔、并发、接续和最近运行。
- recovery 逻辑识别“执行者自身不可用”，避免把恢复任务继续派给同一个坏执行者。
- 子任务顺序执行与依赖门禁，不再只靠主子嵌套表达顺序。
