# 编排：主单拆子单、同经办、串行意图下的「抢单」— 现状与修理方向

**状态：** 现状已确认（人类实例侧观察 + 如下叙述）；**修理**为方向草案，**未**立项执行单，**不**替代 `doc/` 产品契约。  
**关联：** 与 **[`编排-run执行语义与观测边界-现状说明.md`](编排-run执行语义与观测边界-现状说明.md)**（run 语义、触及多事务、可选「一单一线」拓扑）对照阅读。

## 1. 现象（现状）

### 1.1 业务场景（人类描述，已压缩为可验收语言）

- **CEO** 从**主单**拆出 **6 条子单**；子单**均指派给同一经办**（实例中为 **CTO** / `cto-codebuddy-*`）。
- **策略意图：** 限制 **CEO 侧仅触发一路**（不允许 CEO 同时起多个并行唤醒），从而期望 **6 条子单在 CTO 侧「串行」执行**（一次处理一条，顺序推进）。
- **观察到的问题：** **第一个进入执行的 `heartbeat_run`** 在事实行为上**把后续本应由其它 run 承担的子单工作一并吃掉**（后续 run 侧表现为无事可做、noop、或与第一条 run 在事务/评论上「撞车」）。参与者口语称 **「抢单」**。

### 1.2 界面侧可佐证线索（非完整证据链）

- 同一条 run **「触及的事务」**可列出**多条**（例如一条 **已完成**、一条 **进行中**），而 run 仍为 **运行中**、**时长较长**。
- **「触及的事务」**在产品上不等于「本 run 只应承担这一条」——但在**本场景**下，与 **「6 子单应串行、一单一线」** 的**意图**发生冲突时，即构成本文所称的 **抢单问题**（编排层未把「执行范围」与「唤醒/run 边界」对齐）。

### 1.3 问题陈述（抽象）

在 **「主单拆多子单、同 assignee、限制上游单次唤醒条数」** 的组合下，**执行面**未保证：

- **每条 `heartbeat_run` 仅服务一条子单**（或等价：**单一 issue 范围封闭**），且  
- **后续 run 不会被前序 run 的上下文/行为半径吞并**，

导致 **首条 run 垄断多条子单工作**，与 **串行执行的经济与责任边界**不一致。

## 3. 修理方向（草案，供立项用）

以下分层列出，**实施须另开 `长期需求/` 或 `执行/` 单** 并通过评审；此处**不设死方案**。

### 3.1 产品与不变量（应先拍板）

- **一单一线（在本场景下）是否强制：** 每个子单是否必须 **独立终态 run**，且 **禁止** 同一条 run 在编排意图上同时承担多条子单？
- **「CEO 只起 1 个」** 与 **子单侧实际产生的 run 条数** 是否一致：需核对是否仍存在 **对多子单各起一条 `heartbeat_run`** 的路径（若存在，与「串行」策略冲突）。
- **串行** 定义：**队列顺开** 还是 **同一 assignee inbox 内严格一次只一条 in_progress**（或二者兼而有之）。

### 3.2 执行面可能落点（技术方向，非承诺）

- **唤醒与 fan-out：** 子单唤醒是否 **按序**、**上一条 run 终态后再允许下一条**；避免与「CEO 单次」语义打架。
- **上下文与指令范围：** 每条 run 注入的 **issue 范围** 是否 **仅当前子单**（其余子单不出现在默认 prompt / 工具上下文中，除非显式接力协议）。
- **签出 / 执行锁：** `issues` 与 `checkout_run_id` / `execution_run_id` 等字段是否保证 **一 run 一单** 的可审计性；后续 run 对 **已被其它 run 签收的事务** 行为定义为 **等待 / 跳过 / 明确 no-op 并记账**，而非静默合并进首 run 的工作集。
- **观测与排障：** 抢单类问题能否通过 **「当前 run 声称的 issue 范围」** 与 **活动日志** 快速对照（便于验证修理是否生效）。

### 3.3 验证思路（届时）

- **构造用例：** 1 主单 → N 子单，同 assignee，**禁止上游并行**；观测 **N 条 run** 是否 **各绑定唯一子单**、**无交叉完成**。
- **回归：** 不改变「接力 / 续跑」等**明示**多阶段语义的路径（若有）。
