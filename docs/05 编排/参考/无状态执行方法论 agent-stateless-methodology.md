# Agent 编排与无状态执行（方法论本体）

## 1. 大脑与双手解耦

- **大脑**：大模型的规划、决策、任务分解。
- **双手**：在无状态或弱状态环境里执行代码、调用工具、读写沙箱。
- **原则**：二者职责边界清晰，避免把「规划逻辑」与「具体执行环境与存量状态」焊在同一生命周期里。
- **协议侧含义**：工人落在具体厂商 API 上时，「双手」表现为 **按序的多轮消息、工具定义、以及可选的流式增量**；规划与编排应 **以回合级汇合结果与节点产出为准**，而非绑定某一客户端内部的增量实现细节。

## 2. Harness / 补丁与模型迭代

- 为弥补当前模型短板而堆积的 **harness、补丁式代码**，会随模型换代迅速贬值。
- **推论**：修补代码的速度通常跟不上模型能力提升速度；规模化时应减少「深度耦合模型缺陷」的工程债，把赌注押在 **可替换的执行单元** 与 **清晰接口** 上，而非长期维护庞大补丁树。

## 3. 运维隐喻：宠物 vs 蜂群

- **宠物模式**：执行环境与记忆、配置长期绑定在同一实例上；一次跑偏、污染或膨胀会导致整体越来越难维护，往往需要「整体重装」式恢复。
- **蜂群模式**：上层只下发指令；每个执行实例尽量 **无状态、可丢弃**；故障或脏状态 → **终止并拉起新的干净实例**，不把修复负担转嫁为「供养单一长生实例」。

## 4. 编排器（Coordinator）

- 专职 **拆任务、派活、汇合结果**，不替代工人做具体落地细节。
- 可挂载 **多路并行工人**；每路可有独立的 **系统提示、工具集、MCP 服务端配置**。
- 用于缓解 **单会话长任务上下文衰减、压缩丢记忆**：由编排维持全局意图与分工，而非指望单一对话线程承载全流程。
- **持久化边界**：编排器侧以 **任务图/分工状态、汇合后的结构化产物** 为主；**单 Worker 内与模型的多轮往返轨迹** 归入 **Session 工件**（或其快照），与协调器状态 **分列存储**，避免混成单一不可分拆 blob。

## 5. Agent 与 Session

- **Agent**：相对静态的 **模板** —— 选用的模型、工具、系统提示词、默认策略等。
- **Session**：由模板实例化出来的 **运行时**：对话轨迹、中间记忆、挂载工件等；同一 Agent 模板可同时存在多个 Session，彼此隔离。
- **分工口诀**：Agent 回答「我是谁、默认怎么干活」；Session 回答「这一轮具体在干什么、积累了什么状态」。

## 6. Session 外置与无状态 Agent

- 将会话状态 **持久化到外置存储**，支持从历史点 **恢复或分叉上下文**。
- **目的**：执行进程本身可 **随时销毁、按需新建**；状态不靠进程长生，而靠外置会话工件加载，使 Agent 集群更接近 **无状态服务的运维模型**。
- **最低可恢复性**：外置工件须能还原 **按序的多轮对话与内容块**；若存在 **工具调用闭环**，须能还原每一次 **工具调用标识** 与其 **工具结果** 之间的 **稳定对应**（否则无法无损重放或分叉会话）。

## 7. 四层栈（关注点分离）

由下至上叠加：

1. **Agent ⊕ 沙箱**：模板级 Agent 与执行沙箱解耦；双手坏了换一双，不牵动大脑定义。
2. **编排器 ⊕ 多 Agent**：一个协调者调度多个专职 Agent（或专职工人配置）。
3. **Session 作为工件**：Session 是可调度对象，支持 **回滚、克隆、快照式恢复**，作为编排的操纵面。
4. **会话持久化层**：会话数据外置，使 Agent **进程级无状态**，弹性扩缩与销毁。

## 8. Worker 节点参数化

- 每个 Worker 可独立指定：**模型、harness、skill、恢复所依赖的 Session 快照点**。
- 可外挂 **凭据、SSH、工作目录、主控下发的任务图边**，使节点专责单一子任务。

## 9. 任务图（DAG）与异构模型

- 用 **有向无环任务图** 表达依赖：并行分支后期 **汇入** 聚合节点。
- 每个节点任务边界清晰 → 每节点可使用 **独立上下文**。
- **推论**：可对不同阶段选用 **不同规格模型**（大模型攻坚 / 小模型批量），以任务形态约束 token 与延迟，而非强制全程单会话单模型。
- **消息层隔离**：在协议可表达的范围内，**每个 DAG 节点对应一份可独立维护的对话上下文**（如各自的系统提示、工具表与 `messages` 序列）；上游产物经编排 **结构化注入** 下游（如新 `user` 内容或节点入参），**默认不共享**同一长对话线程，除非产品显式要求全局单上下文字段。

## 10. 落地参考拓扑：Codex 主脑 + 无状态 CLI–模型工人（本工作方向）

以下为 **当前单机实现的选型**，与 §1–§9 原则一致；若日后更换主脑或工人形态，**方法论条款仍适用**，仅替换具体运行时。

- **主脑（编排器的实例化）**：**Codex** 承担 **任务分解、向工人发单、读取工人交付物、并行分支的结果汇合、以及决定下一步**。全图意图与分工挂在 **Codex 侧上下文** 及/或与 **外置 run 台账**（§4）对齐，**不要求** 工人进程长生保存「大脑才该管的事」。
- **工人（双手的实例化）**：**无状态 CLI 流程** 与 **模型调用** 绑在一起使用——一次工人的可验收单元，对应 **一次（或短链）命令行执行**：进程内可做模型推理或封装对模型的调用；**进程结束后** 仅以 **约定路径上的文件、或 stdout 等明确协议** 留下结果，作为 **节点产出 / Session 工件**（§6）。脏状态或失败 → **结束进程再启干净实例**，从外置工件恢复，符合 §3 **蜂群** 隐喻。
- **职责切割**：Codex **不包办** 工人在沙箱里 **逐文件落地** 的细节；CLI–模型工人 **不维护** 全域 DAG。编排平面与 Session 平面 **分列存储**（§4、§7）。

## 11. 任务上下文分层

- **派单上下文**：由主脑裁剪后交给工人，回答「这一节点要做什么、边界在哪里、验收看什么」。
- **工人 Session**：记录工人执行过程，回答「它如何探索、调用工具、产生中间结果」；默认作为过程证据，不直接传给下游。
- **收单结果**：工人对主脑的结构化交付，回答「完成了什么、证据是什么、还有什么不确定」。
- **长期记忆**：从原始会话和过程证据抽取而来，用于复盘、追溯、热/冷记忆，不替代收单结果。
- **原则**：下一节点默认只继承 **上游收单结果 + 主脑选中的必要证据**；完整工人 Session 只有在明确需要续接同一会话状态时，才由主脑摘取片段注入。

## 12. 续跑闸门（Continuation Gate）

- **问题**：Session 外置只能解决「记得住」，不能自动解决「是否继续、何时继续、带什么上下文继续」。
- **续跑闸门**：编排器在每个节点收单后，必须显式判断下一步：继续派单、要求人类补充、关闭任务、或暂停。
- **职责边界**：机械 watcher 可以发现结果、校验格式、更新状态、唤起 Codex；但 **是否继续与上下文裁剪** 仍属于主脑判断。
- **低风险自动续跑**：探查、整理、格式化、只读验证等可配置自动继续；涉及实现、生产、账务、鉴权、真实凭据、不可逆操作时默认进入人工确认。
- **原则**：续跑状态必须外置到 run 台账，不能只存在于主脑对话感觉里；否则系统会退回人工复制提示词。

## 13. 参考产品形态：Claude Code

- **判断**：Claude Code 已经把蜂群编排的关键能力产品化：子代理、技能、Hook、沙盒、权限、会话恢复、分叉、上下文压缩与会话落盘。
- **用法**：本文不把 Claude Code 当作当前实现依赖，而把它当作 **蜂群编排产品形态的参考实现**。
- **抽象**：从 Claude Code 中抽取的是机制，而不是外壳：**Worker manifest、Hook 生命周期、Sandbox 边界、Session 生命周期、Skill 能力包**。
- **当前实现**：本工作方向仍以 **Codex 主脑 + 无状态 CLI–模型工人** 为单机可控载体；若未来 Claude Code 链路可稳定使用，也可替换为更接近参考形态的实现。

## 14. 主控也必须可重置

- **修正**：无状态不只约束工人，也约束主控。若工人可丢弃、主控会话却无限长生，系统只是把「宠物」从工人换成了主控。
- **原则**：主控会话是一次编排运行环境，不是编排状态本身；真正不可丢的是 **外置任务图、节点结果、续跑位置、未决问题与恢复摘要**。
- **主控恢复包**：新主控应从 `run.json`、最近 `result.json`、`digest.md`、`open_decisions`、`next_action` 等外置材料恢复，而不是依赖旧主控完整聊天历史。
- **职责边界**：Codex 主脑默认只做任务分解、上下文裁剪、派单、收单、续跑裁决；若主控亲自执行落地工作，必须把自己登记为一个 Worker 节点，并同样产出可验收结果。

## 15. 上下文生命周期与占用视图

- **问题**：Session 外置解决「记得住」，但不自动解决「何时上下文已经拥挤、腐化、需要压缩或重启」。
- **生命周期**：主控与工人 Session 都会经历 **空白期 → 充分期 → 拥挤期 → 腐化期**；编排的价值之一，是在腐化前把成果固化为 `result.json`、`digest.md` 与长期记忆。
- **占用视图**：系统必须提供代码级工具估算上下文占用，而不能只靠模型自觉判断自己是否「老化」。
- **双轨 token 口径**：第一轨是本地估算 token，用于运行时闸门；第二轨是官方 usage / billing，用于事后成本统计与校准。第一版不得依赖官方计费作为唯一判断依据。
- **闸门动作**：当上下文进入拥挤期，应停止继续灌入大文件，优先压缩、收单或切分节点；进入腐化风险期，应触发工人停止回单、主控 handoff 或新主控重启。

## 16. 编排台账与人类视图分离

- **修正**：人类看板、四表、阶段地图等 Markdown 文档可以作为人工样本和阅读视图，但不能替代机器编排状态。
- **原则**：执行过程中的状态变更应首先落在 `run.json`、节点目录、`job.json`、`result.json` 与事件记录中；人类视图只在明确 checkpoint 回填。
- **边界**：工人不得修改人类看板；主控也不得在执行中为保持同步而反复人工改表。看板应是由编排台账回填或渲染出的视图，而不是工人和主控共享涂写的状态源。

## 17. 消息传递必须经编排器路由

- **修正**：`job.json -> worker -> result.json` 只能表达单次派单与收单，不足以表达多上游汇入、请求补信息、进度事件、上下文注入、主控交接与失败重试。
- **原则**：工人之间默认 **不点对点通信**；所有可传递消息经主控 / 编排台账路由。主控负责选择哪些上游结果、证据或摘要注入下游。
- **消息不是 Session**：Session 记录过程，消息传递结构化事实、请求、结果、裁决或 artifact 引用。下游默认不继承完整上游 Session。
- **信封与内容分离**：`job.json` / `result.json` 仍是内容真源；消息信封只记录谁在何时把什么内容交给谁、因为什么交付、属于哪个 run。
- **可重放性**：派单、回单、补信息请求、artifact 引用、上下文注入、主控 handoff 与续跑裁决都应能从消息记录和 run 台账重放，而不是只存在于聊天历史里。
